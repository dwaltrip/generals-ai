# Strakam Paper: Takeaways for Our FFA Project

**Date:** 2026.05.02
**Companion to:** `5.02-5-strakam-paper-summary.md` (pure paper summary).
**Purpose:** What the paper means for our 8-player FFA project. Updates to existing decisions, things that transfer cleanly, things that need genuine rethinking, and new open questions surfaced.

This doc assumes familiarity with the existing project planning docs (notes-and-decisions, MVP sketch, repo survey).

---

## 1. TL;DR of project-relevant updates

- **Architecture lean changes from "ResNet-style" to U-Net (DeepNash style).** The paper uses U-Net specifically because it suits per-cell action heads on grid games. Updates notes-and-decisions §5.
- **Memory: start with hand-crafted features, defer learned memory to Phase 2.** Strakam reached top 0.003% of 1v1 with no recurrence — pure feedforward + four hand-crafted memory feature-maps. Strong existence proof that we don't need learned memory for the BC MVP.
- **Auxiliary heads: defer.** Paper used just policy + value. Add aux heads only if BC underperforms.
- **Add a player-rating filter to the BC corpus.** Strakam used ≥70 stars; we should pick an analogous FFA threshold rather than relying on placement-based filtering alone.
- **Add an upper-bound length filter to the BC corpus.** Strakam dropped >500-turn games as noisy.
- **Reward shaping is qualitative-behavior-changing, not just sample-efficiency-changing.** Without it, self-play in 1v1 converges to "aggressive but exploitable." This is a Phase 2 concern but worth holding onto: shaping tilts which local optimum self-play finds, so it can't be skipped as a "we'll add it later if we need it."
- **Compute calibration confirmed.** 3h BC + 36h self-play on a single H100 in 1v1. Our Phase 1 budget assumption (sub-day BC on a single GPU) was right.

## 2. Updates to existing decisions

### 2.1 Architecture: U-Net, not ResNet

Notes-and-decisions §5 had us leaning toward ResNet-style trunk (8-12 residual blocks, 128-256 channels). The paper uses **U-Net** — explicitly cited as the same architecture as DeepNash (Perolat et al. 2022, Stratego).

Why this is the right update:

- DeepNash is a well-documented, validated architecture for an imperfect-information grid game. Direct method transfer, not surface-level resemblance.
- U-Net's encoder-decoder + skip connections fits per-cell action heads naturally — global context flows down through downsampling, local precision comes back through skip connections. ResNet keeps spatial resolution constant throughout, which is fine but doesn't match the inductive bias of "global situation, applied locally" as cleanly.
- Strakam's failure mode of "agent thinks it can pass through mountains" (paper §6.1) is partly a CNN-can't-compute-distances problem. U-Net's multi-scale receptive field helps but doesn't fully solve this — flagging because for FFA, with potentially larger maps, this could be more pronounced.

**Action:** look up DeepNash's exact spec (depth, channel widths, normalization) before the architecture session. Paper points to Perolat et al. 2022 supplementary material; arXiv 2206.15378.

### 2.2 Memory: hand-crafted, learned later

Notes-and-decisions §5 listed memory mechanism as open: LSTM vs. transformer-over-window vs. stacked frames. The paper provides a useful data point — they used **none of the above** and reached top 0.003% in 1v1.

Their four hand-crafted memory feature-maps (paper §4.2 / our summary doc §4.2):

1. Revealed castles/generals positions
2. Cells the agent has explored
3. Cells the agent knows the opponent has seen
4. Last-7 moves on each side

These are all per-cell binary or short-history features stacked as additional input channels. Cheap, simple, no recurrence required.

For us:

- **Start with the same approach for the BC MVP.** Cheaper, simpler, gets us a working pipeline faster.
- Some categories generalize directly to FFA: explored cells (same), revealed-but-fogged enemy structures (now per-opponent), opponent's scouting (now per-opponent — though "the agent knows opponent X has seen cell Y" is information we might not have for opponents we've never met).
- Last-N-moves history needs per-opponent or aggregated handling for FFA — design choice.
- **Learned memory (LSTM/transformer) becomes a clear Phase 2 lever** if BC underperforms. The paper itself flags this as future work, with specific motivation (timing of discovery, exact army counts at time of last sighting). For FFA the case is stronger since memory burden grows with player count, but we don't have to commit upfront.

### 2.3 Auxiliary heads: defer

Notes-and-decisions §5 listed auxiliary heads (placement prediction, attack-target prediction, who-eliminates-whom) as a possible component. The paper uses **just policy + value**, no aux heads, and reaches SOTA.

For us:

- **MVP architecture: policy head + value head, no aux.** Keeps things simple.
- Aux heads are a known regularizer (multi-task learning shapes trunk representation), so they remain a real lever for Phase 2 if BC underperforms.
- For FFA specifically, aux heads have more candidates to choose from than 1v1 (placement prediction is meaningful in FFA but trivial in 1v1) — which makes them genuinely worth trying as a Phase 2 lever rather than a Phase 1 default.

### 2.4 Corpus filtering: add length cap and rating filter

Notes-and-decisions §4 listed three filters: integrity cuts, early-elimination cutoff (turn 50–75), and bottom-half placement filter. The paper's filter set is different in informative ways:

- **Length cap (drop games >500 turns).** They observe long games are atypical/suboptimal. We should add an analogous upper-bound filter — likely a different threshold for FFA (FFA games are typically longer than 1v1).
- **Patch filter (newest patch only).** We should match this. Easy to implement if patch info is in replay metadata.
- **Skill filter (≥70 stars).** Stronger version of our bottom-half placement filter. Worth layering both in: rating filter cuts the lobby-quality dimension, placement filter cuts the within-game-quality dimension. They're not redundant.

Our existing **early-elimination cutoff** doesn't have a strakam analog — they don't have early-elimination because 1v1 doesn't have that failure mode. It remains a uniquely-FFA filter we should keep.

### 2.5 Compute and sample-size calibration

Confirmations rather than updates, but worth pinning down:

- **BC compute:** 3h on a single H100 for 1v1. Our assumption was sub-day on a single GPU. ✅ Confirmed.
- **BC corpus size:** 16k filtered games for 1v1 was the proof point. FFA has 8 trajectories per game vs. 2, so equivalent supervised samples come from fewer raw games (modulo within-game correlation across players). Our planned 25-70k post-filter range is comfortably on the right order of magnitude.
- **Phase 2 RL compute:** 36h on a single H100 for 1v1. For FFA (sparser placement reward, longer credit assignment, more opponents per game) plausibly 2–5× more wall-clock. Single H100-class machine probably still in scope.

### 2.6 Reward shaping is load-bearing for self-play (Phase 2 reference)

Not a Phase 1 decision update, but a calibration of expectations: the paper ablation shows reward shaping is the largest single win-rate jump after BC→SP, *and* the qualitative claim is that without shaping, self-play converges to a different (worse) equilibrium.

For us this means: when we get to Phase 2, we should not treat reward shaping as a "nice optimization" but as part of the core recipe. The summary doc §6 captures the form. The multi-opponent aggregation question (max/mean/sum over opponents) becomes the substantive open question — see §5 below.

## 3. Things that transfer essentially unchanged

- **Per-cell H×W×9 policy structure.** Action space is identical between 1v1 and FFA — still 4 dirs × 2 splits + pass per cell. Direct transfer.
- **Potential-based reward shaping form** (Phase 2 reference). Just need to pick the multi-opponent aggregation for `x_enemy` (open question, §5).
- **Argmax opponents in self-play; small pool (N≈3); 45% replacement threshold** (Phase 2 reference). Notes-and-decisions §7 already settled on simple historical-pool self-play (no league play), and the paper's specific numbers give us concrete starting points.
- **BC corpus filtering philosophy:** quality > quantity, multiple complementary filters layered.
- **Two-stage recipe (BC → self-play):** the whole project framing.

## 4. Things that need genuine rethinking for FFA

### 4.1 Memory features (per-opponent generalization)

The four hand-crafted categories don't transfer literally:

- **Revealed castles/generals:** transfers. Just multiply: per-opponent revealed-positions maps, or one aggregated map plus opponent-identity channel.
- **Explored cells:** transfers unchanged.
- **Cells opponent has seen:** in 1v1, "the opponent" is well-defined and we can sometimes infer their visibility. In FFA, "opponent X has seen cell Y" is information we may not have for opponents we've never directly interacted with. Likely need to either limit to opponents we've encountered or drop this feature for FFA.
- **Last-N moves of each side:** per-opponent? Aggregated? Selective (only top-k threats)? Real design choice.

### 4.2 Observation channels (per-opponent representation)

The paper has a single "opponent" — repo's `opponent_cells`, `opponent_land`, `opponent_army` channels. FFA has 7 opponents. Notes-and-decisions §5 already flagged this as the most consequential observation decision.

Choices to make in the architecture session:

- **Slot count:** how many opponents to expose to the network. Top 3? Top 5? All 7?
- **Sort key:** by what scalar do we order opponents into slots? Army count? Land count? Distance? Learned threat score? Static slot-by-original-id is probably wrong (slot-permutation-invariance is needed).
- **Per-opponent feature set:** ownership mask + summary scalars (army, land, last-seen castle position)? More?

### 4.3 Value head target

The paper's value head predicts a scalar — straightforward in 1v1 because the outcome is binary. In FFA there's a real design choice:

- Probability of winning (binary, ignores the placement signal entirely)
- Expected placement (scalar, captures placement)
- Distribution over placements (8-way categorical)
- Probability of placing in top-k (binary or vector)

The paper gives no signal here. Likely worth picking expected placement for simplicity in MVP, with categorical as a Phase 2 upgrade if the value signal seems weak.

### 4.4 Reward shaping multi-opponent aggregation (Phase 2)

The form `φ_x(s) = log(x_agent / x_enemy) / log(max_ratio)` requires defining `x_enemy` for FFA. Options:

- **Max** over remaining opponents — paranoid, focuses on biggest threat, possibly over-defensive.
- **Mean** over remaining opponents — balanced, ignores the highest-variance opponents.
- **Sum** over remaining opponents — explicitly placement-oriented.
- **Per-opponent terms** — separate sub-potential per opponent, summed with weights.

The "agent gets aggressive without shaping" lesson likely applies more strongly in FFA, since aggressive play in FFA gets ganged up on faster. Plausible defaults: try mean first (balanced), try max if mean produces passive play.

## 5. New open questions surfaced

Things the paper raised that weren't on our radar before, or sharpened existing questions:

- **U-Net depth, channel widths, normalization.** Need to look at DeepNash supplementary. Concrete blocker for the architecture session.
- **BC loss formulation.** Joint cross-entropy over the full 5-element action vector? Factored across (pass, i, j, direction, split)? Action masking applied inside the loss or only at sampling? The paper doesn't say. We need to make a choice; probably worth looking at the strakam codebase for how they actually do this.
- **Data augmentation via board symmetries.** Free 4× (rotations + reflections) if it works. Paper doesn't mention. Cheap to try.
- **Network normalization scheme.** BatchNorm, LayerNorm, GroupNorm. Not stated in paper. DeepNash supplementary likely has this.
- **Training-data perspective for skill-filtered games.** When a game has one ≥70-star and one lower-rated player, do we supervise on both perspectives or just the strong one? Paper is ambiguous. For FFA the analog is more involved — likely we want to supervise per-player based on individual rating + placement, not the whole game's qualifying status.
