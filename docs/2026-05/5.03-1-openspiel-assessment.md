# OpenSpiel: Assessment for FFA Project

**Date:** 2026-05-03
**Context:** Evaluated whether Google DeepMind's OpenSpiel framework (github.com/google-deepmind/open_spiel) is useful for this project. Initial assumption was that it's primarily a small-game / exact-solution toolkit — that turned out to be wrong, but the conclusion (not the right foundation for us) holds for different reasons than expected.

---

## 1. What OpenSpiel actually is

OpenSpiel is broader than its reputation suggests. It's a general-purpose research framework for RL and game-playing, not a solver for toy games. Specifically:

- **Game support:** n-player, imperfect information, simultaneous moves, stochastic, general-sum — all the properties generals.io has. Includes large-scale games (Chess, Go, Atari, Stratego, Hanabi, Hearts 3-6 players, Chinese Checkers 2-6 players, Poker 2-10 players).
- **Algorithm coverage:** PPO, A2C, DQN in the RL category; NFSP, PSRO, Deep CFR, AlphaZero, Exploitability Descent in the MARL category; α-Rank, Nash Averaging, Replicator Dynamics for evaluation. PPO and several MARL algorithms are marked as "lightly tested" (~); PSRO, AlphaZero, Deep CFR, and the evaluation tools are thoroughly tested.
- **Languages:** games in C++ with Python bindings; algorithms in Python (mostly TensorFlow, with stated interest in JAX/PyTorch support).

## 2. Where it falls short for us

### 2.1 Environment execution model: CPU-bound

Game simulations are C++ on CPU. Neural network training (via TF) uses GPU, but the environment stepping — which dominates self-play throughput — is CPU-bound with CPU↔GPU data transfer overhead for observations and actions.

The strakam repo's JAX-native approach (and projects like Pgx, which benchmarks 10-100x speedups over OpenSpiel on a DGX-A100) puts both game simulation and network on the same device. No data transfer, and `jax.vmap` parallelizes thousands of games on the accelerator. This matters in Phase 2 (self-play PPO) where millions of environment steps are interleaved with network inference. Strakam achieved 36h on a single H100 for 1v1 partly because the whole pipeline is on-device; for 8-player FFA with bigger grids, throughput pressure is higher.

Phase 1 (BC) doesn't touch the environment at all, so this distinction is irrelevant there.

### 2.2 Network architectures: generic, not game-specific

This is the most concrete gap. OpenSpiel's algorithm implementations ship with:

- **AlphaZero:** ResNet, conv2d (conv+batchnorm+relu blocks), or MLP — parameterized by width and depth, auto-sized to game dimensions.
- **DQN:** MLP with configurable hidden layers (default `[64, 64]`).
- **PPO and others:** likely similar generic MLP/conv patterns.

No U-Net. No DeepNash-style Pyramid Module. Nothing resembling the architecture that produced strong results on Stratego (DeepNash) or generals.io (strakam). The architectures are research-reference-grade — sufficient to demonstrate an algorithm on Connect Four or Breakthrough, not designed for production performance on a complex spatial game.

This means that even if we built generals.io as an OpenSpiel game, we'd still write our own network from scratch: U-Net torso, per-cell H×W×9 policy head, hand-crafted memory features, value head. The part that most determines agent quality is entirely custom work regardless of framework.

Worth noting: DeepNash was a DeepMind project that likely used OpenSpiel's Stratego environment, but its architecture and training infrastructure were custom — the Pyramid Module was not contributed back as a reusable OpenSpiel component. The strakam paper says "same architecture as Perolat et al. 2022" but reimplemented it independently too.

### 2.3 Integration cost vs. benefit

Building generals.io as an OpenSpiel game (C++ extensive-form interface with information states, legal actions, chance nodes) is nontrivial porting work. We'd need an FFA environment either way — the question is whether to build it in OpenSpiel's API or as a JAX-native env. Given the performance story above and the fact that we're already committed to a JAX/Equinox stack, building a second environment implementation for OpenSpiel has poor ROI.

## 3. Where it could genuinely help

### 3.1 PSRO as reference implementation

Policy-Space Response Oracles (PSRO) is thoroughly tested in OpenSpiel and is the principled version of opponent-pool self-play we'd be doing in Phase 2. If we go beyond strakam's simple N=3 argmax pool toward something more rigorous, reading OpenSpiel's PSRO implementation before designing ours could save real design time. Same for NFSP (Neural Fictitious Self-Play), which is designed for imperfect-info games and is a well-tested alternative to vanilla self-play + opponent pool.

These are "read the code carefully" recommendations, not "import the library" ones.

### 3.2 α-Rank for population evaluation

Once we have multiple trained agents (BC checkpoint, BC+PPO iterations, different hyperparameter runs), α-Rank gives principled population-level evaluation beyond pairwise win rates. This doesn't require generals.io in OpenSpiel — just a payoff matrix from running agents against each other (computable from our own env), fed into OpenSpiel's evaluation tools as a standalone call. Low integration cost, real analytical value.

### 3.3 Algorithm design patterns

Several MARL algorithms in OpenSpiel encode design patterns relevant to Phase 2 decisions that aren't fully settled yet: how to structure opponent pools, how to handle n-player general-sum self-play, when to swap opponents, how to measure convergence. These are design-level insights available from reading the implementations, independent of whether we use the framework.

## 4. Bottom line

**Not the right foundation for environment or training infrastructure.** The CPU game simulation, generic network architectures, and TensorFlow-centric algorithm implementations don't match our JAX-native stack or the architectural requirements (U-Net, per-cell policy head) for a competitive agent.

**Genuinely useful as a reference library for Phase 2 self-play design.** PSRO implementation, NFSP patterns, α-Rank evaluation — read before building our own. The fact that we're rebuilding training infrastructure from scratch (strakam's published PPO is a 313-line sketch, not a real training pipeline) makes "study how OpenSpiel structures these algorithms" more valuable than it would be if we were extending a working system.

**No relevance to Phase 1 (BC).** BC is supervised learning on parsed replays. No environment, no self-play, no opponent pool. OpenSpiel offers nothing here.

**One concrete action item for Phase 2 planning:** read OpenSpiel's PSRO and NFSP implementations before committing to an opponent-pool design. Estimated time: a few hours of code reading, no integration work.
