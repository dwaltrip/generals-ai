## Diagnostic data — replay `YC-ZjiMiY` (smallest mismatch)

**Listings (winner first):** `[4, 3, 2, 6, 0, 7, 5, 1]`
**Our deduced ranking:** `[4, 3, 0, 2, 6, 7, 5, 1]`

The contention: our `has_kill[0] = True` (slot 0 captured slot 7 at t=149 per our capture_events), but listings rank slot 0 BELOW no-kill slots 2 and 6, suggesting listings consider slot 0 no-kill.

Our death_events (by timestep): `[1@121, 5@148, 7@149, 0@239, 6@314, 2@336, 3@468]` — matches the implied death order from listings exactly, so **death-order tracking is correct**.

Our capture_events: `[(4,5)@148, (0,7)@149, (4,0)@239, (3,6)@314, (4,2)@336]`.

**Bundle's has_kill computation** (line 68009–68011): `has_kill = capturedPlayers[e].length > 0`. Monotonic — once you capture someone, you have has_kill forever, even after being captured yourself. Our bool model matches this. So our `has_kill[0] = True` matches what the bundle would compute.

**So the disagreement is bundle/our-sim vs server-listings, not bundle vs our-sim.** Hypotheses:
1. **Server-listings computes ranking differently from the bundle's lbSort.** Possible — design doc §10 calls out that the bundle is the replay-viewer's reconstruction, not server-authoritative. But 12/28 mismatch is too high for "occasional bundle bug."
2. **Our capture detection has a false positive at t=149 (slot 0→slot 7).** Maybe slot 7 actually surrendered/AFK'd at t=149 (not captured by 0), and we wrongly fire execute_player_capture because slot 7's general tile happened to get attacked by slot 0 right at the surrender moment. **Worth checking the wire afks for this replay** — earlier inspection said zero AFK events but I haven't re-verified specifically for YC-ZjiMiY.
3. **Some other simulator bug** in the middle of the ranking — combat math, dependency loop, etc. — that's affecting other replays too.

## Filter helper not yet authored

A reusable `is_vanilla_ffa(wire) -> bool` was prototyped inline during debugging but not committed. When we resume, lift it to a module (e.g., `filter.py` or extend `decode.py`) and use it from the test harness. The full §4 hard-filter list is:

```python
def is_vanilla_ffa(wire: list) -> bool:
    if wire[12] is not None: return False              # teams
    if wire[13] is not None: return False              # custom map
    if wire[16]: return False                          # swamps
    if wire[21]: return False                          # modifiers
    if len(wire) > 22 and wire[22]: return False       # observatories
    if len(wire) > 23 and wire[23]: return False       # lookouts
    if len(wire) > 24 and wire[24]: return False       # deserts
    if len(wire) > 27 and wire[27]: return False       # generalTrades
    if len(wire) > 28 and wire[28]: return False       # tunnels
    if len(wire) > 30 and wire[30] is not None: return False  # chessClock
    if len(wire) > 34 and wire[34]: return False       # strongholds
    return True
```

## Corpus-size correction

Earlier in the session I claimed ~27% vanilla survival from the v15+ corpus. **That was wrong** — it was a sampling-window bias (first ~25k v15 rows in the DB are from 2025-04-01, the v15 launch day, which had Defection enabled as a launch event). Stratified sampling shows v15 is mostly vanilla post-launch. Real vanilla-FFA survival is closer to the design doc's ~60% estimate, possibly higher (~80%+).
