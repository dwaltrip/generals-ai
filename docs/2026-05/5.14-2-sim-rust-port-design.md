# Sim Rust Port — Design Decisions

Captures the load-bearing choices made before any porting work starts.
Separate from the [5.14-1 session notes](./5.14-1-sim-perf-and-rust-port-plan.md),
which framed the options; this doc is the landing point.

## Boundary

`parse_replay` stays a thin Python orchestrator:

```python
def parse_replay(raw):
    replay = decode_wire(raw)
    state = sim_core.simulate(replay)   # one FFI call, full sim
    return state, replay
```

Decode stays Python (one-shot, not hot). `State` is a Rust `#[pyclass]`.
`Replay` stays a Python object — Rust reads its numpy arrays zero-copy
via the `numpy` crate during the one-time setup inside `simulate()`.

## State ownership

Rust owns every State field. No `#[pyo3(get/set)]` on hot internals — Python
literally cannot reach in mid-sim. Outputs (snapshots, damage matrices,
events, final scalars) exposed via getters at parse end only.

GIL released during the hot loop via `py.allow_threads`. Sets up
thread-parallel parsing as a future option (replacing the multiprocessing
plan).

## Migration strategy

**Standalone-then-swap, not piece-by-piece in `parse_replay`.** Build the
Rust sim out of band; Python sim keeps running in `parse_replay` as the
reference oracle throughout. Validate Rust against Python out-of-band
(`replay-parser/tmp/parity_rust_vs_python.py` — per-replay snapshot diff
over a corpus sample). When parity holds, **one commit** flips
`parse_replay`. Avoids the dual-rep / numpy-aliasing problem that a
plumbed transition would have forced.

After the swap, the Python sim either moves to `tests/reference_sim.py`
(kept for future Rust A/B) or gets deleted.

## Snapshots

Accumulated in Rust as `Vec<Vec<T>>`; per-tick cost is the overflow gate
+ 3 buffer clones (no FFI). Bulk-emitted as `list[np.ndarray]` at parse
end via getters — same shape as today's output, downstream consumers
unchanged.

## Workspace + toolchain

- `sim-core/` is a uv workspace member, shares the root `.venv`.
- PyO3 0.28 (verified working with Python 3.14 in this repo via the
  throwaway `tmp/pyo3-314-check/` crate; rust-py-demo's 0.22 + 3.13
  pin is stale by comparison).
- Phantom-dep risk from the shared venv acknowledged but unguarded;
  `deptry` is the mitigation if it bites later.

## First port target

`apply_production` — smallest mutation surface (writes only `armies`),
self-contained against State only, ~7.8% of runtime per the post-int32
profile. Built standalone + parity-checked against Python before any
wiring change to `parse_replay`.
