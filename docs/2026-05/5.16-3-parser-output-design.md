## 1. Architecture: two sibling files per game

Each parsed game produces two files:

- `<id>.npz` — **sim output**. Frozen alongside the sim; regenerating means re-running the sim.
- `<id>.meta.npz` — **per-perspective metadata sidecar**. Cheap to regenerate; the part we'll iterate on.

**Why split.** The sim file is the bulky one (raw per-timestep state dominates byte count). Metadata is tiny by comparison. Bundling them means any change to how metadata is defined — redefining the rolling-rate window, adding a new per-perspective field, fixing the curated-player list — rewrites the bulky sim half too. Across ~170k games at ~50–65 GB compressed, that's a lot of disk I/O for what amounts to a few KB of changed bytes per game.

Splitting also gives a clean writer surface: `write_sim_output(...)` takes no perspective information at all. Perspectives are entirely the metadata builder's concern.

**Join key.** Both files share `<id>` in the filename and store `replay_id` internally for sanity-checking.

## 2. Sim-output file (`<id>.npz`)

### 2.1 Schema

Let `T = snapshots_len` (= initial snapshot + one per completed step), `P = num_players`, `HW = map_width × map_height`.

| Key | Dtype | Shape | Notes |
|---|---|---|---|
| `replay_id` | `<U16` | scalar | for sibling-file cross-check |
| `version` | `int32` | scalar | wire version |
| `map_width`, `map_height` | `int32` | scalar | |
| `mountains` | `int32` | `[M]` | flat tile indices |
| `initial_cities` | `int32` | `[C0]` | flat tile indices |
| `initial_city_armies` | `int32` | `[C0]` | parallel to `initial_cities` |
| `initial_neutrals` | `int32` | `[N]` | flat tile indices |
| `initial_neutral_armies` | `int32` | `[N]` | parallel |
| `initial_generals` | `int32` | `[P]` | per-slot tile; `-1` for empty (empirically never in filtered FFA) |
| `ownership` | `int8` | `[T, HW]` | per-snapshot grid; `-1` neutral, `-2` mountain, `0..P-1` player |
| `armies` | `int16` | `[T, HW]` | per-snapshot grid; i16-ness inherited from `state.snapshots_armies` (sim narrows from i32 + overflow-asserts at snapshot time) |
| `cities` | `int32` | `[C_final]` | initial + new in chronological order; mirrors `state.cities` |
| `cities_present_at` | `int32` | `[C_final]` | snapshot index at which each city is first present on the map; `0` for initial, `e+1` for new (see §2.2) |
| `death_events` | `int32` | `[Nd, 2]` | `(timestep, player)` |
| `capture_events` | `int32` | `[Nc, 3]` | `(timestep, captor, captured)` |
| `neutralize_events` | `int32` | `[Nn, 2]` | `(timestep, player)` |
| `actions_source` | `int16` | `[P, T-1]` | `-1` = pass sentinel |
| `actions_dest` | `int16` | `[P, T-1]` | `-1` = pass sentinel |
| `actions_is50` | `int8` | `[P, T-1]` | `-1` = pass, `0` = normal move, `1` = split move (wire-aligned for non-pass) |

### 2.2 Cities representation — `cities` + `cities_present_at`

Cities are added during the game in two ways:

- General captured → captured player's general tile becomes a city for the captor.
- General neutralized (AFK) → AFK'd player's general tile becomes a neutral city.

`cities` mirrors `state.cities`: initial cities first, then new ones in chronological order. `cities_present_at[i]` is the snapshot index at which `cities[i]` is first present on the map.

- Initial cities: `cities_present_at[i] = 0`.
- New cities: `cities_present_at[i] = e + 1`, where `e` is the pre-increment event timestep (same convention as `death_events` / `capture_events` / `neutralize_events`).

The `+1` is applied at parse time so the dataloader can index with a plain `<= t` comparison:

```python
present = cities[cities_present_at <= t]
```

Storage: `C_final ≤ C0 + (P-1)` (at most one new city per player elimination, capped at 7 in 8-player FFA). ~150 bytes per game.

### 2.3 Per-player action arrays — `[P, T-1]`, not perspective-keyed

Sim records actions for **all P players**, not just the K curated perspectives. The dataloader slices `actions_*[perspective_player_ids, :]` at load time.

**Why per-player:**
- `sim_core` has zero awareness of perspectives. Cleaner Rust surface.
- Storage difference is negligible at `P=4..8` vs `K=4..8`.
- Perspective selection becomes a dataloader concern, freely revisable without re-parsing the corpus.

**Sentinel convention.** All three action arrays use `-1` as the pass sentinel. For `source` and `dest` this is out of their valid range (`0..HW-1`). For `is50`, the array is `int8` (not `uint8`) to make room for `-1`; `0` and `1` retain their wire-aligned meaning ("normal move" / "split move").

### 2.4 Snapshot ↔ action binding

`actions[:, t]` is the decision acted on `snapshot[t]`, producing `snapshot[t+1]`.

`actions` has `T-1` columns; BC pairs `(ownership[t], armies[t], actions_*[:, t])` for `t = 0 .. T-2`. The terminal snapshot at index `T-1` has no paired action. See §9.5 for why the length mismatch is encoded structurally rather than padded.

### 2.5 What's recorded vs. pass

Per `2026-05/5.13-1-parser-implementation-plan.md` §9:

**Recorded:** any move that survived the buffer-pop validity gate (`is_valid`). Includes moves that subsequently get inward-first-canceled or cycle-dropped during resolution — those were real player intent.

**Sentinel (pass):** empty buffer at this tick, or all queued moves discarded at pop time as type-A invalid (stale decisions conditional on now-impossible state).

## 3. Sim_core (Rust) additions

Two surface additions on top of the current `State`.

### 3.1 Per-player action arrays

```rust
pub actions_source: Vec<Vec<i16>>,  // P entries, each grows per step; sentinel -1
pub actions_dest:   Vec<Vec<i16>>,  // sentinel -1
pub actions_is50:   Vec<Vec<i8>>,   // sentinel -1
```

**Recording mechanism.** At the top of each `step()`, push a sentinel row for every player (length grows by 1 for each player vec). At `select_candidates(...)` — the existing post-pop, post-`is_valid`, pre-priority-sort point — overwrite the last entry for each player whose move was popped.

Final lengths: `actions_*[p].len() = snapshots_len - 1` for every `p`.

**Python getters.** `actions_source` and `actions_dest` return `np.ndarray[P, T-1]` int16; `actions_is50` returns int8. Stack-and-convert at access time, matching the existing `snapshots_*` pattern.

### 3.2 City presence tracking

`state.cities: Vec<i32>` already grows monotonically (initial + appends on capture/neutralize), but doesn't record *when* each non-initial city was added.

Add a parallel array tracking the snapshot index at which each city becomes present:
```rust
pub cities_present_at: Vec<i32>,   // parallel to `cities`; 0 for initial, e+1 for new
```

Initial cities push `0` at `build_initial`. Non-initial appends (in `execute_player_capture` and `try_neutralize_player`) push `state.timestep + 1` — the parse-time `+1` from §2.2 lives here.

Python getters: `cities() -> np.ndarray[C_final]` int32, `cities_present_at() -> np.ndarray[C_final]` int32.

### 3.3 Recording-in-Rust rationale

Action recording is *logically* separable from the simulation math — no priority sort, no combat, no production. But the recording moment is **after AFK processing, before move resolution**, and the state at that moment isn't directly any snapshot. It's `snapshot[t]` with AFK-induced ownership flips applied.

A Python re-derivation path would replay AFK effects on `snapshot[t]` + reimplement buffer-drain + `is_valid` evaluation. Roughly 50–100 lines of orchestration logic that has to stay in sync with `sim_core`. The Rust path is ~5 lines added to a code path already doing this work.

The seam is clean enough that we could move recording to Python later if its semantics start churning faster than sim semantics. Not today.

## 4. Metadata sidecar (`<id>.meta.npz`)

### 4.1 Schema

Let `K` = number of perspectives in this game (curated players who played in it).

| Key | Dtype | Shape | Notes |
|---|---|---|---|
| `replay_id` | `<U16` | scalar | cross-check with sim file |
| `sim_core_version` | `<U16` | scalar | cargo pkg version or git SHA of the sim binary that produced the paired `<id>.npz`; supports "which sim built this?" lookups after a mid-corpus regression |
| `perspective_player_ids` | `int8` | `[K]` | slot in this game's ranking (0..P-1); join key for `actions_*`, `death_events`, etc. |
| `placement` | `int8` | `[K]` | final placement (1..P), narrowed from DB `replay_players.placement` |
| `stars_at_start` | `float32` | `[K]` | from wire `static.stars[slot]` |
| `elim_timestep` | `int32` | `[K]` | `-1` if survived to game-end; from sim `death_events` |
| `rolling_1st_rate` | `float32` | `[K]` | `-1.0` = stubbed/unknown |
| `rolling_top3_rate` | `float32` | `[K]` | `-1.0` = stubbed/unknown |
| `prior_games_count` | `int32` | `[K]` | `-1` = stubbed/unknown |

### 4.2 Field provenance

Most fields are independent of the sim:

| Field | Source |
|---|---|
| `perspective_player_ids` | curated-list ∩ this game's ranking |
| `placement` | DB `replay_players.placement` |
| `stars_at_start` | wire `static.stars[slot]` |
| `rolling_*`, `prior_games_count` | per-player chronological DB walk |
| **`elim_timestep`** | **sim `death_events`** (only sim-dependent field) |

This is what makes the sidecar structure clean: only `elim_timestep` reaches across the seam.

### 4.3 What's not in the meta file (yet)

- `perspective_index` from the original design (§5.2) is collapsed: it was "slot ordinal in this game's ranking" — i.e., the same value as `perspective_player_ids[k]`. No second field needed.
- Username / stable corpus identity. Derivable via `replay_id + slot → wire.static.usernames[slot]`. Easy to add if a downstream consumer wants it self-contained.

## 5. On-disk layout

```
replay-parser/data/intermediate/<2-char-prefix>/<id>.npz
replay-parser/data/intermediate/<2-char-prefix>/<id>.meta.npz
```

- **Sharded** by 2-char prefix of `<id>`. Generals.io IDs are base-~64, so ~1024 dirs × ~170 files each at corpus scale. Flat dir works on APFS but `ls` and `rsync` degrade.
- **`np.savez_compressed`** (per-array zip-deflate inside `.npz`). No external `.gz` wrapper; standard tools handle it.

**Path commitment is tentative** — revisit after smoke-testing on a handful of fixture replays. Renaming 170k files later is annoying but bounded.

**Storage envelope.** The `replay-parser-design.md` §3 / appendix A.1 estimate of 50–65 GB compressed was computed with a per-timestep cities mask included. The mask is highly compressible (mostly-zero bool), so the actual delta from dropping it is modest. Smoke test should report the on-disk size of the sample game so the first real data point lands before scaling.

## 6. Implementation seams

### 6.1 `sim_core` (Rust)

Additions per §3:
- `actions_source/dest/is50: Vec<Vec<...>>` fields + sentinel-push hook at `step()` top + record-overwrite at `select_candidates`.
- `cities_present_at: Vec<i32>` parallel to `cities` + `cities_present_at()` getter.

### 6.2 `replay-parser/replay_parser/output.py` (new)

Two functions:
```python
def write_sim_output(state: sim_core.State, replay: ReplayData, out_path: Path) -> None: ...
def write_metadata(metadata: dict[str, np.ndarray], out_path: Path) -> None: ...
```

`write_sim_output` is sim-aware only — knows nothing about perspectives or DB. `write_metadata` is a thin wrapper around `np.savez_compressed`.

### 6.3 `replay-parser/replay_parser/metadata.py` (new, stubbed initially)

```python
def build_metadata(
    state: sim_core.State,
    replay: ReplayData,
    perspective_player_ids: list[PlayerIndex],

    rolling_top3_rate: list[float] | None = None,
    prior_games_count: list[int] | None = None,

    ...
```

Stub semantics: `None` → fill with `-1.0` / `-1`. The real rolling-rate computation is a per-player chronological DB walk — out of scope for this session.

## 9. Decisions worth flagging (load-bearing)

These choices are easy to second-guess later, so the rationale is captured once here.

### 9.1 Why sibling files, not bundled

Bundled `.npz` per game looks simpler — one file, one load. But it couples regeneration: any metadata change rewrites the bulky sim half. The asymmetric churn rate is the load-bearing factor: sim output is "compute once and freeze" once validated; metadata definitions will iterate (rolling window, curated list updates, new derived fields). Splitting matches the churn shape.

### 9.2 Why per-player `[P, T-1]` action arrays, not `[K, T-1]`

`sim_core` has no perspective concept — recording for all `P` keeps the Rust surface free of perspective state. The dataloader gains flexibility to revise perspective selection without re-parsing. Storage and runtime cost is negligible (`K ≈ P` always; `K ≤ P` strictly).

### 9.3 Why `cities` + `cities_present_at`, not a per-timestep mask

Cities never decrease, so a per-timestep `[T, HW]` mask carries redundant information. Parallel arrays `cities[C_final]` and `cities_present_at[C_final]` encode the same content at <1% of the byte cost. Baking the `+1` snapshot-visibility conversion at parse-write time means the dataloader does a plain `cities[cities_present_at <= t]` slice with no pre/post-increment convention to remember.

### 9.4 Why action recording in `sim_core`, not Python

The recording moment isn't a snapshot — it's mid-step state (post-AFK, pre-resolution). Python recreation would reimplement AFK + buffer-drain + `is_valid` logic and keep it in sync with Rust. Rust-side is ~5 LOC where the state already lives.

### 9.5 Why `actions` length = `T-1`, not `T` (with terminal sentinel)

Terminal snapshot has no paired action by definition. Encoding the absence structurally (different array length) is clearer than padding with a sentinel column that means the same thing. Trade: dataloader iterates `0..T-1` and naturally stops one short.

### 9.6 Why `-1` sentinel across all three action arrays

`source` and `dest` are int16 with valid range `0..HW-1`, so `-1` is naturally out-of-domain. `is50` is wire-aligned to `{0, 1}` (no-split / split); using `int8` instead of `uint8` makes room for the same `-1` sentinel rather than picking an arbitrary in-domain value like `255`. One sentinel rule for all three arrays — each is self-evident in isolation, no AND-masking required to read any single one.

### 9.7 Why `np.savez_compressed`, not external gzip

`.npz` is already a zip container; `savez_compressed` flips per-array deflate. External gzip wrapper would compress the zip — redundant work that hurts compression ratio (zipped-then-gzipped doesn't compose well).
