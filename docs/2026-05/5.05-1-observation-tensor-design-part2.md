## 1. Context and scope

Part 1 settled the "current frame" channels: map structure, ownership, army magnitude, broadcast scalars, scoreboard-derived features, and dense recent spatial history. That covers what the agent can see *right now* and what's changed *over the last several timesteps*.

This document designs everything else — the information a human player carries in their head across the game that isn't visible in any single frame:

- **What was there before the fog came back?** Remembered ownership, army counts, and the reliability of those memories.
- **What kind of structure is that?** Permanent resolution of mountain vs. city once explored.
- **Which opponents have I encountered?** Contact history per opponent.
- **Who eliminated whom?** The capture graph — globally announced, strategically critical.
- **How should newly-visible tiles interact with the history channels?** A deferred encoding detail from Part 1.

After designing these channels, the document assembles the complete observation tensor and evaluates it as a whole.

---

## 2. Design principles

All principles from Part 1 §1 carry forward. This session added:

- **Persistent knowledge vs. current visibility.** When a fact is permanently true once learned (structure type, contact history), encode it as persistent state rather than adding a separate memory channel alongside a current-visibility channel. Prefer the simpler representation when the network can recover any needed distinctions via cross-referencing with other channels (e.g., `fog_cells`).
- **Categoricality-aware encoding.** Integer-encoded categorical values (where the numeric magnitude is meaningless) are appropriate for broadcast scalars and sparse spatial signals. They are inappropriate for dense spatial channels where the network must detect spatial patterns like regions and borders — these require one-hot encoding. The distinction turns on whether a conv kernel needs to perform spatial reasoning over the categorical values. (See §6.1 for the full analysis.)
- **Capture the full wish list, then trim.** Design channels based on information value first. Evaluate the channel budget as a complete picture rather than trimming prematurely during per-feature design.

## 3. Channel design decisions

This section covers the reasoning behind each new channel category. For the definitive channel list with encoding details, see the table in §4.

### 3.1 Structure persistence (0 new channels)

**Problem:** the Part 1 spec defined `mountains`, `cities`, `generals`, and `structures_in_fog` as reflecting current visibility. A mountain explored and then lost to fog would revert from `mountains` back to `structures_in_fog`. But structure type is immutable — once you know a fogged structure is a mountain, that knowledge never expires.

**Solution:** redefine the existing structure channels to carry persistent knowledge rather than current visibility:

- `mountains` = 1 for any cell *known* to be a mountain (currently visible OR previously explored). Once set, never unset.
- `cities` = 1 for any cell *known* to be a city. Once set, never unset.
- `generals` = 1 for any cell *known* to be a general. Once set, never unset (cleared only if that player is eliminated, at which point the general becomes a city — handled via capture events).
- `structures_in_fog` = structures whose type has *never been determined*. Shrinks monotonically as the agent explores.

**Why no new channels:** the distinction between "city I can see right now" and "city I remember but is currently fogged" is recoverable from `fog_cells`. If `cities[r,c] = 1` and `fog_cells[r,c] = 1`, it's a remembered city in fog. This is an AND operation on two co-located binary inputs — a single neuron in the first conv layer can learn exact weights for this (e.g., `+1 × cities + 1 × fog - 1.5`, positive only when both are 1). No information is lost, and the encoding is simpler than maintaining separate visibility and memory channels.

**Precedent:** strakam includes "revealed castles/generals positions" as a separate memory feature alongside the base visibility channels. This appears to be an implementation artifact (base obs = raw game state, memory = bolt-on layer) rather than a deliberate design choice — their revealed-structures channel lumps castles and generals together, suggesting it was a quick addition rather than a carefully reasoned encoding.

### 3.2 Contact history (7 new channels)

**Encoding:** per-opponent `opp_N_contacted` broadcast scalar. Binary, set to 1 once the agent has had vision of any tile owned by opponent N. Monotonically increasing — once set, never unset.

**Why per-opponent broadcast rather than a count:** "have I ever seen opponent 3?" is a qualitatively different question from "how many opponents have I seen?" The strategic distinction is between "opponent 5's location is completely unknown" (never contacted — they could be anywhere unexplored) vs. "I saw opponent 5 forty turns ago and they're now behind fog" (contacted — I have spatial information, even if stale). This per-opponent signal shapes attention allocation and threat assessment in ways a scalar count cannot.

**Why broadcast rather than spatial:** contact is a global fact about the agent's relationship with each opponent, not a spatial feature of any particular cell. Broadcasting makes it available at every position from layer 1.

**Redundancy check:** once the fogged cell state channels exist, contact is *theoretically* recoverable — if any cell on the board has `last_seen_owner = opponent N`, that implies prior contact. But recovering this requires a global OR-reduction across an entire spatial channel, which is exactly the kind of operation convolutions are bad at without many pooling layers. The broadcast scalar hands it over directly.

### 3.3 Capture events (7 new channels)

When a player is eliminated, the game announces it globally: "Player 1 captured Player 3." This is one of the most strategically significant events in FFA — the captor inherits all territory and armies (halved), the remaining player count drops, and every survivor must update their threat model.

**What the agent already knows vs. what's missing:** death itself is covered by existing channels — a dead opponent has `opp_N_army_count = 0` and `opp_N_land_count = 0`. The captor's identity is *partially* inferable from scoreboard correlation (some opponent shows a large positive land delta the same tick). But "partially inferable" isn't "encoded" — during chaotic mid-game phases with multiple simultaneous land changes, or when the capture happens entirely behind fog, the scoreboard signal is ambiguous.

**Encoding:** per-opponent `opp_N_captured_by` broadcast scalar, integer-encoded:

- `0` — still alive
- `-1` — captured by the agent (self)
- `1` through `7` — captured by opponent slot k

The negative value for self-capture preserves the sign convention from the ownership-transition channels (negative = self-involved), giving the network a consistent pattern across multiple channel types.

**Why identity matters (not just the fact of elimination):** the capture graph is a spatial reasoning tool. Example: if the agent has vision of opponent 2 (red) in the early game, and opponent 4 (blue) captures red, and opponent 6 (green) captures blue — then the agent can infer green's general is likely "three generals away," since red was adjacent, blue was adjacent to red, and green was adjacent to blue. Combined with explored territory and map geography, capture chains encode an approximate proximity graph of generals that is otherwise completely hidden behind fog.

**Why integer encoding works here (categoricality analysis):** `captured_by` is a broadcast scalar — the kernel sees one constant value paired with one weight. The decoding task is "learn a response function for one scalar." The signal is also sparse (at most 7 events in the entire game) and permanent. These properties are favorable for integer encoding. See §6.1 for the full categoricality analysis contrasting this with `last_seen_owner`.

**`captures_made` (dropped):** the brainstorm considered per-opponent capture counts (7 additional channels). Dropped because the snowball signal is largely redundant with scoreboard channels — a player who's made multiple captures will be visibly dominant in `opp_N_army_count` and `opp_N_land_count`. The mechanism (captures vs. organic growth) is slightly different information, but not 7 channels worth for v1. Additionally, deriving capture count from `captured_by` requires the network to perform equality-matching across 7 channels ("how many of these equal 3?"), which is genuinely hard for conv layers — so if we wanted the count, we'd need to provide it explicitly. The information value doesn't justify the cost.

### 3.4 Fogged cell state (19 new channels)

This is the core long-term memory block — the degraded snapshot of what the agent last observed at cells now hidden by fog. Five components, forming a coherent unit: *I saw [this owner] with [this many armies] [this long ago], and [whether I've ever seen it] and [which opponents have explored it].*

#### 3.4.1 `historically_seen` (1 channel)

Binary spatial mask. 1 for any cell the agent has ever had vision of, 0 otherwise. Persistent — once set, never unset.

**Purpose:** separates the map into cells the agent has *some* memory of vs. cells that are completely unknown. The most direct use case is general-hunting: when pushing into enemy territory, opponents will reclaim tiles behind you, causing you to lose ownership and vision. Without this channel, the agent can't distinguish "fogged area I've already swept" from "fogged area I've never explored." It would waste moves re-searching corridors it already checked.

**Relationship to `turns_since_seen`:** technically redundant — `turns_since_seen` with a -1 sentinel for never-seen cells carries the same binary distinction. But `historically_seen` is a single binary mask that's trivially cheap and provides the signal in the simplest possible form. Belt and suspenders.

#### 3.4.2 `opp_N_has_seen` (7 channels)

Per-opponent binary spatial masks. For each opponent N, 1 at any cell the agent has ever observed to be within opponent N's Moore-neighborhood vision range. Monotonically increasing.

**Derivation:** whenever the agent sees a tile owned by opponent N, all 8 neighbors of that tile plus the tile itself are marked as "seen by opponent N." This is a derived feature computed by the parser, not a raw game observation.

**Why per-opponent rather than aggregated:** "some opponent has seen this area" is much less actionable than "opponent 3 specifically has seen this area." The most critical use case is general visibility — once an opponent has seen your general, they can launch a targeted attack. Knowing *which* opponent has this information lets you assess the specific threat direction and strength. Similarly, knowing which opponents have scouted a particular corridor informs predictions about future attack paths.

**Potential redundancy with `last_seen_owner`:** if cell (r,c) has `last_seen_owner = opponent 3`, a first-layer 3×3 conv kernel can infer opponent 3 had vision of the surrounding neighborhood. This recovers the most common case but misses historical owners who were overwritten — if opponent 3 owned a cell, then opponent 5 took it, `last_seen_owner` shows only opponent 5. The `opp_N_has_seen` mask retains the cumulative information. Flagged as a trimming candidate in §7 if channel budget is tight.

#### 3.4.3 `last_seen_owner` (9 channels)

One-hot spatial encoding of the last observed owner at each cell. 9 channels: self, 7 opponents, neutral.

For currently visible cells, this matches the current ownership channels (redundant but harmless). For fogged cells, it records the owner at the moment fog closed. For never-seen cells, all 9 channels are 0.

**Why one-hot rather than integer:** this is where the categoricality analysis (§6.1) is most consequential. `last_seen_owner` is spatial and dense — after any meaningful exploration, most of the board has a non-zero value. The critical spatial patterns the network needs to extract are things like "this is a region belonging to opponent 3," "here's a border between opponent 3 and opponent 5," "this neutral corridor separates two opponent territories." With integer encoding, detecting "a region of value 3" requires the network to approximate an equality test using linear combinations plus ReLU — possible but takes multiple layers and is fragile. Detecting a *border* (adjacent cells with different integer values) is even harder — "value 3 next to value 5" and "value 2 next to value 6" are both borders but look completely different numerically.

With one-hot, all of this becomes natural spatial reasoning. Opponent 3's remembered territory is a connected blob in channel 3. A border is an edge in one channel adjacent to an edge in another. These are exactly the spatial patterns convolutions are designed to detect.

**9 channels is expensive but justified.** This is the most channel-hungry single encoding decision in the tensor. The remembered ownership map is the foundation of the agent's mental model of the game behind fog — it's what route planning, threat assessment, and the capture-chain proximity reasoning are built on. Every other memory feature is either cheaper or less decision-relevant.

#### 3.4.4 `last_seen_armies` (1 channel)

Scalar spatial channel. Log-scaled army count last observed at each cell. Uses `log(1 + armies)`, the same transform as `army_magnitude` in Part 1, so remembered and current armies are on the same scale and directly comparable by the network.

For currently visible cells, matches `army_magnitude` (redundant but harmless). For never-seen cells, uses -1 sentinel.

**Purpose:** the basis for estimating what the agent would encounter if it pushed back into fogged territory. "Opponent 3 controlled this area" is useful; "opponent 3 had 45 armies concentrated on this tile" is much more useful for deciding whether to attack through there.

**Reliability depends on cell type and staleness.** A remembered army count on a neutral empty tile is still exactly correct (neutral tiles never change). A remembered count on an opponent's city is at minimum the remembered value + `turns_since_seen` (from per-turn production). The network has the information to learn this interaction — it can cross-reference `last_seen_armies` with `turns_since_seen` and the structure channels. This is a multiplicative interaction that takes two layers to compute, but the components are all present.

**Strakam precedent:** the paper explicitly flags the absence of remembered army counts as a known limitation: *"the agent does not know ... what were the exact army counts of these cells at that time."*

#### 3.4.5 `turns_since_seen` (1 channel)

Scalar spatial channel. Number of timesteps since the agent last had vision of each cell, log-scaled via `log(1 + turns)`.

For currently visible cells: 0. For never-seen cells: -1 sentinel (applied *after* the log transform — don't feed -1 through `log(1 + x)`). For fogged previously-seen cells: incrementing counter, log-scaled.

**Purpose:** the confidence weight on the entire memory block. Without this, every remembered observation looks equally trustworthy — a cell seen 3 turns ago is treated identically to one seen 300 turns ago. Human players naturally discount stale information, and the network needs the same ability.

**Why log scaling:** the difference between 5 and 10 turns of staleness matters far more than between 305 and 310. Log compression matches the actual information decay pattern — early staleness is differentiated more finely than late staleness. Consistent with the log-scaling convention used for army counts throughout the tensor.

**Strakam precedent:** the paper explicitly flags the absence of temporal information as a known limitation: *"the agent does not know when it uncovered certain parts of the map."*

## 5. How different channel groups divide labor across timescales

An important insight from this session: the observation tensor separates into three temporal layers, each serving a distinct purpose. Understanding this division clarifies why all three are necessary and why they aren't redundant.

**Immediate state (§A–§F, 46 channels).** What the agent can see or know right now, this timestep. Map structure, ownership, armies, scoreboard, scoreboard-derived inferences. These channels are updated every tick from the raw game state. They answer: *what is the current situation?*

**Tactical history (§G, 2N channels).** What changed over the last several seconds of game time. Ownership transitions and army deltas at per-timestep granularity. This enables coherent multi-step execution — tracking an attack in progress, detecting an approaching army stack, following the progression of a border skirmish. The window is short (seconds, not minutes) and the resolution is fine (per-tick, per-cell). It answers: *what's actively happening and how is it evolving?*

**Strategic memory (§H–§I, 33 channels).** What the agent has learned over the course of the entire game. Contact history, capture graph, remembered ownership, staleness estimates. This persists indefinitely at fixed channel cost, but at low resolution — no per-tick granularity, just the last snapshot before fog closed. It answers: *what do I believe about the parts of the game I can't see?*

These three layers operate on fundamentally different timescales and serve fundamentally different purposes. Making the tactical history window larger doesn't substitute for strategic memory — it just gives a slightly longer tactical window at 2 channels per timestep. Conversely, strategic memory can't replace tactical history — it lacks the temporal resolution needed for reactive combat decisions. The immediate state channels are the foundation both other layers are interpreted against.

This division of labor also clarifies an otherwise puzzling aspect of strakam's design: why does an agent with only 7 timesteps of move history still play effectively? Because the short history window handles tactical coherence (its actual job), while the separate memory features handle strategic context. Extending the history to 50 timesteps wouldn't make the strategic memory unnecessary — it would just be an expensive way to encode a slightly longer tactical window.

### Why strakam's N=7 doesn't constrain our choice

Strakam's "last 7 moves" is *action history* — the literal moves each player made. Our dense history is *observation history* — per-cell ownership transitions and army deltas across the entire visible board at each timestep. The information content per channel is fundamentally different. A single timestep of our observation history captures all visible changes simultaneously — multi-front combat, land ticks, city production, opponent movements in field of view. Strakam's encoding captures one move per player per step. If strakam found 7 sufficient for action history, it doesn't directly constrain what we need for observation history. They solve different problems at different information densities.

## 6. Key encoding analysis

### 6.1 When integer-encoded categoricals work vs. when they don't

Several channels in this tensor encode categorical information (opponent identity) as numeric values. Whether this works depends on the channel's properties:

**Integer encoding works when:**

- **Broadcast** — the kernel sees one constant value paired with one weight. The decoding task is simply "learn a response function for one scalar."
- **Sparse** — few non-zero values across the game. Non-zero values are isolated signals, not neighborhoods of related values the kernel must jointly interpret.
- **Redundant with other channels** — even if the kernel can't perfectly decode the fractional/integer value, other channels carry supporting information.

Channels meeting these criteria in our tensor: `opp_N_captured_by` (broadcast, at most 7 events per game, permanent).

**Integer encoding fails when:**

- **Spatial and dense** — after meaningful exploration, most cells have values. A 3×3 kernel sees a *neighborhood* of categorical values and must detect spatial patterns.
- **Spatial reasoning required** — the network needs to identify regions ("connected area owned by opponent 3"), borders ("transition between opponent 3 and opponent 5"), and corridors. With integer encoding, detecting "a region of value 3" requires approximating an equality test via linear-combination-plus-ReLU. Detecting a border requires recognizing that numerically dissimilar pairs (3 vs. 5, 2 vs. 6) are semantically similar (both are "different-owner boundaries").

The channel meeting these criteria: `last_seen_owner`. One-hot encoding gives each opponent a dedicated channel where their remembered territory forms a connected blob — exactly the spatial pattern convolutions are designed to detect.

**The ownership-transition channel in §G is a middle case** that works despite being spatial, because it is sparse (ownership changes are rare per-timestep events — most cells are 0), and the hierarchical structure provides large gaps between primary categories (sign separates self-involved from informational, magnitude separates neutral from opponent). This matches the properties of DeepNash's dense move encoding, which demonstrated empirical success with a similar scheme.

### 6.2 Sign convention

Negative values consistently encode "self-involved" across multiple channel types:

- `ownership_transition`: -1 = agent lost this tile
- `captured_by`: -1 = agent captured this opponent

This gives the network a learnable pattern: across channel types, negative = personally relevant, urgent. Positive = informational, about others.

## 7. Open questions and deferred details

### 7.1 Dense history window size (N)

Not committed. The tactical purpose of the window (§5) suggests a moderate size — enough to track complete tactical exchanges, not so much that channels are wasted on increasingly irrelevant history. FFA's more complex tactical situations (multiple simultaneous fronts, more opponent interactions in field of view) argue for a larger window than strakam's 7.

**Recommendation:** make N a hyperparameter from day one. The parser generates any window size; the network reads the channel count from config. Start at N=10–12, tune empirically. The channel budget at N=10 is 99 (comfortable); at N=15 it's 109 (within range but worth scrutinizing).

### 7.2 Newly-visible-tile handling for army deltas

**Problem:** army delta channels encode `army[t-k] - army[t-k-1]` per cell. When a cell transitions from fogged to visible, there's no previous observation to diff against. The delta is undefined.

**Design space considered:**

- **Zero** — treat as "no change." Simple, but conflates "nothing happened" with "first contact with enemy position."
- **Sentinel value** — marks "delta undefined." Clean semantically, but newly-revealed cells contribute nothing to history.
- **Absolute value instead of delta** — for newly-visible cells only. Mixes two semantics in one channel.
- **Per-timestep newly-visible mask** — N additional binary channels. Expensive.

**v1 decision: zero.** The most decision-relevant information from newly-revealed tiles is *ownership*, not army magnitude. The ownership-transition channel handles the newly-visible case cleanly — a cell going from fogged to "owned by opponent 3" is a genuine ownership transition. The absolute army count is captured by `army_magnitude` in the current frame. What's lost is the temporal detail of *when during the history window* the first observation occurred, which is a marginal signal for v1.

This is a known simplification with a clear upgrade path if training reveals that first-contact army information matters.

### 7.3 Trimming candidates if channel budget is tight

If the total channel count needs to come down, these are the candidates in rough order of expendability:

1. **`opp_N_has_seen` (7 channels).** Partially recoverable from `last_seen_owner` via first-layer 3×3 conv: if a cell's last-seen owner is opponent N, the neighborhood was within opponent N's vision. This misses overwritten historical owners but captures the most common case. Removing these 7 channels saves meaningful budget.

2. **`historically_seen` (1 channel).** Redundant with the -1 sentinel in `turns_since_seen`. Removing saves only 1 channel; probably not worth the loss of the clean binary signal for general-hunting.

3. **Reducing N.** Each unit decrease in N saves 2 channels. Going from N=12 to N=10 saves 4 channels and likely has minimal impact on tactical coherence.

4. **`last_seen_owner` one-hot (9 channels) → coarser encoding.** A 3-channel split (self / neutral / any-opponent) plus integer opponent identity saves 6 channels but sacrifices the per-opponent spatial reasoning that is the primary justification for one-hot. This is a significant capability loss and should be a last resort.

### 7.4 Other open items

- **Normalization scheme for broadcast scalars.** Part 1 notes that `self_army_count`, `self_land_count`, and the per-opponent equivalents need normalization but doesn't specify the method. Constant divisor (e.g., typical late-game values), min-max scaling across the current game, or log-scaling are all viable. Should be resolved before parser implementation.
- **Smoothing window k for scoreboard-derived features.** The city-inference and land-delta features each have a tunable k. Starting values proposed in Part 1 (j≈3, k≈8–10 for city inference) but not committed.
- **`passable` channel scope.** Part 1 defines this as persistent once explored. With the structure persistence update, `passable` can now be computed from `mountains` alone (passable = NOT mountains, for all resolved cells). Whether to keep it as an explicit channel or derive it is an implementation simplification question.
