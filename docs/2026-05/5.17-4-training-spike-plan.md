## 1. Framing

We've been methodical through replay-collection, parser, sim, and parsed-corpus stages. Now we move agile: **build the smallest pipeline that produces a real BC model checkpoint**, using inherited design decisions rather than refining them.

Spike target bar (per doc A §3.2):
- One training run that doesn't crash end-to-end.
- Train + val loss curves recorded across multiple epochs.
- Top-k next-action accuracy on a held-out split.
- Action-distribution sanity (model doesn't collapse to one action).
- At least one move-review eyeball check (user is a top-50 FFA player; their qualitative read is one of our highest-leverage eval signals).

What we explicitly **don't** need from the spike:
- Optimal architecture / hyperparameters — width sweep, value-head shape, contraction depth are deferred.
- Cloud H100 — local M1 (MPS) is the primary target; cloud only if local proves out and we want a longer run.
- Live deployment harness — Phase 1 success is judged on output-move quality, not ladder play.
- Auxiliary heads, player conditioning, offline-RL extensions.

The spike is **proof-of-concept**: did the inherited design actually train? Once we have that signal, we can decide what to push on next.

## 2. Where we are

### 2.1 What's shipped

- **Corpus:** ~180k v15+ FFA replays in `replay-collector/data/generals.sqlite`. Per-game intermediate written: **~165k games × 2 sibling `.npz` files, 9.2 GB on disk** under `replay-parser/data/intermediate/<2-char-prefix>/`.
- **Per-game sim file (`<id>.npz`)**: snapshots `ownership` `[T, HW]` int8 + `armies` `[T, HW]` int16, per-player `actions_source/dest` `[P, T-1]` int16, `actions_is50` `[P, T-1]` int8 (`-1` pass sentinel across all three), event lists, city presence tracking. Schema in doc A §2.2.
- **Per-game meta sidecar (`<id>.meta.npz`)**: per-perspective `placement`, `stars_at_start`, `elim_timestep`, `rolling_1st_rate`, `rolling_top3_rate`, `prior_games_count`, `perspective_player_ids` (`[K]` slot indices into the sim file), `sim_core_version` git stamp. Real rolling stats (200-game window) — not stubs. Schema in doc A §2.3.
- **Pipeline code**: `sim_core` (Rust) → `replay_parser.parser.parse_replay` → `output.write_sim_output` + `metadata.build_metadata`. Multiprocessing corpus driver with two-stage filter + skip-CSV logging.
- **Validation**: sim integration test green (16/16), output smoke test green (3/3), ranking-match validator at ≥99.85% over n=11.6k sample (residuals are pre-existing edge cases, not regressions).
- **Schema drift notes**: meta sidecar shipped with `sim_core_version <U24` (not `<U16` per 5.16-3 §4.1, to fit the `-dirty` suffix) and an added `perspective_usernames <U32 [K]` field. Treat the code as canonical; 5.16-3 §4.1 is design-time.

### 2.2 What's missing for the spike

Nothing on the parsing/sim side. The gap is:
1. **PyTorch dependency** — not in any `pyproject.toml`.
2. **Dataloader** — channel assembly from the per-game intermediate, fog computation, action encoding, augmentation, train/val split.
3. **Model code** — Pyramid Module / U-Net + policy + value heads.
4. **Loss + training loop**, **eval metrics**, **logging + checkpointing**.

### 2.3 Things we already know that constrain the spike

- **Pass-frame rate may be high.** One sample (3KXOs-pPx) showed 56.7% pass frames vs. the parser-design napkin of 30–45%. This single sample is plausibly an open-turtle outlier, but a corpus-level sweep before locking the pass head is cheap (30 minutes of `np.load`) and load-bearing for the loss-balance call.
- **Curated list has 334 names**, K per game averages 1–2. Stratified-by-player split is impractical at this scale; game-id-level split is the right v1 call.
- **Noise floor is already applied** at parse time. The corpus on disk is pre-filtered to `prior_games ≥ 50 ∧ 1st-rate > 12.5% ∧ top-3 > 37.5%`. No additional training-time filter needed for v1.
- **PyTorch 3.14 wheels available** (`torch 2.12.0` ships `cp314`). Spike isn't blocked by missing wheels.

## 3. The plan

Four phases, sized so each lands a verifiable artifact. Ordering is by dependency. Phases 1–2 are local-only and free; phase 3 is the first run worth calling a "spike"; phase 4 is the eval pass.

### Phase 1 — Environment + dataloader skeleton

**Deliverable:** a `training/` workspace subpackage with `torch` as a dep, an `IterableDataset` that yields `(obs_tensor, action_target, value_target, action_mask)` tuples from the per-game intermediate, and a smoke test that pulls 100 batches without crashing.

Steps:

1. **Add `training/` subpackage**, parallel to `replay-parser/`, `sim-core/`, `replay-collector/`, `utils/`. Same workspace pattern. Mirror `replay-parser/scripts/run_corpus_driver.py` with `training/scripts/run_train.py` as the CLI entry point.
2. **`uv add torch`** in `training/pyproject.toml`. Confirm `import torch` works on Python 3.14; confirm `torch.backends.mps.is_available()` on the user's M1.
3. **One-pass MPS benchmark** (5.05-3 action-item carryover): forward + backward pass on a placeholder U-Net + 64-batch input. Wall-clock the first epoch's worth. If MPS shows ≥5× speedup over CPU, we're in business; if it doesn't, drop to CPU for the spike and revisit (an MPS fallback to CPU on some op is the most likely culprit).
4. **Dataloader skeleton** — single-worker `IterableDataset`. Per call:
   - Pick a random `<id>.npz` + `<id>.meta.npz` pair from a precomputed train-set file list.
   - Load both files (`np.load`).
   - For each `k` in `perspective_player_ids`, walk `t = 0 .. T-2` and emit one frame per timestep.
5. **Frame emission**, simplest version (channels added incrementally — see Phase 2):
   - Inputs (placeholder): `ownership[t]` one-hot per slot (8 channels), `armies[t]` log-scaled (1 channel), `mountains` mask (1), `cities` mask up to t (1). 11 channels total, no fog yet.
   - Action target: encoded from `(actions_source[k, t], actions_dest[k, t], actions_is50[k, t])`. See §4.3 for the encoding.
   - Action mask: `[H, W, 9]` boolean of legal moves.
   - Value target: `placement[k]` (1..8), converted to 0..7 class index.
6. **Smoke test**: pull 100 batches through the dataloader; assert shapes, dtypes, no NaN/inf, action target is in-range, mask covers ≥1 legal cell per non-pass frame.

Exit criterion: skeleton dataloader pulls batches off the real corpus without crashing.

### Phase 2 — Full obs tensor + model + loss

**Deliverable:** a full training step (forward + loss + backward + optimizer step) on a single batch, on M1 MPS, with the v1 spike obs tensor and the half-width Pyramid Module.

Steps:

1. **Fog computation in the dataloader** — the load-bearing new piece. For each frame, per perspective:
   - **Vision mask:** the union of the Moore neighborhood (8 surrounding cells + the cell itself) around every cell owned by the perspective player at this snapshot. (`generals-io-game-mechanics.md` line 81.)
   - **Running per-cell memory** (updated as the trajectory walks forward):
     - `last_seen_owner` (9-way one-hot: self / 7 opponents / neutral).
     - `last_seen_armies` log-scaled.
     - `turns_since_seen`.
     - `historically_seen` mask.
   - **Persistent structure channels:** `mountains` (static), `cities` (monotonic from `cities_present_at <= t`), `generals` (monotonic; cleared on capture/neutralize via `capture_events` / `neutralize_events`).
   - **Distance-from-known-generals** (8 channels): BFS on the known-passable graph per general; log-scaled; -1 sentinel for unreachable or unrevealed. Always recompute per frame.
   - **Capture-event channels** (`opp_N_captured_by`, 7 channels): from `capture_events` and `death_events`.
   - **Contact channels** (`opp_N_contacted`, 7 channels): monotonic, set once any tile of opp N has been seen.
   - **Dense recent history** (2N channels): ownership-transition + army-delta for the last N timesteps. **v1 spike: N=5.**
2. **Channel count for v1 spike:** 89. Per `observation-tensor-design.md` §2.1 the baseline is 86 + 2N (N=10 → 106); v1 spike drops the 7 `opp_N_has_seen` (per `5.05-1` §7.3 #1) and uses N=5. If the dataloader gets bogged down, drop to N=0 (79 channels) — the dense history block is a defensible v1 cut.
3. **Slot canonicalization** — dataloader-side, per emitted frame:
   - Permute the per-player axis so the perspective player is at slot 0; opponents at slots 1..P-1 in canonical-by-ID order (stable across frames within a trajectory).
   - All per-opponent channels (`opp_N_owned`, `opp_N_army_count`, `opp_N_captured_by`, `opp_N_contacted`, `last_seen_owner_opp_N`, dense-history ownership-transition values) are permuted in lockstep.
4. **Map padding:** pad every map to 32×32. Pad value: -2 (mountain sentinel for ownership; 0 for other channels). Action mask zeros out padded cells.
5. **Action encoding:**
   - For non-pass: `(cell, direction, split)`. `cell = source`, `direction = decode(dest - source, map_width)` ∈ {N, S, E, W}, `split = is50`.
   - For pass: a separate scalar pass logit (see §4.3 for why we split this out).
6. **Pass-frame sweep — measure first.** Before locking the loss, run a 30-minute `np.load` pass over ~500 random meta+sim file pairs to compute the corpus pass-rate distribution. If median game pass rate is in the 30–45% range (design napkin), we use the simple loss form. If 50%+, we may want to weight pass frames down. **Output: a one-line empirical number.**
7. **Model code:** small Pyramid Module per `network-architecture-design.md` §3, §4.
   - **Widths: 128 / 128 / 160 (the 0.5× variant)** — fast iteration on M1, ~4× compute savings over the inherited 256/320 baseline. Sweep to inherited widths post-spike if results are promising.
   - **Contraction depth: 2.** 32² → 16² → 8² → 16² → 32² (cleaner than the 25→13→7 path).
   - ResBlocks: C → C/2 → C with 1×1 residual on stride. N=2 outer, M=2 middle, M=2 inner.
   - Skip connections at each level.
   - LayerNorm + ReLU (default-inherited).
8. **Policy head:** small Pyramid Module → `Conv → 8` (4 directions × 2 splits) channels @ 32×32. **Plus a separate scalar pass logit** (pooled from the trunk). Masked softmax over legal moves at training/eval time. Why split pass out: "per-cell pass" is semantically odd, and the pass logit can carry its own gradient signal regardless of the per-cell distribution. Falls back cleanly to the architecture-doc 9-channel form if the empirical rate from step 6 says we need a different loss balance.
9. **Value head:** small Pyramid Module → global avg pool → linear → 8-way softmax over placement (per network-arch §3.2 tentative).
10. **Loss:** `policy_ce + λ * value_ce + μ * pass_bce`. Initial `λ = 0.5`, `μ = 1.0`. Pass BCE uses the (was-pass / wasn't-pass) binary label; policy CE is computed only on non-pass frames.
11. **Optimizer:** AdamW, `lr=3e-4`, `weight_decay=1e-4`, `betas=(0.9, 0.999)`. No warmup, no decay schedule. (Locking these so later experiments aren't reverse-engineering "what were the defaults.")
12. **Single-batch overfit sanity:** train on one fixed batch repeatedly; loss should drop to near-zero in a few hundred steps. If not, gradients are broken — debug here, not in the corpus run.

Exit criterion: single-batch overfit reaches near-zero loss; full-batch forward+backward runs on M1 MPS without falling back to CPU on some critical op.

### Phase 3 — Training run on a corpus subset

**Deliverable:** a model checkpoint produced from a real (not single-batch) training run, with train + val loss recorded across multiple epochs.

Steps:

1. **Train/val split** — game-id-level, deterministic via `hash(replay_id) % 100 < 5` (or fixed-seed shuffle over `sorted(replay_ids)`). Saved to a JSON file in `training/data/splits/v1.json` for reproducibility.
2. **Subset selection** — for the first run, take ~10–20k games (~6–13% of the corpus). Estimated wall-clock at half-widths:
   - M1 MPS, 11k games × 500 frames/game = 5.5M samples; ~5–10 ms/batch at batch 64 → ~6 minutes/epoch raw forward+backward time. With dataloader overhead realistically 3–5× slower → ~20–30 min/epoch. 3–5 epochs in an evening of wall-clock.
   - Cloud H100 (only if we go cloud): ~1–2 hr per epoch at inherited widths, ~15–30 min at half-widths.
3. **Augmentation:** board symmetry (D₂ for rectangular, D₄ for square; ~4–8× effective). **Slot permutation deferred** for v1 spike — defensible because (a) canonical-by-ID slot ordering already gets implicit randomness from generals.io's unstructured slot assignment (5.05-3 §3.3), and (b) implementing slot-permutation in the dataloader is extra code we don't strictly need to verify the pipeline works.
4. **Direction-channel handling under board symmetry:** N/S flip under v-flip; E/W flip under h-flip; both under 180° rot. Unit-test this — it's the highest-blast-radius augmentation bug.
5. **Training loop:**
   - Outer: epochs. Inner: dataloader batches.
   - Per batch: forward → loss → backward → optimizer step → metric accumulators.
   - Per epoch end: evaluate on val split; log to JSONL; save checkpoint.
6. **Logging:** stdout + JSONL file per run at `training/data/runs/<timestamp>/`. No wandb for the spike. Capture: hyperparams, train/val loss per batch + epoch, top-1 + top-3 action accuracy, action-distribution histogram, pass-fraction, model + parser git SHAs.
7. **Checkpoints:** `torch.save({state_dict, optimizer, epoch, step, hyperparams})` every epoch + at end. Per-run dir `training/data/runs/<timestamp>/checkpoints/`.

Exit criterion: training run completes, train loss visibly decreases, val loss recorded, top-k accuracy beats random (1/9 → ~11% per-direction; with action masking, probably much higher).

### Phase 4 — Eval pass

**Deliverable:** an honest read on whether the model learned anything.

Steps:

1. **Held-out next-action accuracy** — top-1 + top-3 across the val split. Cheap, automated; already in phase 3's per-epoch eval.
2. **Action-distribution sanity** — per-direction frequencies; pass rate; split rate. Compare to corpus-level priors. Catches collapse to one direction or always-pass.
3. **Value head calibration** — predicted vs. actual placement distribution on val. Bucket and bin.
4. **Move-review on saved positions** — pick ~10–20 positions from val games (mix of early/mid/endgame), dump the model's top-3 moves, eyeball with the user. **Highest-information signal we have for "does the model play sensibly."**
5. **Inspection of failure modes** — pull positions where the model is most-confident-and-wrong. Are they pathological (mountain dead-ends, stuck in corners, ignored attacks)?

Exit criterion: the user has enough signal to decide what to push on next (architecture? data? obs tensor? eval? give up on BC and pivot?).

### 4.3 Action encoding — the round-trip detail

Wire moves are `(source_idx, dest_idx, is50)`. Convert to model targets:

```python
def encode(source_idx, dest_idx, is50, map_width):
    if source_idx == -1:  # pass sentinel
        return PASS
    delta = dest_idx - source_idx
    if delta == -map_width: dir = N
    elif delta == map_width: dir = S
    elif delta == -1:        dir = W
    elif delta == 1:         dir = E
    else: raise ValueError(...)  # diagonal moves don't exist in generals
    return (source_idx, dir, is50)
```

Padding wrinkle: when the board is padded from `W=27` to `W_pad=32`, neighbor relationships shift. The encode function must use the *unpadded* map_width and then re-index `source_idx` into the padded grid (`padded_idx = (source_idx // W) * W_pad + (source_idx % W)`).

**Round-trip test** to write first: take known `(source, dest, is50)` from a recorded game, encode, decode, confirm equality. Catches off-by-one and direction-permutation bugs at the dataloader-unit-test layer rather than at the training-run layer.

## 5. Risks + mitigations

Ranked roughly by likelihood × impact:

| Risk | Mitigation |
|---|---|
| MPS falls back to CPU on some op, throughput tanks | Benchmark step in phase 1; fallback to CPU + smaller subset if necessary. The spike doesn't need MPS to succeed — it needs *training* to succeed. |
| Action encoding / direction permutation bug under augmentation | Round-trip unit test (§4.3) + augmentation-parity test before any corpus training. Highest-blast-radius bug type. |
| Fog computation is subtly wrong | Manually compute fog for one perspective at one frame on a small fixture; compare to dataloader output. Cheap check, catches most variants of "I confused current vision with persistent memory." |
| Pass-frame rate is much higher than expected | Empirical sweep step (§3 phase 2 step 6) settles it before the loss is locked. Pass head split out keeps the loss-weighting lever explicit. |
| Single-batch overfit doesn't reach near-zero | Debug at this stage, not in a corpus run. Gradient checks, init checks, loss-output inspection. |
| PyTorch 3.14 op coverage gap | Pre-checked: wheels available. If specific ops are missing, the error message will be obvious; fall back to a stable older torch if needed. |
| Map-size variability breaks BFS distance computation | Restrict to known-passable cells; mountains as deleted nodes per obs-tensor §2.1. Test on small + large boards. |
| 66 `skip:exception` cases from the corpus run are signal of broader sim issues | One-pass categorization (15 min: `csv` parse on `_skipped_*.csv`, count `type(e).__name__`). Defer the fix unless the bucket reveals something structural. |
| Schema relock surprises | The shipped meta-sidecar dtypes already drifted from 5.16-3 §4.1 (doc A §2.3). Code is canonical; rewrites are cheap (~6 min); don't gate the spike on schema work. |

## 7. What the spike won't answer

To set expectations: when the spike checkpoint exists, we'll know whether the *pipeline* works (data flow correct, gradients flow, loss decreases, the model produces non-pathological outputs). We **won't** know:

- Whether the inherited Pyramid Module is the right shape for FFA at our compute scale.
- Whether the v1 obs tensor channels all pull their weight.
- Whether the categorical-over-placements value head is the right call.
- Whether the half-width spike model can do anything interesting on its own.
- Whether the rolling-rate noise floor produces a corpus that trains a useful model.

These are all post-spike investigations, scoped against whatever the spike tells us.
