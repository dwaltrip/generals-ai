## 1. Overview

The parser's job: take a `.gior`-decoded wire array, replay the game forward via a NumPy + plain-Python simulator, and emit a compact per-game intermediate for the dataloader to consume.

Three properties shape every decision below:

1. **Sequential per-game CPU work.** No vectorization across games; parallelism is `multiprocessing.Pool` at the per-game level.
2. **Cell state in NumPy, event state in Python.** Grid mutations (ownership, armies, cities) are SIMD/C-level. Event sequences (deaths, captures, kills) are small Python lists.
3. **Parser is decoupled from obs-tensor schema.** Per design doc §3, channel assembly happens in the dataloader. Parser emits raw per-timestep grids and per-perspective action streams; no fog reconstruction.

## 2. Decisions resolved this session

Compact recap of the six open questions from the session prompt and the conventions that emerged. Rationale lives in `replay-parser-design.md`; this section just states the conclusions.

### Q1 + Q3 — Data model & aggregate caching (merged)

Hybrid layout:
- **Grid (NumPy):** `ownership: int8[H*W]`, `armies: int16[H*W]`, `cities_mask: bool[H*W]`.
- **Per-player (Python lists, len = num_players):** `generals`, `alive`, `has_kill`, `input_buffer`.
- **Game-level scalars + event lists:** `timestep`, `alive_count`, `updates_since_move`, cursors, `death_events`, `capture_events`.
- **Per-player aggregates** (army total, tile count) **recomputed on demand** via `np.bincount` — only when `lbSort` needs them (game end + the rare all-AFK fallback). Not maintained incrementally.

Pure-grid approach rejected: event-derived state (death order, has_kill) isn't recoverable from grid diffs. Per-player owned-tile lists rejected: bookkeeping cost on captures exceeds the SIMD mask op for land tick.

### Q2 — Move / AFK iteration

Columnar NumPy arrays per design doc §7. Per-tick walk via monotonic cursors (`moves_cursor`, `afks_cursor`) — no precomputed offset table; we never look back. Per-player buffers are `deque[MoveRowIndex]` storing row-indices into the columnar arrays (not unpacked tuples) — one source of truth for move data.

### Q4 — Per-perspective output

Action streams + per-perspective metadata only. No fog/obs materialization. Action stream is three dense parallel arrays per perspective: `(source: int16, dest: int16, is50: uint8)` per timestep, with `source = -1` as the pass sentinel.

### Q5 — Production / land-tick

Generals + cities use list-loops with a cell-level "produces" check (`ownership[t] >= 0`). Land tick is one NumPy mask op:

```python
state.armies[state.ownership >= 0] += 1
```

The `ownership >= 0` predicate naturally handles all three player states (alive / dead-not-neutralized / neutralized), so we don't need per-player gating.

### Q6 — Capture & cities-mask transitions

Two paths add to cities (general-capture, neutralize). Both are monotonic appends to `cities: list[int]` + `cities_mask: bool[H*W]` in lockstep. The general tile is excluded from the `replaceAll` mask naturally — combat resolves first, so the captor already owns it when `ownership == captured` is computed. Halving = `(a + 1) // 2` (matches bundle's `Math.round(a * 0.5)` for positive ints, and game-mechanics doc's "rounded upward"). `generals[p]` nulled to `-1` in BOTH capture and neutralize paths.

### Conventions adopted

- `captures[p]: int` → `has_kill[p]: bool`. Kill-count inheritance arithmetic is unnecessary for `lbSort`; the bool flips on capture and is monotonic.
- `capturedPlayers[]` lineage list dropped. `capture_events` event list (output-side) carries the same information.
- Wire `start`/`end` → internal `source`/`dest`. Wire `turn` → internal `timestep` (per mechanics-doc §3, "the field named `turn` is actually a half-turn"). Translation happens once at the decode boundary.
- Action stream records candidates **post-pop, pre-resolution**. This captures player intent for inward-first-chase cases where m1 cancels m2 — m2 was a real tactical decision and belongs in the supervised target.
- Python floor bumped to `>=3.14` (was `>=3.11.2`; `.python-version` already pins 3.14.4). Enables PEP 695 `type X = Y` syntax.

```python
    actions_dest:        np.ndarray   # int16[K, T_max+1]; -1 = pass sentinel
    actions_is50:        np.ndarray   # uint8[K, T_max+1]

    snapshots: SnapshotBuffer        # appends (ownership, armies, cities_mask) per tick
```

`build_initial_state(replay)` constructs all of the above from a decoded `ReplayData` (= `Moves` + `Afks` + `GameStatic`). Sequence mirrors bundle's `populateFromReplay` (5.11-2 §3.7):

1. Allocate grid; `ownership[:] = -1` (TILE_EMPTY); `armies[:] = 0`.
2. `ownership[mountains] = -2`.
3. For each `(idx, army)` in `(initial_cities, initial_city_armies)`: `cities.append(idx); cities_mask[idx] = True; armies[idx] = army`. (Ownership stays -1 — cities start neutral.)
4. For each player slot `p` with `initial_generals[p] != null`: `ownership[g] = p; armies[g] = 1; generals.append(g)`. If `initial_generals[p]` is null: `alive[p] = False`, `alive_count -= 1`, `generals.append(-1)`. (Per 5.11-2 §3.7 + §2.1, empirically zero in the filtered FFA corpus, but bundle handles it.)
5. For each `(idx, army)` in `(initial_neutrals, initial_neutral_armies)`: `armies[idx] = army`. (Ownership stays -1; not added to cities.)
6. Initialize per-player flags: `alive[p] = True` for valid generals; `has_kill[p] = False`; `input_buffer[p] = deque()`.
7. Initialize perspective slots from the curated-player list filtered to this game.
8. Pre-fill action streams with pass sentinels: `actions_source[:] = -1`, `actions_dest[:] = -1`, `actions_is50[:] = 0`.

## 6. Per-tick orchestration

`step.py` exposes one entry point (`step`) and decomposes the procedure into a small set of named helpers. Magic numbers from the bundle lift to module-level constants.

```python
# step.py

MAX_ALL_AFK_TIMESTEPS = 2000      # bundle line 67932
MAX_GAME_TIMESTEPS = 50000        # bundle line 67930


def step(state: State, replay: ReplayData) -> bool:
    """Advance one timestep. Returns False when the game has ended."""
    # Game-end short-circuit (bundle's isOver at start of nextTurn, 5.11-2 §3.1)
    if state.alive_count <= 1:
        return False

    process_pending_afks(state, replay)
    buffer_pending_moves(state, replay)
    candidates = select_candidates(state, replay)

    for i in candidates:                       # action stream records intent (pre-resolution)
        record_action(state, i, replay.moves)

    any_ran = resolve_and_execute(state, replay, candidates)

    # All-AFK / turn-cap fallback (5.11-2 §3.5; bundle line 67930–67937)
    state.updates_since_move = 0 if any_ran else state.updates_since_move + 1
    if state.updates_since_move > MAX_ALL_AFK_TIMESTEPS or state.timestep > MAX_GAME_TIMESTEPS:
        kill_all_but_strongest(state)

    state.timestep += 1
    apply_production(state)
    state.snapshots.append(state.ownership, state.armies, state.cities_mask)
    return True


def parse_replay(wire: list) -> PerGameIntermediate:
    replay = decode_wire(wire)
    state = build_initial_state(replay)
    state.snapshots.append(state.ownership, state.armies, state.cities_mask)  # t=0 initial
    while step(state, replay):
        pass
    return assemble_intermediate(state, replay)
```

The four sub-steps with non-trivial bodies are extracted; one-liners (game-end check, timestep increment, snapshot, action-recording loop, all-AFK tracking) stay inline since wrapping them adds a name without a contract.

### 6.1 `process_pending_afks(state, replay)`

Drain AFK events with `timestep <= state.timestep`. Stop mid-batch if a kill drops `alive_count` to 1 — don't fire further AFK events past game-end (design doc §11.1).

```python
def process_pending_afks(state, replay):
    while (state.afks_cursor < len(replay.afks.timestep)
           and replay.afks.timestep[state.afks_cursor] <= state.timestep):
        p = PlayerIndex(replay.afks.index[state.afks_cursor])
        if state.alive[p]:
            kill_player(state, p)
        else:
            try_neutralize_player(state, p)
        state.afks_cursor += 1
        if state.alive_count <= 1:
            break
```

### 6.2 `buffer_pending_moves(state, replay)`

Drain move events with `timestep <= state.timestep` into per-player input buffers. Pure cursor advance, parallel to `process_pending_afks`.

```python
def buffer_pending_moves(state, replay):
    while (state.moves_cursor < len(replay.moves.timestep)
           and replay.moves.timestep[state.moves_cursor] <= state.timestep):
        p = PlayerIndex(replay.moves.index[state.moves_cursor])
        state.input_buffer[p].append(MoveRowIndex(state.moves_cursor))
        state.moves_cursor += 1
```

### 6.3 `select_candidates(state, replay) -> list[MoveRowIndex]`

Pull at most one valid candidate per player; discard type-A invalids (source no longer owned at tick start) silently.

```python
def select_candidates(state, replay) -> list[MoveRowIndex]:
    candidates: list[MoveRowIndex] = []
    for p in range(state.num_players):
        while state.input_buffer[p]:
            i = state.input_buffer[p].popleft()
            if is_valid(i, state, replay.moves):
                candidates.append(i)
                break
            # else: discard — move was issued against now-stale state
    return candidates
```

### 6.4 `resolve_and_execute(state, replay, candidates) -> bool`

Compose V3 sort + dependency loop + execution. Returns `any_ran` for the all-AFK tracking in `step`.

```python
def resolve_and_execute(state, replay, candidates) -> bool:
    ordered = dependency_loop(
        priority_sort(candidates, state, replay.moves),
        state, replay.moves,
    )
    any_ran = False
    for i in ordered:
        if still_valid(i, state, replay.moves):    # defensive against type-B invalidation
            execute_attack(state, i, replay.moves)
            any_ran = True
    return any_ran
```

### 6.5 `apply_production(state)`

Per-turn (RECRUIT_RATE = 2) + land tick (FARM_RATE = 50). Bundle line 67942–67949.

```python
def apply_production(state):
    if state.timestep % 2 == 0:
        for gen in state.generals:
            if gen >= 0:
                state.armies[gen] += 1
        for c in state.cities:
            if state.ownership[c] >= 0:
                state.armies[c] += 1
    if state.timestep % 50 == 0:
        state.armies[state.ownership >= 0] += 1
```

**Snapshot timing.** Initial snapshot is the post-populateFromReplay state (no step has run yet). Subsequent snapshots are at the end of each step (post-production). Final timeline length is `T_final + 1` where `T_final` is the last step that ran.

**Game end via mid-step capture.** If a player's general is captured mid-step (during `resolve_and_execute`), `alive_count` drops to 1 but we don't short-circuit — the remainder of the step still runs (no live mover, production on the lone survivor's tiles, snapshot taken). The next step's game-end check exits the outer loop. Same final-snapshot semantics as the AFK path.

## 7. Move resolution

### 7.1 Candidate validation

`is_valid(i, state, moves)` — checked at buffer-pop time:

- `state.ownership[moves.source[i]] == moves.index[i]` (mover still owns source)
- `state.ownership[moves.dest[i]] != -2` (dest is not mountain)
- `state.armies[moves.source[i]] >= 2` (need ≥ 1 to send + ≥ 1 to leave)

`still_valid(i, state, moves)` — same predicates, re-checked at execution time (defensive against type-B invalidation from an earlier-resolved move in the same tick).

**Trust-the-wire assumption:** wire `dest` is always orthogonally adjacent to `source` for valid play (per format doc §"Move record"). Not re-validated. If this turns out false in some edge case, surface a `WireFormatError` and skip-and-log per design doc §7.

### 7.2 V3 priority sort (5.11-2 §2.4)

Sort key for each candidate move row index `i`, where `p = moves.index[i]`:

```python
(
    0 if state.ownership[moves.dest[i]] == p else 1,                    # defensive first
    0 if not (state.generals.get_for_other_player(p) == moves.dest[i]) else 1,
                                                                         # general-attacks last
    -int(state.armies[moves.source[i]]),                                 # larger source army first
    base_priority(i),                                                    # input-order tiebreak
)
```

**`base_priority` semantics:** since each candidate is the front of a different player's buffer, the bundle's per-player `inputBuffer` position is always 0 for all candidates. The actual tiebreak in the bundle resolves to player-index iteration order, but the exact line 67375–67421 read is on the deferred list (§12). Practical impact is rare (requires defensive class + general-attack class + source army all equal); confirm during implementation.

### 7.3 Inward-first dependency loop (5.11-2 §2.4)

After sort, walk through candidates picking the highest-priority move whose source tile is NOT the destination of any other candidate. Defer those that are. Cycle fallback at line 67415–67418: if no non-deferred move remains, take the highest-priority deferred move (the "1-in-10,000" fallback).

```python
def dependency_loop(sorted_cands, state, moves):
    remaining = list(sorted_cands)
    result = []
    while remaining:
        for j, m in enumerate(remaining):
            src = moves.source[m]
            blocked = any(moves.dest[other] == src for other in remaining if other != m)
            if not blocked:
                result.append(remaining.pop(j))
                break
        else:
            result.append(remaining.pop(0))   # cycle fallback
    return result
```

O(N²) over ≤ 8 candidates — trivial. Not worth optimizing.

## 8. Combat & capture

### 8.1 `attack(state, move_idx)`

Pure cell-level combat. No capture detection (caller's job).

```
move_reserve = ceil(armies[source] / 2) if is50 else 1
incoming     = armies[source] - move_reserve
armies[source] -= incoming                              # always subtracted
```

Then by case:
- **Friendly** (`ownership[dest] == mover`): `armies[dest] += incoming`. Ownership unchanged.
- **Hostile, defender holds** (`armies[dest] >= incoming`): `armies[dest] -= incoming`. Ownership unchanged. Equal-army case sits here (defender's advantage).
- **Hostile, attacker captures** (`armies[dest] < incoming`): `armies[dest] = incoming - prior_dest_armies; ownership[dest] = mover`.

### 8.2 `execute_attack(state, move_idx)`

Wraps `attack` + general-capture detection (5.11-2 §3.3, bundle line 68088–68093):

```python
def execute_attack(state, i, moves):
    end = moves.dest[i]
    old_owner = state.ownership[end]
    attack(state, i)
    new_owner = state.ownership[end]
    if (old_owner != new_owner
        and old_owner >= 0
        and state.generals[old_owner] == end):
        execute_player_capture(state, captured=old_owner, captor=new_owner)
```

`old_owner >= 0` excludes neutral captures. `old_owner` may have `alive == False` (the "captured during surrender countdown" path) — that case is handled correctly inside `execute_player_capture` via the `if alive[captured]` guard around `kill_player`.

### 8.3 `execute_player_capture(state, captured, captor)`

Bundle line 68233–68282. State changes in order:

1. **Capture tiles** (general tile naturally excluded — combat resolved first, so `ownership[general_tile] == captor` already):
   ```python
   mask = state.ownership == captured
   state.ownership[mask] = captor
   state.armies[mask] = (state.armies[mask] + 1) // 2     # halving = (a+1)//2
   ```
2. **Mark kill** (idempotent): `state.has_kill[captor] = True`
3. **Death record, if not already dead:**
   ```python
   if state.alive[captured]:
       kill_player(state, captured)
   ```
4. **General becomes city:**
   ```python
   state.cities.append(general_tile)
   state.cities_mask[general_tile] = True
   ```
5. **Null captured general slot:** `state.generals[captured] = -1`
6. **Append capture event:** `state.capture_events.append(CaptureEvent(state.timestep, captor, captured))`

**Gotcha:** the general tile's army count is whatever combat left after `attack()` — NOT halved. The `(a + 1) // 2` halving applies only to the captured player's *other* tiles.

### 8.4 `try_neutralize_player(state, p)`

Bundle line 68295–68303 (replay-class). State changes:

1. **Capture general tile reference** (must read before flipping):
   ```python
   general_tile = state.generals[p]
   ```
2. **Replace all of p's tiles with neutral** (no halving — armies preserved):
   ```python
   mask = state.ownership == p
   state.ownership[mask] = -1
   ```
3. **General becomes neutral city** (preserves accumulated army):
   ```python
   state.cities.append(general_tile)
   state.cities_mask[general_tile] = True
   ```
4. **Null general slot:** `state.generals[p] = -1`

**Gotcha:** step 4 is non-negotiable. Without it, the per-turn general-tick (`for gen in generals: if gen >= 0: armies[gen] += 1`) keeps incrementing a now-neutral tile every 2 timesteps. The `gen >= 0` check in the general-tick is doing the work; nulling here keeps it correct.

### 8.5 `kill_player(state, p)`

The 1st AFK event (surrender countdown begins). State changes:

```python
state.alive[p] = False
state.alive_count -= 1
state.death_events.append(DeathEvent(state.timestep, p))
state.input_buffer[p].clear()             # bundle's killPlayer clears their move queue
# generals[p] stays — tile still owned, still produces during countdown (5.11-2 §3.4)
```

The grid is **untouched**. The player is flagged dead but their tiles, cities, and general continue to exist on the board, producing under the `ownership[t] >= 0` predicate of the production loops. They become capturable by other players during the countdown window (the "captured during countdown" path in `execute_player_capture` above).

### 8.6 `kill_all_but_strongest(state)`

All-AFK / turn-cap synthetic game-end (5.11-2 §3.5, bundle line 67957–67979). Computes per-player totals lazily and kills all but the strongest:

```python
def kill_all_but_strongest(state):
    living = [p for p in range(state.num_players) if state.alive[p]]
    owned_mask = state.ownership >= 0
    armies_per_p = np.bincount(
        state.ownership[owned_mask], weights=state.armies[owned_mask],
        minlength=state.num_players,
    )
    tiles_per_p = np.bincount(
        state.ownership[owned_mask], minlength=state.num_players,
    )
    # Sort ascending by (armies, tiles, index); strongest is last
    living.sort(key=lambda p: (armies_per_p[p], tiles_per_p[p], p))
    for p in living[:-1]:
        kill_player(state, p)
```

After this, `alive_count == 1` and the next step's step-0 check exits the loop. Bundle behavior is "kill all but the leader" — neutralization doesn't fire from this path (no synthetic AFK events emitted).

## 9. Action-stream recording

Per design doc Q4 refinement: action stream records the player's **decision at this tick**, not the move's execution outcome. Recording fires at step 4 of the orchestration — post-pop, pre-sort.

```python
def record_action(state, i, moves):
    p = PlayerIndex(moves.index[i])
    ps = state.perspective_indices.get(p)
    if ps is None:
        return    # non-curated player; drives simulation but isn't emitted
    t = state.timestep
    state.actions_source[ps, t] = moves.source[i]
    state.actions_dest[ps, t]   = moves.dest[i]
    state.actions_is50[ps, t]   = moves.is50[i]
```

**What's recorded:**
- Move that survived the buffer-pop validity gate (`is_valid`). Includes moves that subsequently get inward-first-canceled or cycle-dropped during resolution.

**What's NOT recorded (stays as pass sentinel):**
- Empty buffer (player had no queued action this tick).
- Type-A invalid only (every move in the buffer was discarded at pop time — stale decisions conditional on now-impossible state).

**Length:** action streams are dimensioned `[K, T_max + 1]` where `T_max` is `max(moves.timestep) + safety_margin` from the wire (preallocated). Padded with pass sentinels through the full game length. Dataloader uses each perspective's `elim_timestep` metadata to skip frames after the player was eliminated.

## 10. Output intermediate format

Per design doc §3, with this session's refinements. Per-game serialized output:

### Game-static (written once)

- `id, version, map_width, map_height`
- `mountains: list[TileIndex]`
- `initial_cities, initial_city_armies` (parallel lists)
- `initial_neutrals, initial_neutral_armies` (parallel lists)
- `initial_generals: list[TileIndex]` (per slot, pre-capture)
- `listings_ranking` (from `replay_players` DB table — feeds ranking-match validator)
- `modifiers` (preserved but always empty for filtered FFA)

### Per-timestep raw state (length `T_final + 1`)

- `ownership: int8[T+1, H*W]`
- `armies: int16[T+1, H*W]`
- `cities_packed: uint8[T+1, ceil((H*W)/8)]` — bitpacked via `np.packbits(cities_mask)` at snapshot time

### Game-level metadata (written once)

- `death_events: list[DeathEvent]`
- `capture_events: list[CaptureEvent]`

### Per-perspective records (K records, one per curated player who was in this game)

- `perspective_metadata: list[PerspectiveMetadata]` (per §5.4)
- `actions_source: int16[K, T+1]`, `actions_dest: int16[K, T+1]`, `actions_is50: uint8[K, T+1]`

### On-disk format

Gzipped `.npz` per game (working default per design doc §14). Pinning the exact key schema is deferred until we have a sample game's output to size against.

## 11. Implementation sequencing (lightweight)

Maps to the phase plan in `replay-parser-status.md`. Each phase is one or two contained chunks of work:

- **Phase 3 (wire → typed records).** `types.py` + `decode.py` + the boundary rename. First concrete step; mechanical; locks shared vocabulary before any simulator code lands.
- **Phase 4 (simulator).** `state.py` (State + `build_initial_state`) → `combat.py` (combat + capture primitives) → `moves.py` (sort, dep loop, validity, recording) → `step.py` (orchestration) → `parser.py` (top level). Order matters: each module depends on the previous.
- **Phase 5 (ranking-match validator).** `validator.py`, built **alongside** Phase 4 — running on a small sample after each module's worth of work, full ~170k corpus run only once the simulator stabilizes.
- **Phase 6 (perspective output + intermediate writer).** `metadata.py` for rolling-rate enrichment (independent of simulator; can be drafted in parallel). `intermediate.py` for the `.npz` writer. Action-stream recording is already wired into `moves.py:record_action` from Phase 4.

Per-game parallelism (`multiprocessing.Pool` per design doc §6) is added once the single-game path is stable. Not part of any phase's critical path.
