## Design decisions

### Future Rust-version A/B uses branch labels + `sweep_match_rates`

Whatever shape future Rust-sim refactors take (perf opts, mechanics
fixes), the pattern for verifying "no regression" is: tag the baseline
with a `tmp-before-<name>` branch, commit the change on the working
branch, run `sweep_match_rates` on each, diff. Branch labels survive
context-switches better than stashes and are explicit in `git log`.
Ground-truth signal (`replay_players.position`), no oracle drift, no
in-process parity scaffold needed. Same shape as the int32 perf change
per [5.14-1](./5.14-1-sim-perf-and-rust-port-plan.md), refined.

### Principle: no in-tree consumer → drop the field

The cleanup pulled `actions_source/dest/is50`, `perspective_indices`,
`damage_sym_all/pre`, `damage_off_pre`, and their plumbing. These were
sim-written, nothing-read. Re-add when a real consumer needs them;
[[project-actions-perspective-deferred]] documents the expected shape
(`[K, T_max]`) and ~50-line re-derivation cost for the
actions/perspective cluster.

### Fixture-based regression tests — design landed, build queued

After the cleanup, the only integration signal is
`sweep_match_rates.py` (needs the collector DB). Fixtures fill the
"portable in-CI" gap. Designed but not yet built:

- ~5-8 hand-picked replays covering distinct sim corners (snowball
  win, multi-capture, AFK-heavy, neutralize edge, max-timestep,
  multi-way kill).
- Per replay: `wire.bin` + `expected.npz` with sampled snapshots at
  timesteps `[0, T/4, T/2, 3T/4, T-1]` + end-state vectors + event
  lists + damage matrices.
- A `tests/regen_fixtures.py` script for deliberate regeneration when
  sim behavior intentionally changes. **Discipline rule: understand
  why fixtures fail before regenerating** — fixtures failing
  unexpectedly is a feature, not a chore.
- One pytest file (probably replacing the now-deleted `test_state.py`)
  parametrized over fixture IDs.
- Estimated ~150-250 KB total, ~100 lines of test + regen code.

Ordering: cleanup first (done), fixtures next session against the
post-cleanup Rust surface.

## Bonus Note: Surprising finding - A latent Python sim bug

The parity sweep at n=8,700 surfaced a real bug that n=1,000 had
missed. Mechanism:

- `process_pending_afks` in the (now-deleted) Python sim called
  `try_neutralize_player(p)` on AFK events for players whose general
  had already been captured (so `state.generals[p] == -1`).
- The function does `general_tile = state.generals[p]` then
  `state.cities.append(general_tile)` and
  `state.cities_mask[general_tile] = True`. With `general_tile == -1`,
  Python's negative-indexing silently corrupts the *last tile*:
  appends `-1` to the cities list (which then accumulates phantom
  production on the last tile via `apply_production`) and flips
  `cities_mask[-1]`.
- The Rust port reproduced the call logic faithfully and panicked on
  `usize::MAX = -1 as usize`.

The trigger rate is ~1-in-3000 (hit at replay 3,400/8,700; not in our
n=1,000 sample). Fix in commit `eb239be`: early-return guard in both
sims when `generals[p] == -1`. Both implementations now no-op the
nonsensical case.

The lesson worth carrying forward: **"999/1000 parity" carries a
wider confidence interval than the point estimate suggests.** Wilson
95% on that was [99.4%, 99.98%] — consistent with up to ~0.5% true
divergence. Bumping to n=8.7k tightens the bound by roughly an order
of magnitude.
