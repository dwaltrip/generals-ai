# Session Notes — Sim perf pass + plan for a Rust port

## What we did

Profiled the simulator, found the obvious win, took it. Then talked through what comes next.

### Profile

Wrote `replay-parser/tmp/profile_sim.py`: pulls N vanilla-FFA blobs from the collector DB, runs `parse_replay` on each, reports `replays/sec` + `µs/timestep` + cProfile by cumulative and tottime. Also has a `--no-snapshots` flag that monkey-patches `SnapshotBuffer.append` to a no-op, for sizing the snapshot-copy cost in isolation.

Baseline on a 100-replay sample (seed=42, all parsed):
- 15.9 replays/sec, 63 ms/replay, 96 µs/timestep
- **`set_army` was ~58% of total runtime** — the per-write `np.any(value > _ARMY_MAX)` overflow check, ~946k calls per 100 replays, dominated.
- Snapshots were only ~2.4% (confirmed via `--no-snapshots`).

### The fix — int32 sim, narrow at snapshot time

Commit `e45cc12`. `state.armies` allocated as int32 instead of int16; `set_army`/`increase_army` deleted; the int16 narrowing happens inside `SnapshotBuffer.append`, which doubles as the overflow gate (`armies.max() > _ARMY_MAX → ArmyOverflowError`). Skip-on-overflow behavior unchanged. All raw writes inlined back into `combat.py` / `step.py` (`state.armies[idx] = value`, `state.armies[idx] += 1`, etc.).

**2.28× speedup: 36.4 replays/sec, 27 ms/replay, 42 µs/timestep.**

A/B'd the change against pre-change code on a 1000-replay sample (seed=1) via `git stash` — *exact same* match counts and miss IDs. Confirms the change is arithmetically equivalent (int32 holding values in [0, 32767] performs identically to int16 for all our ops; we don't depend on int16 wrap anywhere).

Validation script: `replay-parser/tmp/quick_validate.py`.

### Profile after

No single dominant cost; runtime spread across `is_valid` / `attack` / `apply_production` / `dependency_loop` / `priority_sort` / `select_candidates` / `buffer_pending_moves` / `SnapshotBuffer.append`, none above ~10% of the budget. Classic "CPython interpreter overhead is binding."

### Process note worth keeping

The **A/B-via-git-stash pattern** for verifying a perf change didn't regress behavior is great. Stash → run validator → compare → pop. Identical miss IDs across the boundary are about as ironclad as you get without writing a parity test. Worth reaching for whenever a perf change *should* be behavior-neutral.

## Where the Python ceiling actually sits

Walked through the full options ladder (multiprocessing, Tier B micro-opts, PyPy, Cython, Numba, mypyc, Rust, JAX-style cross-replay vectorization). The honest framing for *corpus parsing*:

- 36 rps × 8 cores via multiprocessing → 170k corpus in ~10 min. Plenty for the parser's actual job (one-shot per corpus version).
- Tier B Python polish (drop `int()` casts, vectorize `apply_production`, fix `dependency_loop` O(n²)) probably gets ~50-65 rps single-thread. ~1.5×.
- Compiled inner loop (Numba/Cython/Rust) could realistically do 10-30× single-thread, ~50-100× for Rust specifically.

For the parser alone, the marginal value above multiprocessing is small. But two reasons it might still be worth doing:

1. **Tighter validator-iteration loop.** Full-corpus ranking-match sweep is in "up next"; ~10 min vs. ~1 min meaningfully changes how much you'll re-run it.
2. **Possible reuse for Phase 2 self-play.** Currently deferred to "fork strakam's JAX." A fast Rust sim already corpus-validated against the BC training data is a credible alternative — avoids the JAX learning curve and the fork-divergence maintenance.

## Loose suggestion for Option A — port `step()` to Rust

The `rust-py-demo/` reference (commit `32f6304`) covers most of the setup tax — maturin + uv + PyO3 toolchain, the `cache-keys` block that fixes the uv↔maturin clobber, mixed-layout pattern, `.pyi` stubs + `py.typed`, the numpy-crate / `allow_threads` notes for "when you do this for real." That's the upfront cost mostly paid.

A reasonable starting shape (not prescriptive — flag your own preferences):

- **Port `step()` and what it transitively calls inside the timestep.** Decode, `build_initial_state`, `parse_replay` orchestration, and the validator stay Python.
- **Boundary surface:** Rust function takes the State (or its component arrays) + the per-replay `Moves`/`Afks` arrays, advances one timestep, returns `bool` (game-not-over). FFI overhead is ~100-200 ns per call → ~10-15 ms across a 65k-timestep corpus, negligible vs. current.
- **Parity oracle is free.** Keep both sims live during the port; A/B via the existing ranking-match validator on a bucketed sample. Exactly the pattern that worked for the int32 change. The Python sim eventually becomes either a `tests/reference_sim.py` or gets deleted once the Rust one is trusted.

Open design choices worth deciding collaboratively at the start:

1. **Where does the crate live?** Own subproject (`sim-core/`) vs. nested (`replay-parser/rust/`). The README captures the tradeoff — own subproject if multiple Python consumers, nested if just the parser.
2. **State as `#[pyclass]` or pass-the-arrays-in?** A `#[pyclass]` is cleaner long-term (single owner, no per-call rebinding) but more PyO3 surface area. Passing the existing Python dataclass's arrays in per call is lower-friction but means the `cities` / `generals` / `alive` / `input_buffer` Python objects re-bind across the FFI on every step.
3. **Increment order.** Two reasonable paths: port whole-`step()`-at-once and parity-test against the corpus, or port piece-by-piece (combat first, then moves, then production) with the Python sim still wired. The first is faster to a result; the second has tighter feedback loops if something diverges.

The natural starting move is probably the smallest possible end-to-end: get a Rust `step()` stub callable from `parser.parse_replay`, even if it just calls back into the Python sim internally — proves the boundary works in this codebase before any real porting starts. From there, port logic into Rust function-by-function with the validator running between steps.
