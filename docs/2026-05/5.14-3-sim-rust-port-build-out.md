# Sim Rust Port — B1 + B2 Build-Out

Implementation notes from porting the per-timestep sim to Rust. The
[5.14-2 design doc](./5.14-2-sim-rust-port-design.md) covers the
boundary + State ownership decisions; this captures what's hard to
re-derive from the source alone.

## Results

### Parity (`replay-parser/tmp/parity_b2.py`, n=1000, seed=7)

999/1000 full parity on snapshots + events + damage matrices. The 1
outlier is `fL08x_E3F` — a `TypeError: 'NoneType' object is not iterable`
inside the existing Python decode path, surfaced incidentally. Not a
Rust port issue.

### Performance (`replay-parser/tmp/perf_rust_vs_python.py`, n=100)

```
Python (decode+sim):    87.8 rps   11.39 ms/replay
Rust   (decode+sim):   739.1 rps    1.35 ms/replay   (8.4× whole pipeline)
  of which decode:    1077.6 rps    0.93 ms/replay
  of which sim:       2352.9 rps    0.43 ms/replay   (~25× sim only)
```

The 36 rps in [5.14-1](./5.14-1-sim-perf-and-rust-port-plan.md) was
cProfile-loaded (`profile_sim.py` runs cProfile). Without profiling, the
Python baseline is 87.8 rps. Don't let the 8.4× whole-pipeline number
read as bigger than it is.

Decode is now ~2/3 of the Rust pipeline time. If perf ever needs to go
further, `decode_wire` is the next leverage point. Today, 739 rps
single-thread = ~4 minutes for the ~180k corpus, ~30 seconds with
8-core multiprocessing. Plenty for the parser's job.

## Why standalone-then-swap

The piece-by-piece alternative (port one function, wire it into Python
`step()`, repeat) requires both languages to mutate the same State
during the transition. The two ways to do that — numpy view aliasing
into Rust-owned buffers, or `#[pyo3(get/set)]` on every hot field —
are both real scaffolding cost that gets ripped out at the end.

Standalone-then-swap pays zero coordination cost between languages.
Python sim stays as the reference oracle in production until the swap;
Rust sim is built and validated out-of-band; one commit flips
production.

## surrender_bonus is not a sim concern

The surrender-bonus `has_kill` rule (see
[5.13-4](./5.13-4-sim-ranking-mismatch-solved.md)) lives entirely in
`validator.py:apply_surrender_bonus`. The sim's contributions are
`damage_off_all` tracking and `NeutralizeEvent` emission — both ported
in B2. The existing Python validator works on Rust State outputs
unchanged; there is no sim-side rule to port.

## PyO3 0.28 gotchas

- **`Python::allow_threads` → `Python::detach`** (renamed in 0.27 for
  free-threading terminology). The `rust-py-demo` README predates this
  rename and will mislead.
- **`#[pyclass(get_all)]` is deprecated.** TODO in
  `sim-core/src/state.rs`. Replacement: per-field
  `#[pymethods] impl Foo { #[getter] fn bar(&self) -> T }`. Cosmetic,
  but should clear before declaring the port done.
- **PyO3 0.22 (rust-py-demo's pin) doesn't support Python 3.14.** 0.28
  is the first version tested against 3.14 final. The
  throwaway-crate verification pattern (build a minimal `add(a, b)`
  extension, confirm import works, then commit to the version) is worth
  reaching for whenever bumping a major version on tooling.

## uv workspace quirks

- **`uv sync --all-packages` is required** to install workspace
  members. Bare `uv sync` only installs the root project; the venv
  silently ends up missing the members. Symptom: a sync that prints
  "Uninstalled N packages" with no corresponding installs.
- The `[tool.uv] cache-keys` block in `sim-core/pyproject.toml` is
  load-bearing. Without it, `uv run` reinstalls from uv's wheel cache
  every invocation and clobbers the latest maturin build.
- uv workspace member ≡ shared root `.venv` (no per-member venv option
  in uv's model). Phantom-dep risk is real but unguarded — `deptry` is
  the mitigation if/when it bites.

## Validation strategy + the from_python bridge

Two layers used during the port:

- **Per-method parity** (`parity_b1.py`) — each Rust method called on
  a `State.from_python(py_state)` copy, post-state diff'd against the
  Python method's effect. Localizes divergences fast: the failure
  message identifies the specific call site.
- **End-to-end parity** (`parity_b2.py`) — full `simulate()` vs full
  `parse_replay`, every snapshot + event + damage matrix diff'd. The
  natural gate for declaring the port done.

`State.from_python(py_state)` is the classmethod that makes per-method
parity work without porting `build_initial_state` first. Worth keeping
post-swap for any future Rust sim changes that want the same A/B
muscle.
