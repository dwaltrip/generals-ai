## 1. Session context

The replay-parser design doc (`docs/replay-parser-design.md`) §7 calls for a "filter-counts report" to run before any corpus parse, surfacing the actual distribution of filter survival, modifier mechanics, and per-player skill across the cached `.gior` corpus. Several Tentative sections of that doc — §4 (filter frequencies), §5 (perspective-filter thresholds), §8 (sample-volume math), §11.3 (`generalTrades` filter-vs-implement decision) — depend on these numbers. This session built that report end-to-end and ran it to produce the first concrete measurements.

Terms used throughout, defined here once:

- **Vanilla FFA Filters** — our name for the §4 game-level corpus-filter rules: free-for-all only, 4–8 players, format version ≥ 15, no game-mode modifiers, no custom map, no chess-clock, no modifier tile arrays (the last as a defensive cross-check against the modifiers slot). The report's funnel applies them in a fixed order.
- **Curated list** — the union of four text files of generals.io usernames believed to be strong players. Used by the collector's `sweep-metadata` pass to drive which players' replay listings get fetched into the DB, and used in this report to define which players' game perspectives we count.
- **Perspective** — one player's vantage in one game; one prospective training trajectory. A game with N curated players in its ranking yields N perspectives.
- **`wire_data`** — SQLite BLOB column on the `replays` table. Gzip-compressed JSON of the wire-shape replay payload. Replaces the legacy `raw` column (which used lz-string compression); ~24× faster to decode.
- **Rolling window (perspective-level)** — for each perspective, the same player's 100 most-recent FFA games strictly before this game's start, used as a skill estimate at game-time. Computed via a SQLite window function.
- **Trajectory-quality filters** — a class of perspective-level filter that operates on the parsed trajectory (move sequence and per-turn state), as opposed to corpus-scope filters which operate on the wire-shape replay. Examples: long-game cuts, bad-start, died-quickly. Per the MVP sketch, these are config-toggleable at training time rather than baked into the corpus, and so don't live in this report.

## 2. At a glance

### Decisions made

- **"Vanilla FFA Filters"** as the report's name for the §4 funnel. Pairs cleanly with §5's perspective-scope filters and is pulled directly from the §4 heading text ("Corpus filtering — vanilla FFA only").
- **Curated list = unique union of four files**, listed below. 334 raw lines total; **205 unique valid names** after username-validity filtering and de-dup.
  - `replay-collector/data/leaderboards/leadeboard-s42-ffa-elite-gsheets.txt` — season-42 FFA elite, gsheets export.
  - `replay-collector/data/leaderboards/leadeboard-s42-ffawin-elite-gsheets.txt` — season-42 FFA-win-rate elite, gsheets export.
  - `replay-collector/data/top-players/2026-05-10-new-top-players-from-wr-start-bucket-analysis.txt` — shortlist derived from further filtering/processing of `winrate_star_buckets.py`'s 50-game-bucket win-rate analysis (raw output: `2026_05_10_wr_star_analysis.csv`). Filename implies they are all new, but in practice it had substantial overlap with previous "strong player" lists. However, it did uncover a decently sized set of genuinely new names.
  - `replay-collector/data/top-players/_archive/2026-04-30-leaderboard-ffa-top-100-combined.txt` — archived top-100 combined leaderboard snapshot from 2026-04-30.
- **Output location**: `replay-collector/tmp/`. Script lives at `replay-collector/scripts/filter_counts_report.py`. A dedicated `replay-parser/` subproject is not yet stood up — the script is read-only on the collector DB and naturally lives next to other analysis scripts.
- **Funnel scope: game-level §4 rules only.** Curated-list intersection and rolling win-rate are perspective-level stats and live in a separate section of the report, not as funnel stages. This matters because the curated list is effectively the sweep source set (see Key Context), so "≥1 curated player in the ranking" is a near-no-op as a game-level filter. The interesting per-game statistic is the collision rate (≥2 curated players), which informs §8's per-game perspective-yield estimate.
- **Rolling win-rate via a per-perspective sliding window**, computed by a SQLite window function over ~500k `replay_players` rows. Runtime is single-digit seconds.
- **Decode failures land as funnel stage 0** — the honest read of "what's actually parseable from this DB." Turned out to be zero for this corpus.
- **Rule 5 (modifier tile arrays empty) is one funnel bucket** with a per-array breakdown shown inline when non-zero. Surfaces "did rule 5 ever drop a game rule 4 missed?" without overweighting a defensive check.

### Key context provided that re-shaped the plan

- **The corpus is larger than the design doc reflects.** Continued collection between the previous session and this one brought the cached corpus to ~197k FFA replays with `wire_data` present (~178k at v15+), up from the 135k baseline that `compute-considerations.md` and `replay-parser-design.md` were written against. Numbers in §8 of the design doc and the `A.*` items in the grab-bag need to be re-derived against the new corpus size.
- **`wire_data` replaces the legacy `raw` blob** — landed as a separate workstream before this session. The new format is gzip+JSON instead of lz-string, decoding ~24× faster. The migration paid off concretely here: at lz-string speeds the wire-decode pass over ~197k rows would have been around ~100 minutes single-threaded and likely wanted multiprocessing; at gzip+JSON speeds it's ~150 seconds and single-threaded sequential is fine.
- **The 4-file curated union is effectively the sweep-metadata source set.** Almost every cached game has at least one curated-list player in its ranking by construction, so "≥1 curated player" cannot meaningfully be a game-level filter. The actionable per-game statistic is "how many curated players collide in this ranking" — which is the §8 perspective-yield question, not a corpus-membership question.
- **`winrate_star_buckets.py` already exists and feeds one of the four curated source files.** It runs a per-player 50-game-bucket win-rate analysis over the corpus; its raw CSV output (`2026_05_10_wr_star_analysis.csv`) was further filtered/processed into the `2026-05-10-new-top-players-from-wr-start-bucket-analysis.txt` shortlist. The "new-top-players" filename is a slight misnomer — many resulting names overlapped with previous lists; however, some are genuinely new. The report's secondary 50-game-bucket view (saved as a sidecar CSV) aligns with this script's conventions so the two are interchangeable downstream. Convention worth flagging: in the DB, `replay_players.position = 0` is the winner (0-indexed), not `position = 1` — easy to get wrong.
- **The 60% survival estimate in the design doc combines corpus-scope AND trajectory-quality filters.** It should not be read as a pure-corpus-scope number; the 86% measured here is the corpus-scope-only survival rate. Trajectory-quality filters (long-game, bad-start, died-quickly, etc.) operate per-perspective on parsed trajectory output and don't live in this report.
- **The rule-5 anomaly (observatories/lookouts in regular FFA) is expected.** Some weekends generals.io added these tiles to the regular FFA ladder. These are legitimate FFA games — the design doc's framing of rule 5 as a "defensive cross-check against rule 4" was slightly off; the two slots aren't redundant, they describe different dimensions (game-mode-level modifier flags vs. on-map tile presence).

### Empirical findings

All numbers from the run against `replay-collector/data/generals.sqlite` on 2026-05-11. Specific values are in the artifact `tmp/filter_counts_report-20260511-000440.md`.

- **Vanilla FFA Filters survival: 86.3%** (197,144 → 170,098). Most drops are at the version ≥ 15 cut (18.3k) and player_count out-of-[4,8] (4.4k). The `teams_null`, `map_null`, and `chess_clock_null` rules drop nothing — the collector's `ladder_id='ffa'` pre-filter already excludes those modes.
- **`generalTrades` (slot 27) at v16+: ~1.8% of games** (1.88% at v16, 1.79% at v18). This is the FFA position-swap mechanic (when two players capture each other's generals on the same timestep, both survive and swap all tiles instead of being eliminated). Filtering rather than implementing in the v1 simulator costs ~870 games out of ~170k. Resolves `replay-parser-design.md` §11.3 in favor of "filter."
- **Map aspect: 15.1% square / 42.3% wide / 42.5% tall.** Weighted theoretical board-symmetry multiplier: **4.61×** (D₂ on rectangles gives 4× from horizontal flip + vertical flip + 180° rotation; D₄ on squares gives 8×). Pins `A.A4` in the grab-bag exactly; previously a 2–4× working range.
- **Mean curated players per survivor game: 1.576** (vs. the design doc's prior 1.4 estimate). Per-game distribution: 0.27% have no curated player, 56.0% have one, 31.8% two, 9.5% three, 2.3% four-or-more. Yields **268,663 curated perspectives** in survivor games — reflows §8 sample-volume math upward.
- **Rolling 1st-rate is unimodal at 0.25–0.30.** The 25% threshold proposed in §5 sits at the modal cliff: a < 0.25 cut drops 40.4% of perspectives, < 0.20 would drop ~20%, < 0.30 would drop ~60%. The histogram makes the threshold-vs-corpus-size trade-off concrete and is the right artifact to consult when picking a final threshold.
- **Top-3 rate has a much cleaner separation.** Mode at 0.55–0.60, with the 37.5% random baseline for 8-player FFA in the lower tail (only 3.7% of perspectives fall below). Suggests top-3-rate may be a cleaner perspective-quality filter than 1st-rate.
- **Prior-games floor barely bites: 2.0%** of perspectives. The §5 selection rule excludes perspectives where the player has fewer than 50 prior FFA games at game-time, on the grounds that a rolling rate over a small sample is too noisy to be a reliable skill estimate. Curated players mostly have deep histories, so the floor catches very little.
- **Decode failures: 0.** Stage 0 of the funnel is a no-op for this corpus.
- **The per-player table surfaced inclusion noise.** `kickapp` has 3,316 perspectives at mean 1st-rate **0.061** — implausibly low for a curated player; a candidate to investigate for being a bot, alt account, or accidental inclusion. `removethisname`, `milosnikolic`, `aeffeawaefc` all sit at mean ~0.20 despite large game counts. At the other end, `Eight Equals D` leads at 0.405 and `onyon` is by far the most active (8,927 perspectives at mean 0.350).

## 3. Implementation notes

The script is single-file at `replay-collector/scripts/filter_counts_report.py`. Runtime against the current ~197k-row corpus is ~3 minutes, almost entirely the wire-decode pass; the SQL window-function aggregation for rolling rates is single-digit seconds.

Structure:

1. **Corpus-at-a-glance** — pure-SQL counts on `replays` (no `wire_data` decoding needed).
2. **Funnel pass** — sequential scan over FFA + `wire_data` rows; decode each, apply rules in order, accumulate drop counts plus per-rule diagnostics and a survivor list with metadata captured for section 2.
3. **Section 2 aggregations** — in-memory over the survivor list (map dims, length stats, `generalTrades` counts), plus one SQL pass for the D check (games without any curated player in the ranking) using a temp table of survivor IDs.
4. **Section 3 stats** — SQL window function for rolling-window stats per perspective; per-player loop for the 50-game-bucket secondary view.
5. **Renderers** — Markdown report with inline histograms drawn from `█` characters; raw-counts JSON; per-player buckets CSV.

The script is read-only on the DB; output filenames are timestamped so re-running is safe.
