# Initial Project Planning Overview

**Date:** 2026-05-02
**Purpose:** Ground the upcoming documentation pass. Summarizes the planning discussion to date, then flags what's worth preserving in-context vs. what can be looked up separately.

---

## 1. Project summary

**Goal:** Build a strong AI agent for 8-player FFA on generals.io. No published strong FFA bot exists; user is a high-ranked FFA player and well-positioned to evaluate.

**Approach (high-level):** Behavioral cloning on expert FFA replays, then self-play PPO fine-tuning. Modeled loosely on the Straka & Schmid 2025 1v1 recipe.

**Key strategic decisions made:**

- **Skip AlphaZero-style methods.** They depend on perfect-information + alternating turns + deterministic transitions to make MCTS principled. Generals has none of those. BC + PPO is search-free, model-free, handles partial observability by default. Schmid (co-author of Player of Games and Student of Games) reached for PPO when *he* built a generals env — that's a strong signal.
- **Skip 1v1 BC pretraining.** FFA replays contain the same fundamentals; 1v1 BC introduces negative transfer (1v1 threat-evaluation is wrong for FFA). Borrow the strakam 1v1 agent only as an *endgame sparring opponent* during eventual self-play.
- **Don't worry much about coordination.** User input: explicit alliances are rare in FFA, mostly emerge only in last-3 situations. Leaderboard signal + threat-aware aggression is enough. This simplifies the population/league design substantially.
- **MVP defers the full engine refactor.** BC is supervised on parser output; doesn't need an FFA-aware engine. Live-ladder eval needs `generalsio_state.py` made FFA-aware (~days), not the full `game.py` rewrite (~weeks). Engine refactor moves to Phase 3 if/when self-play is needed.

**Revised MVP path:**

1. Replay parser → FFA observation tensors (user's side project, in progress; ~10k replays collected, scaling up)
2. BC training loop on parser output
3. FFA-aware `generalsio_state.py` for live deployment
4. Deploy to live ladder, measure
5. Decision point: BC strong enough? → ship. BC clearly weak? → start engine refactor + RL.

**Probability estimates on BC-alone outcome:**
- 30% — "decent FFA bot, possibly the strongest published, plays competently but not at top-human level"
- 40% — "weak-mid bot, validates pipeline, RL fine-tuning needed"
- 20% — "BC plateaus at something disappointing, debugging adventure"
- 10% — "genuinely strong, top-quartile human play, RL only marginal"

**Compute budget for MVP:** sub-day BC training on a single GPU (A100/H100/4090 all viable). Dollars-to-low-hundreds of dollars in cloud cost. The constraints are wall-clock and debugging time, not compute.

## 2. Repo state (separate doc: `5.02-1-strakam-repo-survey.md`)

Strakam env is hardcoded 1v1 — `(2, H, W)` ownership, enumerated 2-player branches throughout `game.py`, single opponent channel in observation. PettingZoo interface is a wrapper around an engine that doesn't support >2 players. Published `_experimental/ppo/train2.py` is a 313-line bare-bones PPO with hardcoded random opponents, no self-play, no checkpointing — *not* the recipe that produced the paper agent. Network drops 5 of 14 obs channels and uses a 9-action policy head.

`remote/generalsio_state.py` is the highest-leverage finding: implements diff decoding for the live websocket protocol, likely matches `.gioreplay` file format. Useful as parser scaffold and on the deployment path. Hardcoded 1v1 on line 10.

## 4. What to preserve in subsequent docs

### Highlight (specific to this project, hard to re-derive)

**Algorithmic landscape — connections to this project, not how each works:**
- BC + PPO self-play (Straka & Schmid 2025) — the reference recipe; 1v1 only, paper-agent training code not fully open
- KL regularization back to BC — handoff stability technique used by AlphaStar; relevant when we eventually run PPO
- Historical-pool / fictitious self-play — opponent diversity mechanism; sufficient for our case (no league play needed without coordination)
- Player of Games / Student of Games — Schmid's principled approach for imperfect-info; deliberately *not* using because too expensive to implement
- Stochastic MuZero — wrong tool (handles stochastic transitions, not partial observability)
- Cicero / no-press Diplomacy — closest published n-player general-sum analogs; reference if coordination becomes a problem
- DAgger-style return to BC — stabilization technique, optional Phase 2 polish

**Architecture decisions and reasoning:**
- One network with shared trunk, policy head, value head, optional auxiliary heads (placement prediction, attack-target prediction)
- BC trains all heads (policy via cross-entropy, value via outcome regression)
- PPO continues training all heads via standard actor-critic
- Auxiliary heads shape the trunk's representation, don't change inference

**FFA-specific design considerations identified:**
- Player-relative observations (sort opponents by threat scalar)
- Threat-conditioned BC heads as a possible mixture-of-experts pattern
- Replay-buffer alliance prior — keep BC data in PPO batches as a strategic anchor
- Endgame upsampling in BC (1800 turns, ~50 endgame positions per game; endgame skill matters disproportionately)
- Strakam 1v1 agent as endgame sparring partner during eventual self-play

**Filtering decisions for replay corpus:**
- Integrity cuts (AFK, disconnect) — required
- Early elimination cut at turn 50-75 — keep simple
- Bottom-half placement filter — quality cut
- Dedup
- Capture rich per-trajectory metadata at parse time, filter at training time (not preprocessing)

**Corrections to prior assumptions:**
- Strakam env *does not* support 8-player FFA out of the box (engine-level rewrite needed for full RL phase)
- The published PPO scaffold is *not* the paper agent's training code (we're rebuilding training infra)
- Engine refactor is *not* on the MVP critical path (BC works on parsed replays alone)

### Skip (general knowledge, look up separately)

- How PPO works mechanically (clipped objective, GAE, advantage estimation)
- How MCTS works in AlphaZero, why tree search bootstraps tabula rasa
- Why AlphaZero needs perfect information / alternating turns
- General BC theory, covariate shift, DAgger details
- AlphaStar / OpenAI Five architecture details
- Generic JAX/Equinox idioms (point at strakam codebase as reference)
- Comparison of all open-source AlphaZero implementations
- N-player general-sum game theory background

These are well-covered elsewhere and can be supplemented in dedicated learning sessions if needed.
