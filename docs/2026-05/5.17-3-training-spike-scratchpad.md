## 2. State of the world — what we have

### 2.1 Corpus (data)

- DB: `replay-collector/data/generals.sqlite`. Multiple sub-tables; the relevant ones for the parser pipeline are `replays`, `replay_players`, `players`.
- Source replays: ~180k v15+ FFA replays in DB ([[project-corpus-size-current]]).
- Parser output on disk: **~165k games × 2 sibling `.npz` files = ~331k files, 9.2 GB total** (verified just now via `du`/`find`).
- Storage envelope: design doc envelope of 50–65 GB was stale; actual ~10 GB. DEFLATE compresses the `[T, HW]` snapshot arrays ~30:1 because of frame-to-frame redundancy ([[project-parser-output-size]]).
- Sharding: `data/intermediate/<2-char-prefix>/<id>.{npz,meta.npz}`. ~64² = 4k shards.
- Skip-log CSV per run records every dropped candidate (`_skipped_<timestamp>.csv`).

### 2.2 Per-game sim file schema (verified empirically on one sample)

One real sample: `3KXOs-pPx.npz` for replay `Z2l6i_RZh`, 8 players, T=346, HW=957 (a 29×33 board, roughly), 75 KB sim file.


In the corpus run, real rolling stats are computed (not stubs). Skip log records reason for dropped candidates.

### 2.4 Code components shipped

- `sim-core/` (Rust + PyO3): `simulate(replay) → State`. Handles step body, AFK processing, move resolution (priority sort + inward-first dependency check), captures, neutralizes, snapshots. ARMY_MAX=32767 guard. Returns `[T, HW]` snapshot stacks + action arrays + event lists.
- `replay-parser/replay_parser/`:
  - `parser.py` — `parse_replay(bytes) -> (State, ReplayData)`.
  - `decode.py` — wire → typed `ReplayData` (static block, moves, afks).
  - `output.py` — `write_sim_output` + `write_metadata`. Per design §6.2.
  - `metadata.py` — `build_metadata`. Per design §6.3 / §4.
  - `driver.py` — corpus orchestrator. Multiprocessing pool, SQL pre-filter, wire filter, perspective resolution, skip CSV.
  - `rolling_stats.py` — single chronological FFA walk, 200-game deque per curated player. ~15s for full corpus.
  - `git_state.py` — runtime SHA capture + dirty-tree gate.
  - `validator.py` — ranking-match against listings (100% on weekly bucketed sample; ≥99.85% at n=11.6k).
- `replay-collector/` — upstream, produces the SQLite + wire blobs.
- Smoke + integration tests green.

### 2.5 What we don't have (relevant for spike)

- **No PyTorch in deps**. The workspace `pyproject.toml` only has `numpy>=2.0` + the local sub-packages + dev tools. No torch, no MPS, no dataloader code.
- **No dataloader**. Channel assembly from the per-game intermediate has not been written.
- **No model code**. No Pyramid Module / U-Net implementation. No heads.
- **No training loop**. No loss code, no metric code, no checkpoint code, no logging.
- **No augmentation**. Board-symmetry + slot-permutation are locked decisions but not implemented.
- **No fog-of-war computation**. The per-game intermediate stores ground-truth ownership/armies for *every* tile at every snapshot. The obs tensor needs to mask out tiles that aren't in the perspective player's vision range — that derived computation lives somewhere (dataloader or a "perspective walk" helper).
- **No train/val split policy code**. Design doc has a tentative spec; not implemented.
- **No eval harness** beyond ranking-match validator (which is a sim-correctness gate, not a model-quality gate).
- **No GPU rental setup**. Modal or otherwise — not provisioned.

## 3. The training spike — what "done" looks like

Trying to be concrete. Multiple plausible "spike done" thresholds:

### 3.1 Minimum viable spike (the bar)

- Pipeline runs end-to-end **without crashing**: parsed corpus → dataloader → model → loss → backward → optimizer step → checkpoint write.
- One full training run (any duration) on a real subset of the corpus, even if just a few thousand games.
- Train loss decreases.
- Val loss computed at least once on a held-out set; not necessarily better than chance.
- Some sanity output — e.g., the model can overfit a tiny batch of a few games to confirm gradient flow.

### 3.2 Slightly more ambitious bar

- Above + smaller-than-full BC run (e.g., 1–5 hours wall-clock on H100 or several days on M1) on the real corpus.
- Train loss visibly converging across multiple epochs.
- Val loss tracked across epochs, gap measurable.
- Top-k next-action accuracy computed on held-out — concrete number, even if mediocre.
- Action-distribution sanity (model doesn't collapse to always-pass or always-one-direction).
- At least one fixture-board move-review: feed a recorded position through the model, look at the policy output. Does it pick something sane?

### 3.3 Stretch (probably not for spike)

- Full BC run on cloud H100 to the original 30–60 H100-hr anchor.
- Width sweep (4 variants × 5–10% of full duration).
- Eval distribution metrics (calibration, entropy).

**Open question:** which bar is the spike target? My read of the user's framing is **§3.2** — "first training run" but with enough rigor to look at outputs. §3.1 is more of a smoke test; §3.3 is past the spike.

## 4. What needs to be built (the laundry list)

Roughly grouped. Order is by dependency, not priority.

### 4.1 Environment / deps

- Add PyTorch to deps. Workspace pyproject vs subpackage pyproject? Probably a new `training/` subpackage (parallel to `replay-parser/`, `sim-core/`) keeps it clean — doesn't pollute parser deps. Or just stuff training code in `replay-parser` for now since they share data. **Open call.**
- Decide CPU/MPS/CUDA. M1 dev → MPS is the obvious local. Cloud → CUDA. Same codebase via `device="auto"` style.
- Test-import torch on M1, run a one-batch forward pass to confirm MPS isn't going to fall back to CPU on something dumb (the 30-min benchmark from 5.05-3 §2 action items that was never done).
- Modal vs RunPod vs Lambda vs Vast for cloud H100. Modal has good auth + ergonomics; user uses M1, so anything that's pythonic works. **Defer.** First runs can be local-only.

### 4.2 Dataloader (the biggest unknown)

Major piece. Has several sub-decisions:

#### 4.2.1 Online fog computation

Per design §3 / §9, the intermediate stores *raw game state per timestep*; the dataloader walks the trajectory forward online, accumulating fog state for the perspective player as it goes.

What "fog" actually requires:
- Vision range = Moore neighborhood (8-neighbor + self) of any cell owned by the perspective player at this snapshot, PLUS any cell adjacent to a cell I currently own that's mine or a city of mine (the standard generals.io vision rule).
- Per-cell `last_seen_owner` (9-way one-hot) — needs a running per-cell record updated on visibility.
- Per-cell `last_seen_armies` log-scaled.
- Per-cell `turns_since_seen` log-scaled.
- Per-cell `historically_seen` mask.
- 7 × `opp_N_has_seen` masks — opp N's Moore neighborhood ever covered this cell. Requires walking *every player's* vision per timestep, not just our own. Expensive? Cheap? Not obvious at the napkin level.
- Persistent structure channels (`mountains`, `cities`, `generals`, `structures_in_fog`) — `mountains` is static, `cities` and `generals` need persistence-from-discovery logic.
- `fog_cells` — current visibility boundary, derived from above.

For the spike, **could we drop some of these channels?** Yes. Going parsimonious for v1 spike makes sense:
- Skip `opp_N_has_seen` (7 channels, expensive to compute, flagged as a trimming candidate in 5.05-1 §7.3 #1).
- Maybe skip the full 9-channel `last_seen_owner` one-hot for v1, use a coarser encoding. Or keep it — the spike doesn't need to optimize this.
- Capture-event channels (`opp_N_captured_by`) — easy, just read `capture_events`.
- Contact-history (`opp_N_contacted`) — easy, monotonic broadcast scalar.

#### 4.2.2 Distance-from-general channels

8 channels (self + 7 opponents). BFS shortest-path on the known-passable graph. Cheap per-frame (microseconds at 25×25). **Recompute every frame** per `observation-tensor-design.md` §2.1. Sentinels: -1 for unreachable, -1 for unrevealed opponents.

Implementation: standard BFS per general. Cache the previous frame's result and only redo if terrain or general-knowledge changed? Probably not for the spike — just always recompute.

#### 4.2.3 Action labels + masking

- The supervised target at `(snapshot[t], perspective p)` is `(actions_source[p, t], actions_dest[p, t], actions_is50[p, t])`.
- Encoded as per-cell `[pass + 4 directions × 2 splits]` = 9 channels (network architecture §4). Need to convert from `(source, dest, is50)` to `(cell_index, direction_index, split_or_not)`.
  - source is the cell index.
  - direction is one of N/S/E/W — derive from `dest - source` modulo width with mountain/edge sanity check.
  - is50 — split or not.
  - pass = (-1, -1, -1) → distinct "pass" target.
- **Pass-frame loss policy:** how does pass interact with the H×W×9 policy head? Two natural choices:
  - (a) `pass` is the 10th class (a "no move" choice). Network architecture doc has it as one of the 9 channels at no-particular-cell, but that's slightly awkward — "per-cell pass" doesn't really make sense. Cleaner if pass is a separate scalar logit and the H×W×9 is "if I'm not passing, which move."
  - (b) `pass` represented as no-move-mask: action loss only computed on non-pass frames; train an extra binary pass/not-pass head separately.
  - **Open question, needs a call before coding.**

#### 4.2.4 Action masking (legality)

- Per `network-architecture-design.md` §4: legal = owned cell + army > 1 + dest passable.
- Implement as a boolean mask of shape `[H, W, 9]` (or `[H*W*9]` flat).
- Apply at training: softmax over the masked logits (set illegal logits to `-inf` before softmax). At eval: same.
- Source of legality info: current snapshot's `ownership` + `armies` + `mountains` mask (passability) + the destination cell's terrain.

#### 4.2.5 Augmentation

Per `5.05-3-session-notes.md` §3.3 (locked):
- **Board symmetry**: D₂ on rectangular maps (4 transforms: id, h-flip, v-flip, 180° rot), D₄ on square (~8 transforms). Need to permute `ownership/armies/mountains/cities/initial_*`, `actions_source/dest` (cells re-indexed under the transform), action direction channels (N/S/E/W permutes too).
- **Slot permutation**: random permutation of the 7 opponent slots. Apply uniformly across all per-opponent channels (broadcast scalars + spatial masks + history) for the same example. Doesn't touch the perspective player's slot (always self).
- These compose. ~2–4× from symmetry + ~2–3× from slot permutation = 4–12× effective.
- For the spike: can defer slot permutation if it makes implementation harder; board symmetry is dataloader-side cheap.
- Direction channels under board flips: e.g. h-flip swaps E ↔ W; v-flip swaps N ↔ S; 180° rot swaps both. Need careful test for this.

#### 4.2.6 Sampling pattern

Per design §9:
- `IterableDataset` style. Each worker picks a random game file, walks its trajectory, emits per-frame (obs, action) tuples.
- Shuffle buffer to approximate i.i.d. minibatches. Floor 30× mean trajectory length; ~16k frames at our distribution.
- For the spike, **a small shuffle buffer is fine** (1–2k); we're not trying to match the asymptotic generalization quality of a real run.

#### 4.2.7 Train/val split

Per `replay-parser-design.md` §9 (tentative):
- If ≥150 strong players: hold out ~10–15% of players entirely (~15–25 players, skill-stratified).
- Else: game-id-level split.

Real number from the corpus: 165k games × ~K perspectives/game. Curated list size unknown without inspecting `curated-player-lists.txt` files; likely a few hundred. **Need to check, easy.**

For the spike: a simpler split works fine. E.g., 95/5% game-level random. Match the design later.

#### 4.2.8 Dataloader scale

A small spike doesn't need 8 workers + 30 GB RAM buffers. A single-worker dataloader with a few-hundred-frame buffer pulls plenty of data through.

### 4.3 Network code

Per `network-architecture-design.md` §3–§4:

#### 4.3.1 Pyramid Module / U-Net torso

- 2-contraction U-Net: 25² → 13² → 7² → 13² → 25². (Map sizes vary; need to handle 24-33 cell side lengths. Either pad to a fixed 25² or 32², or compute spatial dims dynamically. Padding is the easy spike call.)
- Outer/middle/inner widths: 256/256/320 at the 1× baseline. **For the spike, use a smaller width** (e.g., the 0.5× variant 128/128/160) to keep iteration fast and so M1 / mini-cloud isn't pointlessly slow. Cheap to scale up later.
- ResBlocks: C → C/2 → C, 1×1 residual when channels change. Per DeepNash §1.3.
- N=2 outer ResBlocks per side, M=2 middle, M=2 inner.
- Skip connections at each level.
- LayerNorm + ReLU (per default-inherited).

#### 4.3.2 Policy head

- Pyramid module (small) → H×W×9 logits.
- Action masking applied before softmax.
- **Pass head**: separate scalar logit, or 10th channel — see §4.2.3 open question.

#### 4.3.3 Value head

- Pyramid module (small) → global avg pool → 8-way softmax over placement (1..8).
- **Tentative decision** per network-arch §3.2. For the spike, this is fine. Fallback: binary win/loss + BCE if categorical is harder to learn than expected.

#### 4.3.4 Input projection

- ~99-channel input @ map size. First conv: `Conv2d(99, 256, kernel_size=3, padding=1)` (or to the chosen width).
- If perspective is a fixed slot (always slot 0 after canonicalization), input is per-frame. Easier if we slot-canonicalize the perspective player to slot 0 (network sees "self always = slot 0") — keeps the augmentation story simpler.

### 4.4 Loss / training loop

#### 4.4.1 Loss

- Policy: cross-entropy on logits (with masking pre-softmax).
- Value: cross-entropy on 8-way placement target (per-frame target = the perspective player's *final placement* in this game). All frames in a trajectory share the same value target.
- Total: `policy_ce + λ * value_ce`. λ tentative ~0.5 — small enough that value head doesn't dominate, large enough to give trunk-shaping signal. Open.

#### 4.4.2 Optimizer

- AdamW (per network-arch §3.3). `weight_decay=1e-4`, `lr ~ 3e-4` initial. Trunk dropout 0.05–0.10 — defer for spike (no dropout = simpler, dropout addable later).

#### 4.4.3 Batch shape

- Each sample is (obs_tensor[~99, H, W], action_target, value_target, action_mask). Batch of 32–256.
- Variable map sizes: pad to a max dim (32×32 covers 90% of FFA games). Mask padded cells in loss + obs.

#### 4.4.4 Schedule

- For the spike: just a fixed lr, no warmup, no decay. Train for K epochs or N hours, whichever comes first.

#### 4.4.5 Mixed precision

- BF16 / FP16: 1.5–2× free on H100. On M1 MPS: FP16 supported, BF16 maybe limited. Use FP32 for the spike unless throughput is a problem.

### 4.5 Eval / metrics

Per [5.05-3 §3.4](./5.05-3-session-notes.md):
- **Held-out next-action accuracy** (top-1, top-3). Cheap, automatic per-epoch.
- **Action distribution sanity** (fraction pass / per-direction / split-rate). Cheap, automatic per-epoch.
- **Move-review on saved positions** — pick a few interesting positions from held-out games, dump model's top-3, eyeball. Manual but high-leverage (user is top 25–50 FFA player).
- **Value head calibration** — predicted vs. actual placement on held-out. Bin and bucket.
- **Overfit-tiny-batch sanity** — pre-training check.

### 4.6 Infra / observability

- Logging: print to stdout + a CSV / JSONL per run? Or wandb / tensorboard? **For the spike: stdout + a log file is fine.** Wandb adds setup cost for a one-shot.
- Checkpoints: save every N steps + at end. PyTorch's `torch.save` is fine.
- Run metadata: capture the parser's `sim_core_version` + the model code's git SHA + hyperparams + dataset hash, store with the checkpoint. Mirrors the parser's git-state pattern.

### 4.7 What to skip (defer)

- **Live deployment / WebSocket harness** — not on spike critical path. Phase 1 success is output-move quality (5.05-3 §3.2).
- **Phase 2 PPO self-play** — explicit Phase 2 work, deferred.
- **Auxiliary heads** — open, defer.
- **Player/skill conditioning embedding** — open, defer.
- **Output-projection hardening / blunder-prevention** — defer.
- **Full HW-conditional sweep / model-size sweep** — too expensive for spike.
- **Dataset enrichment (more scraping, filter discovery)** — orthogonal.
- **`last_seen_owner` empirical check** — deferred work, not blocking.

12. **Length of trajectories.** Games are ~600 half-turns mean, ~340 p25. Training every frame vs. subsampling (every 2nd)? Subsampling halves compute.

## 6. Risks / things that could go wrong

- **Fog-of-war computation is more involved than I think.** Per-player vision walk, persistent memory channels, multiple sentinel encodings. Easy to mis-implement; easy to silently produce a wrong obs tensor.
- **Direction encoding bugs under symmetry augmentation.** N/S/E/W must permute under flips/rots consistently with the cells.
- **Action label translation bugs.** Wire `(source, dest, is50)` → `(cell, direction, split)` is easy to get right but easy to get wrong. Need a unit test that round-trips.
- **Map-size variability.** Different boards in the same batch. Padding-and-masking is the typical answer; padding can also corrupt distance-feature BFS if not careful.
- **Pyramid Module strided convs on odd-sized inputs.** 25 → 13 → 7 is the inherited path; if we pad to 32², it's 32 → 16 → 8 which is cleaner. Need to pick.
- **Memory pressure on M1.** 99 channels × 32² × FP32 × batch_64 = ~26 MB per batch input. Not large; activations bigger. Probably fine on a 64 GB M1; tight on smaller.
- **PyTorch MPS gotchas.** Some ops fall back to CPU. Possible silent throughput hit. Mitigation: a one-pass benchmark (long-deferred from 5.05-3 §2).
- **PyTorch version compat with Python 3.14.** The project pins `requires-python = ">=3.14"`. PyTorch's 3.14 support is recent; some wheels may not be available. **Easily checked** — install + import.
- **The 66 unexplained `skip:exception` cases from the corpus run** ([5.17-1](./5.17-1-parser-output-v1-shipped.md) §"Open / TBD"). Small absolute number, but they suggest some edge cases in `sim_core` not yet bucketed. Not blocking the spike but worth a glance before scaling up.
- **Schema not fully locked** ([5.17-1](./5.17-1-parser-output-v1-shipped.md) §"Schema-lock posture"). Re-runs are cheap (~6 min), but the spike code shouldn't bake hard assumptions about field shape until we lock.
- **Action target = "pass" interaction with the value head.** Value target is constant across a trajectory; pass frames have the same value target as move frames. No subtlety here unless we weight pass frames differently.

## 7. Performance napkin math

For the spike, we want: hours-not-days iteration loop.

### 7.1 M1 (local) numbers

- M1 Max: ~22 TFLOPS FP16, ~64 GB unified memory.
- Per `compute-considerations.md` §5: torso forward ≈ 5–6 GFLOPS at 256/320 widths, 25² spatial. Halved widths → ~1.5 GFLOPS.
- One forward pass at ~1.5 GFLOPS / 22 TFLOPS / 0.3 (real utilization) ≈ 220 µs. Per-batch of 64 → ~14 ms forward. Plus backward → ~50 ms.
- Per epoch over 1k games × ~500 frames/game = 500k samples → 64-batch ≈ 8k steps × 50 ms = 400s ≈ 7 min. Per epoch!
- For 5 epochs: ~35 minutes. Plenty for a smoke-spike run on M1.
- Real numbers will be 2–10× slower in practice (data loader overhead, MPS quirks). Still: an hour or two for a first run is plausible.

### 7.2 H100 numbers

- Per compute doc: ~30–60 H100-hours for a *full* BC run at inherited widths, 165k games × 530 samples/game × 5 epochs.
- Half-widths: ~7–15 hours.
- 1/10th-subset (~16k games): ~3–6 hours at inherited, ~0.7–1.5 hours at half-widths.
- Cloud H100 cost: $2–10/hr. A spike full run: dollars to tens of dollars.

### 7.3 Map padding cost

- Padding 24-cell maps to 32 = 32² / 24² = 1.78× cells. Compute scales linearly with cells, so ~1.78× wall-clock penalty on the smallest games. Not significant for the spike.

## 8. What's in the existing design docs but might want to revisit for the spike

(Things I'd consider before locking in for the v1 spike.)

- **Inner pyramid width.** Network arch §3.3 plans a sweep. For the spike, just pick one (smaller, e.g. 128/192). Sweep later.
- **Contraction depth = 2.** Tentative in network arch §3.1. Stick with it; not a spike-time call.
- **Value head shape.** Tentative categorical-over-placements. Stick or fall back to binary — easy switch.
- **Channel budget.** ~99 channels. For the spike, dropping the 7 `opp_N_has_seen` channels (per 5.05-1 §7.3 #1) is the easy parsimony move. Cuts the per-frame vision-computation cost meaningfully.
- **Dense history N.** Tentative 10–12. For the spike, smaller (N=5 or even N=0!) keeps the dataloader simpler. The history block is a 2N spatial-channel addition — cheap compute but real engineering cost.
- **Training-time noise floor / quality filter.** Already applied by the parser at parse time (rolling-rate-based). For the spike, no additional filter — use whatever the parser emitted.

## 9. The "if I had to make a call right now" version

(Rough sketch, fully revisable.)

- **Spike target:** §3.2 bar — a first training run that produces a model checkpoint, with a few epochs, train/val loss curves, top-k accuracy, action-distribution sanity, and a couple of move-review eyeball checks. Days-not-weeks of work.
- **Where the code lives:** new `training/` workspace subpackage (parallel to `replay-parser`, `sim-core`).
- **Model:** half-width 2-contraction Pyramid Module. 128/128/160 widths. Categorical placement value head.
- **Obs tensor:** ~92 channels. Drop the 7 `opp_N_has_seen` for v1 spike. Keep distance-from-general channels (the 5.06-1 locked decision).
- **Pass head:** separate scalar logit, sigmoid + BCE. Cleaner than mixing into H×W×9.
- **Action encoding:** action = (cell, direction-or-split). Pass = separate.
- **Map handling:** pad to 32×32.

## 12. Sanity checks I want to be able to run early

Cheap things that would catch big problems:
- **Action round-trip test.** Take a known `(source, dest, is50)` from a recorded game, encode to `(cell, direction, split)`, decode back, confirm round-trip. Catches direction-encoding bugs.
- **Vision/fog parity test.** Manually compute fog for one perspective on one frame of a small game; compare to dataloader output.
- **Overfit a 4-game batch.** Train on a single batch repeatedly; loss should reach near-zero. If not, gradients are broken.
- **Action distribution check on a random untrained model.** Distribution should be approximately uniform (close to the random baseline of 1/9 per direction); collapse to one direction signals a broken mask or init.
- **Value head check on random untrained model.** Output should be uniform 1/8 per placement after softmax.
- **Augmentation parity test.** Augment a sample under each of D₂'s 4 transforms; the model on the augmented input should produce equivalently-permuted predictions. (This is the deepest sanity check; catches direction-permutation bugs.)
- **Forward pass on M1 MPS for a single batch.** Confirms torch deps + MPS path + no CPU fallback gotchas.
