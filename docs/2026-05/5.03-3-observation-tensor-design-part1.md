# Observation Tensor Design — Part 1: Settled Channels

**Date:** 2026.05.03
**Status:** Working spec for v1 observation tensor. Covers all settled channel categories. Long-term memory channels are deferred to a follow-up session.
**Companion docs:** `5.03-2-model-input-brainstorm.md` (prior brainstorm — this doc supersedes its conclusions where they differ), `5.02-8-ffa-strategic-intuitions-interview.md` (strategic grounding), `5.02-5-strakam-paper-summary.md` and `5.02-7-deepnash-summary-and-implications.md` (architectural lineage).

## 1. Design principles

These emerged across the brainstorm session and this design session. They shaped multiple decisions and should anchor any future reconsideration.

- **Conservative v1.** Stay close to the DeepNash/strakam blueprint. Modify where FFA forces us to. Add features beyond the published baselines only where the gain is large and the cost is small.
- **Split when in doubt.** Give semantically distinct information separate channels. Channels are cheap; learned recombination is flexible. But don't split when the network can recover any useful combination in one conv operation from the unsplit version.
- **Time as channel index, not scalar.** For temporal sequences, indexing time in the channel dimension is dramatically more sample-efficient than feeding a timestep scalar and asking the network to learn time arithmetic.
- **Policy-independent inferences as input features.** When a correct derivation from public information can be hand-computed (city count from peacetime growth rate, conflict state from land deltas), encoding it as an input feature is better than asking the network to learn to compute it. This is the DeepNash pattern (public-information tensor, memory heuristic).
- **Broadcast scalars for global information.** Global facts (scoreboard values, game clock) are broadcast as constant-valued spatial channels. Convolutions are local; broadcasting makes global information locally available at every position from layer 1 onward. Follows DeepNash and strakam precedent.
- **Static tensor shape.** The input tensor's shape is fixed at architecture-definition time. Per-opponent slots exist always, even if some opponents are dead or uncontacted; dead/absent slots are zero-filled.

## 2. What's already decided (inherited, not up for re-litigation)

- **Architecture family:** U-Net torso (Pyramid Module from DeepNash), per-cell H×W×9 policy head, value head. Hand-crafted memory features, no recurrence for Phase 1.
- **Action space:** per-cell [pass, 4 directions × 2 splits] = 9 channels per cell. Same as strakam. Transfers from 1v1 to FFA unchanged.
- **Player-relative observations:** the network sees "my cells," "opponent slot 0's cells," etc. — not absolute player IDs.

## 3. Settled channel categories

### 3.1 Map structure (6 channels)

Direct inheritance from strakam. Map mechanics are identical between 1v1 and FFA — no changes needed.

| Channel | Type | What it tells the model |
|---|---|---|
| `mountains` | binary mask | Impassable terrain. Shapes pathfinding and effective board topology. |
| `cities` | binary mask | Visible army-producing structures. Economic targets and defensive anchors. |
| `generals` | binary mask | Visible generals. Capture targets for elimination. |
| `neutral_cells` | binary mask | Unowned passable tiles. Expansion opportunities. |
| `fog_cells` | binary mask | Cells the agent currently cannot see. Defines the information boundary. |
| `structures_in_fog` | binary mask | Known structure positions with unknown type (mountain or city — must explore to determine). Exploration targets. |

### 3.2 Ownership (8 channels)

Per-opponent ownership masks rather than a single aggregated "any enemy" mask.

| Channel | Type | What it tells the model |
|---|---|---|
| `owned_cells` | binary mask | Agent's own territory. Defines vision boundary and production base. |
| `opp_N_cells` (×7) | binary mask | Per-opponent visible territory. Who controls what, border positions, threat proximity. |

**Why per-opponent instead of aggregated:** the network needs opponent identity to do any threat-aware reasoning. A conv kernel seeing "enemy territory" without knowing *which* enemy can't assess whether the neighbor is the strongest player or the weakest. Identity is required for the strategic reasoning the interview describes.

**Why 7 slots:** forced by 8-player FFA. Even if we believe fewer opponents are strategically relevant at any moment, the set of relevant opponents shifts during the game. Fixed 7 slots with zero-fill for dead opponents is simpler and more general than a dynamic top-k scheme.

**Slot ordering: canonical-by-ID.** Opponents are assigned to slots based on their player ID (or join order), and this assignment is stable throughout the entire game. The main alternative — sorting by a threat-relevant scalar like army count — creates discontinuous slot swaps when opponents' relative positions change. These swaps would corrupt spatial ownership masks and history channels simultaneously, making the representation harder to learn from. Canonical-by-ID means the network must learn to assess threat from the features within each slot rather than from slot position, but it has all the features it needs to do so (army count, land count, city inference, spatial ownership, land delta). This is arguably the right inductive bias — don't bake in a fixed threat heuristic.

### 3.3 Army magnitude (1 channel)

| Channel | Type | What it tells the model |
|---|---|---|
| `armies` | int per cell | Army count per visible cell, owner-agnostic. Combat potential, movement capacity, defense strength at each position. |

**Why owner-agnostic rather than per-owner:** the "split when in doubt" principle applies when combining axes *hides information from convs*. Here, ownership × armies is trivially recoverable in the first conv layer — a kernel reading both `armies` and the relevant ownership mask computes "owned armies at this cell" via a single learned weight combination. Per-owner encoding would cost 8 channels for no information gain. Additionally, owner-agnostic queries are useful primitives ("is there a big army anywhere nearby?") that don't require knowing the owner first.

### 3.4 Self broadcast scalars (3 channels)

| Channel | Type | What it tells the model |
|---|---|---|
| `owned_army_count` | broadcast scalar | Agent's total army including behind-fog tiles. Overall military strength. |
| `owned_land_count` | broadcast scalar | Agent's total territory. Production base for land ticks. |
| `timestep` | broadcast scalar | Game clock. Phase awareness, land-tick timing, urgency calibration. |

Inherited from strakam. The brainstorm proposed two additional self-scalars: `num_opponents_alive` and `num_opponents_contacted`. `num_opponents_alive` was dropped because per-opponent broadcast scalars (§3.5) carry the same information more usefully — a dead opponent has zero army and land, which is an unambiguous death signal since living players always have at least their general with ≥1 army. `num_opponents_contacted` is deferred to the long-term memory discussion — it's a memory feature (requires tracking historical vision) and may or may not be useful in per-opponent flag form versus count form.

### 3.5 Per-opponent broadcast scalars (14 channels)

| Channel | Type | What it tells the model |
|---|---|---|
| `opp_N_army_count` (×7) | broadcast scalar | Each opponent's total army from scoreboard. Strength assessment, threat level. |
| `opp_N_land_count` (×7) | broadcast scalar | Each opponent's total territory from scoreboard. Economic position, scaling trajectory. |

**Source:** the in-game scoreboard, which is always visible to all players regardless of fog, updated every timestep.

**Why these matter beyond the spatial channels:** the per-opponent ownership masks (§3.2) only show *visible* tiles. For most opponents most of the time, the majority of their territory is behind fog. The scoreboard totals include everything — they're the primary information channel for opponents the agent can't directly see.

**Alive/dead signal:** no explicit `opp_N_alive` flag is needed. A living player always has at least 1 army (their general) and at least 1 land (their general tile). `army_count = 0` and `land_count = 0` unambiguously mean the player is eliminated. When a player is captured, their territory transfers to the captor immediately, so their scoreboard values drop to zero the same timestep. All other per-opponent channels (spatial masks, scoreboard-derived features, history) are also zero for dead opponents, reinforcing the signal consistently.

### 3.6 Scoreboard-derived features (14 channels)

Two hand-computed per-opponent features derived from scoreboard history over time. These encode strategically important inferences that strong human players make continuously during play (documented in the strategic-intuitions interview). They follow the DeepNash pattern of feeding policy-independent inferences as input features rather than asking the network to learn the derivation.

| Channel | Type | What it tells the model |
|---|---|---|
| `opp_N_city_inference` (×7) | broadcast scalar | Inferred city count (including general) from peacetime army growth rate. -1 when unclear. |
| `opp_N_land_delta` (×7) | broadcast scalar | Net land change over last k timesteps. Expansion vs. contraction signal. |

#### City count inference

Army production is per-turn (every 2 timesteps). During peacetime, a player's army grows by exactly 1 (general) + number of cities per turn. Observing an opponent's army growth over several peaceful turns reveals their city count exactly.

**Computation:** adjusted army delta per turn = `army[t] - army[t-1] - (land_count if land_tick_turn else 0)`. The land-tick bonus is subtracted because it's predictable from the opponent's known land count. The residual during peacetime equals city count (including general).

**Confidence rule (j-out-of-k matching):** look at the last k adjusted deltas. Scan backward from the most recent turn. If j consecutive (or near-consecutive) turns share the same adjusted delta value, report that value as the inferred city count. Otherwise report -1 ("unclear").

**Properties:**
- During peace: all deltas match, confident signal immediately.
- During minor skirmishing with pauses: peacetime turns still produce matching deltas; j < k tolerates noisy turns.
- During sustained heavy fighting: few or no matching deltas, reports -1. This doubles as an indirect conflict signal.
- After capturing a city: the matching value shifts up within k turns as the new production rate stabilizes.

**Parameters:** j and k are independent tunable training-time parameters (no re-parsing needed if stored as raw deltas). Reasonable starting point: j ≈ 3, k ≈ 8–10.

#### Land delta (expansion/contraction)

Land count changes are per-timestep (a player can gain or lose at most 1 tile per timestep under normal single-opponent combat, ignoring general captures which transfer all territory at once). The signal is bounded small integers, mostly in the range {-2, -1, 0, +1}.

**Computation:** `land[t] - land[t-k]` — net land change over the last k timesteps. Positive = expanding, negative = contracting, near zero = stable or mutual fighting.

**What the magnitude tells you:**
- Sustained +1/timestep: one-sided expansion (capturing neutral tiles or winning a fight).
- Sustained 0 with known contact: likely mutual fighting (captures and losses roughly balanced).
- Sustained -1/timestep: being attacked by one opponent, losing ground.
- Sustained -2/timestep: being attacked by two opponents simultaneously.

**Net vs. mean aggregation:** for a fixed k, net and mean differ by a constant factor of 1/k that the first conv layer absorbs into its weights — functionally equivalent. Net is simpler (one subtraction, no division). Mean is more stable across different k values. Lean: net for v1 simplicity, consider mean if k becomes tunable at training time.

**Parameter:** k is an independent tunable training-time parameter, separate from the city-inference k.

#### Features considered and excluded

- **Conflict detection (cross-player land-delta correlation):** too complex to encode compactly. The land-delta signal per opponent gives enough of a hint for the network to infer who's fighting — if opponent A is shrinking and opponent B is growing, the correlation is available from the two separate scalars.
- **Relative trajectory / second derivative:** speculative for v1. The raw land-delta already captures first-order trajectory.

### 3.7 Dense recent history (2N channels)

This section required the most iteration during design, and represents the largest departure from the published baselines. The reasoning is documented in detail below.

| Channel | Type | What it tells the model |
|---|---|---|
| `ownership_transition[t-k]` (×N) | dense scalar per cell | Who lost each tile at each recent timestep. Combined with current ownership, reconstructs full ownership transition chains. |
| `army_delta[t-k]` (×N) | scalar per cell | Production-subtracted army change per visible cell at each recent timestep. Movement direction, buildups, combat intensity. |

#### Design journey: from move history to observed changes

**Starting point: strakam/DeepNash move-history encoding.** Both published baselines encode recent moves as time-indexed channels — 7 moves for 1v1 generals, 40 moves for Stratego. DeepNash uses a dense signed-value encoding that packs source location, destination location, move type (regular vs. attack), and piece type into a single scalar per cell per timestep, achieving ~15× compression over a naive multi-channel encoding.

**Problem 1 — source deducibility.** In generals under fog of war, we often cannot determine where a move originated. If armies appear at a visible cell, they could have come from any adjacent cell outside our vision. Stratego sidesteps this entirely because it has full position visibility (fog only hides piece *types*, not piece *locations*). Encoding source cells we can't observe would either pollute training data with fabricated signals or create inconsistent representations where some moves have sources and others don't.

**Reframe: "observed changes" instead of "moves."** Rather than encoding actions (source → destination), we encode what we directly observe: per-cell changes in army count and ownership between consecutive timesteps. This sidesteps source-deducibility entirely — we record effects, not causes.

**Problem 2 — ownership transitions carry critical information that army deltas lose.** A raw army delta of -7 on a cell means entirely different things depending on whether ownership changed: armies moved away (no flip) vs. combat resolved and a different player now holds the tile (flip). The strategic implications are completely different, but a single delta channel can't distinguish them.

**Solution: separate ownership-transition and army-delta channels.** Two parallel per-timestep channels, each encoding one clean signal.

#### Ownership-transition encoding

**Key insight: encode only who *lost* the tile.** The current-state ownership masks already show who holds each tile *right now*. The information that's actually lost as time passes is who *used to* own it. Encoding only the loser gives the full transition for free: history says who was there, current state says who's there now.

This avoids the two-identity packing problem that would arise from encoding both parties to a transition in one scalar. It also naturally handles multi-step transition chains — if a tile goes Green → Blue → Orange within the history window, the channels show "Green lost at t-5" and "Blue lost at t-2" and current ownership shows Orange.

**Dense scalar encoding:**
- `0` — no ownership change at this cell this timestep
- `-1` — agent lost this tile (most urgent signal)
- `+(1 + k/8)` — opponent k lost this tile (k = 0..6, 7 discrete values at 0.125 spacing)
- `+0.5` — neutral lost this tile (city or neutral army was captured)

**Why this encoding is learnable (analysis via Stratego precedent):**

DeepNash demonstrated that networks can decode dense signed-value encodings with fractional categorical offsets. Our encoding is structurally simpler than DeepNash's:

- **Fewer categories.** 7 opponent identities at 1/8 spacing vs. 12 piece types at 1/12 spacing.
- **Same hierarchical structure.** Sign separates "bad for agent" (negative) from "informational" (positive). Magnitude separates self-involved (±1 range) from non-self (±0.5). Fractional offset identifies which opponent, only within the self-involved category.
- **Same sparsity.** Ownership changes are rare per-timestep events — most cells are 0. Non-zero values are isolated signals against a clean background.
- **The key property that made DeepNash's encoding work:** sparse events, categorical participants, hierarchical structure with large gaps between primary categories, and a small number of evenly-spaced discrete values within each category. All of these hold here.

**What this channel captures strategically:**
- Attacks on the agent (urgent: -1 values appearing on the agent's border)
- Nearby fights between opponents (informational: positive values on contested territory)
- Territorial shifts over the recent window (chains of transitions across timesteps)
- Combined with current-state ownership, full "who took what from whom" reconstruction

#### Army-delta encoding

**Computation:** `army[t-k][cell] - army[t-k-1][cell]` for each cell visible at both timesteps, with predictable production subtracted out.

**Production subtraction:** generals and cities produce +1 army on turn boundaries (every 2 timesteps). All owned tiles produce +1 on land ticks (every 25 turns = 50 timesteps). Both are deterministic and known for all visible tiles (we can see structure type for any visible tile, and we know land-tick timing from the game clock). Subtracting these contributions isolates the strategically interesting residual: armies arriving from or departing to adjacent cells.

**Encoding:** exact production-subtracted delta, possibly log-scaled to compress the range. Not coarsened or bucketed — a +50 army arrival is a qualitatively different event than a +3 residual, and the magnitude carries real tactical information (major army movement vs. minor fluctuation). The `armies` current-state channel shows *how much* is on each tile now; the delta shows *how much just changed*, which is a distinct and complementary signal.

**What this channel captures strategically:**
- Army movement direction near the agent's border (approach vs. withdrawal)
- Buildup detection (accumulating positive deltas on an opponent's tile near the border)
- Combat intensity (large deltas at contested positions)

#### Self-action history: deferred

The brainstorm proposed a separate self-action channel (source/destination markers for the agent's own moves). This was dropped for v1: the army-delta channel already captures the agent's movements as negative deltas at source cells and positive deltas at destinations. Dedicated self-action channels are noted as a fallback to add back if training reveals attribution confusion in scenarios where self-actions and opponent actions overlap on the same cell in the same timestep (rare in practice).

#### Window size (N)

Not yet settled. The tactical queries the window serves (movement tracking, active combat detection, short engagements) suggest N = 7–14 timesteps (3.5–7 seconds of game time). Strakam uses 7 for 1v1. FFA games are longer overall, but the tactical window for "what's happening right now" is similar in duration.

At N=7: 60 channels. At N=10: 66 channels.

For comparison: strakam's 1v1 trained network is estimated at 20–25 channels. DeepNash's Stratego network uses 82 channels. Our v1 is in the same range as DeepNash — larger than strakam's 1v1 setup as expected given the information demands of 8-player FFA.

The long-term memory channels (next session) will add to this total. The brainstorm estimated the full tensor at 65–90 channels; we're currently at 60–66 before memory, so the final total will likely land in the 75–95 range depending on memory design decisions.

## 5. Open questions for the next session

### Long-term memory (primary topic)

The following information requires past observations to derive and is not yet represented in the tensor:

- **Fogged cell state.** What was last seen at cells currently in fog — who owned them, how many armies, how stale is that information? The brainstorm proposed `historically_seen`, `last_seen_armies`, `last_seen_owner` (9-channel one-hot), `turns_since_seen`, and `revealed_structures`. The 9-channel one-hot for `last_seen_owner` is the most expensive single encoding choice and needs fresh justification.
- **Structure type resolution.** Once the agent explores a fogged structure and discovers whether it's a mountain or city, that knowledge is permanent. This needs to persist through re-fogging. May be handled implicitly by the last-seen representation or may need a dedicated channel.
- **Contact history.** Has the agent ever had vision of each opponent? Distinguishes "haven't met opponent 3" from "met opponent 3, they're now behind fog." The brainstorm proposed per-opponent `opp_N_contacted` flags. Whether this is useful (vs. inferrable from other channels) is an open question.
- **Capture events.** When a player captures another player, a system message announces it globally (visible to all players for the rest of the game). E.g., "Player 1 captured Player 3." This is strategically important information — it tells every player who eliminated whom, which reshapes the threat landscape. How to encode these events (and whether to encode them alongside the memory channels or as a separate feature) needs discussion.

### Open details

- **Dense history window size (N).** Discussed 7–14 range, not committed.
- **Newly-visible-tile handling for army deltas.** Computing a delta requires visibility at two consecutive timesteps. Tiles that just entered vision have no baseline for a delta. How to handle this (record 0? separate mask?) is deferred.
