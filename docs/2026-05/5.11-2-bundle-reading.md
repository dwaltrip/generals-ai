## 2. Confirmed mechanics (§11 items)

### 2.1 Surrender / AFK countdown (§11.1)

**Rule (server side, `_goAFK` at line 73439–73489):**
```
neutralizeTurn = afkTurn + 1                                  if disconnect/leave path
neutralizeTurn = afkTurn + floor(2 * game_speed * 25000 / 1e3) if regular surrender
              = afkTurn + 50 timesteps at game_speed = 1
```
Constant: `TIMEOUT_CAPTURE_AFK = 25000` (ms) — line 68750.

**What lands in the replay's `afks` array:**

- **First event per affected player** = surrender/disconnect moment, written by `_goAFK` calling `replay.addAFK(e, this.turn)` at line 73449. Always fires (unless the surrender is at turn 0 — see edge case below).
- **Second event per affected player** = neutralization moment, written by live-class `tryNeutralizePlayer` calling `replay.addAFK(e, this.turn)` at line 73545. **Conditional** — only fires if the countdown completes before the game ends and before the surrendered player is captured.

**Empirical distribution (§7, 5000-game sample):** ~37% of (game, player) AFK records have both events; **~63% have only the first**. Two mechanisms produce 1-event records, both important to understand:

1. **Game ends before countdown expires.** Once `alivePlayers === 1`, `nextTurn` short-circuits via the `isOver()` check at line 67625, so `processReplayAfks` no longer fires. Common when a late-game surrender drops the game to a single remaining player.
2. **Surrendered player is captured before countdown expires.** When another player captures the surrendered player's general during the countdown window, `executePlayerCapture` (line 68233–68282) handles the takeover — `replaceAll(e, t, 0.5)` does the army halving, general → city, `generals[e] = -1`. **`executePlayerCapture` does not emit a 2nd AFK event**; only the `neutralizeTurn` countdown path does. So capture pre-empts neutralization without leaving a trace in the `afks` array.

**Replay-class behavior (`processReplayAfks` at line 67641–67652):** iterates `gameReplay.afks` in order. For each event, dispatches by current state:
- If player is *not* yet dead → `killPlayer(socket)` (marks dead, clears their move queue). Tiles remain owned and capturable.
- If player *is* already dead → `tryNeutralizePlayer(e)` (replaces all player-owned tiles with neutral, halves city/general armies via `replaceAll`, pushes the general tile into `cities`).

**Implications for our parser:**

- **Drive both events directly off the `afks` array.** No need to hardcode the 50-timestep constant; the replay tells us exactly when each event fires (when it fires at all).
- **The simulator must stop processing pending AFKs once `isOver()` returns truthy** — the bundle's `nextTurn` short-circuits on `isOver()`, so following the bundle's structure handles this correctly. Don't fire AFK events past game-end even if more remain in the array.
- **"Killed but not neutralized" is a first-class state at game-end** for ~63% of AFK'd players. Their tiles remain owned, still producing per `RECRUIT_RATE` and `FARM_RATE`, still capturable, through to game-end. The final-state ranking computation must include their tiles + armies under their player index. `lbSort` (§3.6) handles this naturally — `dead && !neutralized` and `dead && neutralized` both sort by death order via `this.deaths`.

**Disconnect vs surrender — not empirically observable.** The bundle distinguishes the two paths via `transferImmediate` (surrender → 50-step countdown via `handleSurrender` at line 73519 → `_goAFK(t, false)`; disconnect → 1-step countdown via `handleLeave` at line 73575 → `_ensurePlayerAfkOrDead(n, true)`). In principle a paired-event gap of 1 would identify a disconnect. Empirically (§7), zero gap=1 events appear in the sample — either disconnects don't happen in ladder FFA, or they always cut to 1-event records (the 1-step countdown gets pre-empted by capture or game-end). Drop the framing for v1; revisit if a use case for the distinction emerges.

**Edge case — turn-0 surrender:** `_goAFK` at line 73489 short-circuits to `queueDeath = true` if `this.turn === 0`. The live-class `update()` at line 72361–72364 then catches this when `this.turn === 1`, calling `_ensurePlayerAfkOrDead(J, true)` — same as a disconnect (1-step countdown). So in the replay, a turn-0 surrender appears as AFK events at turn 1 and turn 2.

**Edge case — null general at game start:** `populateFromReplay` at line 67579–67585: if `generals[t] == null`, the socket is marked dead and `alivePlayers--` before the game even starts. No AFK events for these players; they enter the simulator already eliminated.

### 2.2 Timestep-within-turn ordering (§11.2)

**Confirmed:** array order within `gameReplay.moves` is authoritative for timestep ordering within a turn.

**Mechanism:**
- `bufferReplayMoves` (line 67654–67668) is called once per `nextTurn`, BEFORE `update`. It drains all moves with `move.turn <= this.turn` from `gameReplay.moves` into per-player `inputBuffer[move.index]`, appending in array order.
- `update` (line 67836+) consumes *one* move per player per call via `getNextValidSetOfMovesPrioritized` → `getPlayerMovesPriorityV2` (line 67804–67824) which pulls `inputBuffer[player].splice(0, 1)`.
- Each player can have at most one move executed per timestep. Multiple moves at the same `turn` value spread across consecutive timesteps in array order.

### 2.3 `generalTrades` (§11.3 in design doc)

**Decision: filter.** ~1.8% of v16+ games. Bundle behavior recorded here for completeness — we don't implement it.

**Bundle mechanism:**
- Detected pre-execution in `update()` at line 67856–67867 via `getMutualGeneralSwapMoveIndex` (line 68122–68147). Looks for a pair of moves where each captures the other's general on the same timestep (both `wouldAttackCaptureGeneral` checks pass).
- If found, `executeMutualGeneralSwap` (line 68148–68210) swaps all tiles, armies (halved as in normal capture inheritance), generals, and cities between the two players. Both survive.
- Recorded in the replay as a `generalTrades` entry per slot 27.

### 2.4 Tie resolutions (§11.4 / mechanics §11.1–11.3)

All three reduce to a single mechanism — the V3 priority sort in `MoveResolver.determineMoveOrder` (line 67375–67421) — applied for every `version >= 15` game.

**V3 sort key (line 67386–67398):**
1. **Defensive moves first** (`tile_at(end).owner == mover` → friendly tile is destination).
2. **General-attacks LAST** (among same-defensive class).
3. **Larger source-tile army first** (descending by `armyAt(start)`).
4. **Input order tiebreak** (`basePriority` = order in the per-player `inputBuffer`).

**Dependency loop (line 67402–67419):** after sort, walks the sorted list. For each candidate move, checks if any later-priority move's end-tile is this move's start-tile (`hasDependency`). If yes (the other move is going to land on us first), defer; pick the next non-dependent. If we run out and have a cycle, take the highest-priority move (line 67415–67418, comment in the bundle elsewhere notes this as the "1-in-10,000" fallback).

**Resolves:**
- **Mechanics §11.1 (convergent moves with exact-tie armies):** sort handles it. Larger army resolves first; if armies are equal, input order tiebreaks. Combat resolution in `attack()` then handles the tie semantics per side (defender-advantage on equal armies — see §3.2 below).
- **Mechanics §11.2 (two attackers on one defender):** same sort. Both attackers land sequentially in priority order; each is independently resolved by `attack()` against whatever the tile state is at that moment.
- **Mechanics §11.3 (production timing on capture):** see §3.3 below — production fires after move resolution in the same `update()` call, so the *new owner* gets the production tick. Confirmed.

## 3. Net-new precision (was not previously in docs)

### 3.1 `update()` orchestration order (line 67836–67955)

Per `nextTurn` (line 67624–67640):
1. **`processReplayAfks`** — fire any AFK/neutralize events with `turn <= this.turn`.
2. **`bufferReplayMoves`** — drain moves with `turn <= this.turn` into per-player input buffers.
3. **`processReplayChat`** — chat events with `turn <= this.turn + 1` (note `+1`).
4. **`update()`**:
   a. `getNextValidSetOfMovesPrioritized` → one move per player from input buffers, then V3 sort + dependency loop.
   b. **For each move in resolved order:** mutual-general-swap check first (skip for v1, but the check still runs); else `handleAttack`.
   c. **`updatesSinceLastMove` tracking + all-AFK timeout** (see §3.5).
   d. **`this.turn++`**.
   e. **If `this.turn % RECRUIT_RATE == 0` (= even after increment):** for each general, `incrementArmyAt(general)`; then `handleCityTick` (increment army on every owned city or under-min neutral city); then `handleSwampTick` (no-op for v1 — swamps filtered).
   f. **If `this.turn % FARM_RATE == 0` (= multiple of 50):** for every owned tile, `incrementArmyAt(tile)`; then for each desert, decrement (no-op for v1 — deserts filtered).
   g. `recalculateScores`.
5. **`processReplayGeneralTrades`** — fire trade events (no-op for v1, filtered).
6. **`isOver` check** — winner determined when `alivePlayers === 1` (or single team alive in team modes; not relevant for FFA).

Constants (line 67047–67050 *and* duplicated at 68744–68762):
- `RECRUIT_RATE = 2` (per-turn production cadence in timesteps)
- `FARM_RATE = 50` (land tick cadence in timesteps)
- `MAX_GAME_MOVES_AT_NORMAL_SPEED = 50000` (hard turn cap)
- `MAX_GAME_ALL_PLAYERS_AFK_MOVES_AT_NORMAL_SPEED = 2000` (all-AFK fallback trigger)
- `TIMEOUT_CAPTURE_AFK = 25000` (ms; surrender countdown source)

### 3.2 Combat math (`attack` in map class, line 67283–67313)

For a move `start → end`, `is50` flag:

- `moveReserve = is50 ? ceil(armyAt(start) / 2) : 1` (line 67229–67231).
- `incomingArmy = armyAt(start) - moveReserve` (the army that's "sent").
- `start_army_after = armyAt(start) - incomingArmy` (always equals `moveReserve` after combat, since `incomingArmy` is subtracted from `start` regardless of outcome — line 67310).

**Hostile move** (different team / different owner):
- If `armyAt(end) >= incomingArmy`: defender holds, `armyAt(end) -= incomingArmy`. (Equal-armies → defender holds with `armyAt(end) = 0` post-combat — defender's advantage.)
- If `armyAt(end) < incomingArmy`: attacker captures, `armyAt(end) = incomingArmy - armyAt(end)`, ownership flips to mover.

**Friendly move** (same owner — moving onto your own tile):
- `armyAt(end) += incomingArmy`. Always succeeds. (Ownership unchanged.)

**Defenseless modifier branch** (line 67296–67299) — out of scope for v1 (filtered modifier).

### 3.3 General capture (`handleAttack` line 68063–68099 + `executePlayerCapture` line 68233–68282)

**Detection (line 68088–68093):** after `attack()` resolves, if `(old owner != new owner)` AND `generals[old_owner] == end_tile` AND `old_owner != new_owner`, the move captured a general.

**Execution (`executePlayerCapture`):**
1. **`map.replaceAll(captured_idx, captor_idx, 0.5)`** (line 68234): every tile owned by `captured_idx` flips to `captor_idx` with army count *rounded(army * 0.5)*. Note: the general tile itself is already owned by the captor at this point (combat resolved before this function is called), so `replaceAll` does NOT touch it — the halving applies to the captured player's *other* tiles only. The general tile's army is whatever combat left.
2. **`captures[captor] += 1 + captures[captured]`** (line 68235): captor inherits captured player's kill count plus the new kill.
3. **`capturedPlayers[captor].push(captured)`** (line 68236): track lineage.
4. **`killPlayer(captured_socket)`** if not already dead (line 68238–68242).
5. **`cities.push(captured_general_tile)`, `citySet.add(captured_general_tile)`** (line 68247): the captured general becomes a city. (`Leapfrog` modifier branch at 68243–68246 — out of scope.)
6. **`generals[captured_idx] = -1`**: captured player no longer has a general slot.

**Implication:** the new owner's general count effectively increments by 1 (since the captured general is now a city under their ownership), and they immediately start producing +1 army on that tile per `RECRUIT_RATE` tick.

### 3.4 Production during surrender countdown

Between a player's first AFK event (kill) and second AFK event (neutralize), their tiles remain owned by them — the player is `dead` (can't move) but their tiles haven't been converted to neutral. During this window:
- `handleCityTick` (line 67764–67769) increments army on cities whose `tileAt(city) >= 0` (i.e., owned by any player including the dead one).
- The general-increment loop in `update()` (line 67942–67943) iterates `this.generals[]`; the dead-but-not-yet-neutralized player's general index is still a valid tile index, so it gets incremented too.
- Land tick (`FARM_RATE`, line 67946–67949) increments every owned tile — including the dead player's land.

So a surrendered player's territory keeps producing armies during the countdown. Whoever captures it during this window gets the accumulated armies (then halved by `replaceAll`). Strategic incentive for the still-alive players to mop up quickly.

After `tryNeutralizePlayer` fires (line 68295–68303 in replay class): `map.replaceAll(player_idx, TILE_EMPTY=-1)` — all tiles become unowned. Note the replay class's `tryNeutralizePlayer` calls `replaceAll(e, n)` *without* a multiplier (defaults to 1 — see `replaceAll` line 67332–67344), so armies are preserved. The dead general gets pushed onto `cities` (line 68301): it becomes a neutral city with its accumulated army count.

### 3.5 All-AFK timeout fallback (line 67930–67937)

Net-new — not in `game-mechanics.md` or `replay-parser-design.md`.

```js
const l = this.turn > MAX_GAME_MOVES_AT_NORMAL_SPEED;  // 50000-timestep cap
if (!moves_happened || l) {
  this.updatesSinceLastMove++;
  if (this.updatesSinceLastMove > MAX_GAME_ALL_PLAYERS_AFK_MOVES_AT_NORMAL_SPEED  // 2000
      || l) {
    this._killAllButLastPlayerLargestToSmallest();
  }
} else {
  this.updatesSinceLastMove = 0;
}
```

Two distinct triggers:
- **Hard turn cap:** after 50000 timesteps (=25000 mechanics-doc turns ≈ 7 hours real time at speed 1), kill all but the leader.
- **All-AFK detection:** if no move happens for 2000 consecutive timesteps (~17 minutes at speed 1), kill all but the leader.

**`_killAllButLastPlayerLargestToSmallest`** (line 67957–67979): sort living players by `(total army asc, tiles asc, index asc)`, then `killPlayer` each except the last (= the strongest survivor). They don't auto-neutralize — the second AFK event would need to be in the replay for that, which I haven't traced; for the replay parser path, the replay's existing `afks` array drives everything. Worth verifying with a stalled-game replay sample if any exist in the corpus.

### 3.6 Ranking / winner / lbSort (line 67980–68042)

**`isOver` (line 67980–68001):** for FFA, returns `[winner_socket]` when `alivePlayers === 1`. The first non-dead socket in iteration order is the winner. Once set, `this.winners` is cached.

**`recalculateScores` (line 68002–68024):** rebuilds per-player `scores[i] = {i, color, total, tiles, dead, has_kill}` by iterating every map cell and accumulating armies + tile counts under each player. Then sorts a copy (`leaderboardScores`) via `lbSort`.

**`lbSort` (line 68025–68042) — the ranking tiebreak:**
1. **Players with kills outrank players without** (`has_kill` true beats false).
2. **Alive players outrank dead players.**
3. **Among dead players, sort by death order** (later death = higher rank — uses `this.deaths` array index).
4. **Among alive (or same-kill-status) players, sort by total army desc, then tiles desc, then player index asc.**

**Implication for ranking-match validator:** to reconstruct a replay's ranking, we need to track:
- `this.deaths` (death order — who got killed when)
- `capturedPlayers[]` (which determines `has_kill` for each player)
- Final per-player army/tile counts

The listings-API `ranking` field in `replay_players` is what we match against. Worth a quick verification that listings-API ranking matches `leaderboardScores` order at game end — I believe it does, but checking against a few replays will confirm.

### 3.7 Initial-state population order (`populateFromReplay`, line 67563–67589)

For the filter scope (vanilla FFA, no modifiers):
1. `mountains.forEach(addMountain)` → `setTile(idx, TILE_MOUNTAIN=-2)`.
2. (modifier tile arrays — skipped for v1: observatories, lookouts, tunnels, strongholds, deserts, swamps.)
3. `cities.forEach((tile, n) => addCity(tile, cityArmies[n]))` → `cities.push(tile)`, `citySet.add(tile)`, `setArmy(tile, cityArmies[n])`. Note: tile state is still default (unowned, `TILE_EMPTY=-1`) — cities are passable obstacles with army garrisons, not "structures" in the tile-state sense.
4. `generals.forEach((tile, t) => addGeneral(tile))` → `generals.push(tile)`; if `tile != null`, `setTile(tile, t)` (player `t` owns the general tile) and `setArmy(tile, 1)`. If `tile == null`, the player is marked dead immediately (line 67582–67584).
5. `neutrals.forEach((tile, n) => addNeutral(tile, neutralArmies[n]))` → adds non-city neutral tiles (these are the result of past surrenders that converted territory to neutral land, per `replay-format.md` slot 14/15).

For the parser's setup phase: build map of `mapWidth * mapHeight` tiles, default empty + 0 army. Apply in order above. End state: each tile either mountain / empty / neutral-with-army / owned-by-general / city-with-army.
