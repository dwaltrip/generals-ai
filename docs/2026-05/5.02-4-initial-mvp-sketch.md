# Initial MVP Sketch

**Date:** 2026.05.02
**Status:** Provisional working hypothesis. Sequence and rough scope, not commitment.

This doc captures the current MVP plan. Strategic decisions and reasoning live in `5.02-3-initial-notes-and-decisions.md`; this doc inherits those decisions and focuses on sequencing and concrete scope. Expected to change as data arrives and unknowns resolve.

---

## 1. Goal and success criteria

**Goal:** A trained behavioral cloning agent deployed to the live generals.io FFA ladder, with a measured placement distribution over real games.

**Phase 1 success looks like:** the agent plays competently on the live ladder — at minimum, finishes top half regularly and avoids embarrassingly bad behavior in spot-checks. The probability distribution from the notes doc (§3) anchors expectations: most likely outcome is a "weak-mid bot that validates the pipeline," with real upside if the BC corpus is rich enough.

**Phase 1 ends with a decision point.** Strong outcome → ship and iterate on tuning. Weak outcome → trigger Phase 2 (engine refactor + RL fine-tuning). The decision is informed by real deployment data, not training curves.

---

## 2. Bucket 1 — Prep and infra

The foundation. Bounded work, but real. Output is a parser producing training-ready data and a documented architecture spec.

- **Architecture sketch.** Deliverable is a design doc covering observation tensor schema, network structure, action space, action masking strategy. Gates everything downstream (parser output schema and BC implementation both depend on it). First piece of Phase 1 work. Output is provisional and may iterate as bucket 2 reveals what works.
- **Replay parser implementation.** Reverse-engineer `.gioreplay` format from the AWS bucket (cheap early check: does it match the websocket diff format from `generalsio_state.py`? If yes, parser can crib heavily from that file). Implement parsing → `(obs, action, metadata)` tuples per the architecture spec. Validation: reconstruct known game states from parser output and compare to ground truth. Parser correctness is the highest-risk dependency in Phase 1; cheap to verify, expensive to debug downstream.
- **Dev environment setup.** JAX/Equinox vs. PyTorch decision (open). GPU access (cloud or local). Repo scaffolding for the project, separate from the strakam fork.
- **Data prep / pipeline: filtering and metadata enrichment.** Apply the agreed filters (integrity cuts, early-elim cutoff at turn 50–75, bottom-half placement) to the parsed corpus. Capture rich per-trajectory metadata at parse time (player_id, rating, placement, game length, elim turn, game id, turn-within-game) per the data decision in notes doc §4. Store in a format that lets training-time filters be config-toggled without re-parsing.

---

## 3. Bucket 2 — Training pipeline

The biggest single chunk of Phase 1, and where most things go wrong. Iterative by nature: implement, smoke-test, train, debug, revise.

- **Implementation:** dataloader, network implementation, loss functions, training loop. Built against the architecture spec from bucket 1. Auxiliary heads (placement prediction, attack-target prediction) included if the spec calls for them.
- **Sanity / smoke tests** (before any real training run): forward pass produces correct shapes; loss computes and backprops cleanly; gradients are non-zero and not NaN; action masking is actually applied; model can overfit a tiny batch ("is anything wrong" check); validation split is held out cleanly with no leakage from same game across train/val.
- **In-training evaluation:** validation loss (action cross-entropy on held-out replays); top-k action prediction accuracy on held-out replays as the most interpretable BC signal; action distribution sanity (catch degenerate policies like always-pass); per-auxiliary-head losses monitored separately.
- **Iteration loop:** identify failure modes from metrics and gameplay traces, revise architecture sketch if needed, rebuild affected pieces. Common failure modes to expect: overfitting (train/val divergence), action distribution collapse, auxiliary head dominating gradient, masking bugs.
- **Exit criteria:** validation loss plateauing rather than still descending steeply; top-k action prediction accuracy in a reasonable range (calibrate from initial runs — hard to set in advance); training completed cleanly enough to hand off a checkpoint to bucket 3 for offline eval.

---

## 4. Bucket 3 — Post-training eval and deployment

Goal: trained agent playing live FFA games, accumulating placement data. Eval and deployment are one connected workstream — you can't fully eval without deploying, and you shouldn't deploy without first evaling.

- **Post-training offline eval.** Behavioral spot-checks on saved positions (early-game expansion, mid-game threat assessments, endgame 1v1 collapse). Action distribution statistics on held-out replays compared to human stats. Value head calibration (predicted vs. actual placement on held-out games). Comparison to baseline (nearest-neighbor lookup, scripted bot) for a sanity floor. Acknowledged as judgment-heavy rather than rigorous — no FFA env for simulated games means we lean on the user's FFA expertise for qualitative checks.
- **Deployment surface area.** Live server-state decoding into the same observation format the agent trained on, action sending back to server. Correctness requirement: live format must exactly match training format. A small mismatch (channel ordering, normalization) silently degrades live play and is brutal to debug.
- **Implementation options.** Fork `remote/generalsio_state.py` and FFA-ize it (likely simplest; existing 1v1 version handles the websocket protocol, hardcoded on line 10). Reimplement from scratch if the parser already produces something close to a server-state decoder. Hybrid sharing code between parser and live deployment if `.gioreplay` and websocket diff formats match. Decision deferred until the parser work in bucket 1 reveals what's practical.
- **Pre-live-game validation.** Dry-run on a recorded game. Compare live-decoded observations to parser-decoded observations of the same game. Catches format mismatches before they show up as bad live play.
- **Live deployment.** Connect to ladder, play a small initial batch, sanity-check that behavior is non-pathological. Then accumulate enough games for a meaningful placement distribution.
- **Exit criteria / Phase-2 decision.** Placement distribution after N games (open: 50? 100? 200?) maps to the outcome buckets in notes doc §3. Strong → ship. Weak → trigger Phase 2.

## 5. What's deliberately NOT in MVP

Each is deferred for a specific reason, mostly captured in notes-and-decisions §3 and §7:

- **Engine refactor (`game.py` → FFA).** Not on critical path; BC works on parser output alone. Deferred to Phase 2 if needed.
- **Self-play / PPO infrastructure.** Phase 2 work. Build only if BC results warrant it.
- **League play (AlphaStar-style).** Coordination isn't the bottleneck in FFA per user's input; historical-pool self-play is sufficient when we eventually do RL.
- **Threat-conditioned BC heads / MoE pattern.** Speculative research direction. Pull on if BC plateaus, not before.
- **Reward shaping experimentation.** Only matters under PPO; not relevant to BC.
- **1v1 BC pretraining.** Skipped per notes-and-decisions §2 (negative transfer concern; FFA replays contain the fundamentals).
- **Rigorous offline eval suite.** Lacking an FFA env, rigorous offline eval isn't really possible. Don't fake it; lean on judgment + live ladder as ground truth.

---

## 6. Decision points and exit criteria

Two explicit gates:

- **End of bucket 2 → bucket 3.** BC training completed cleanly? Validation loss plateauing, top-k accuracy reasonable, no obvious pathologies. If not, debug in bucket 2 before deploying. Don't burn live-ladder games on a model that hasn't passed offline checks.
- **End of bucket 3 → ship vs. Phase 2.** Placement distribution from live deployment maps to the outcome buckets in notes doc §3. Strong outcome (top-quartile play) → ship and tune. Weak outcome (mid-pack at best) → trigger Phase 2 (engine refactor + RL). The decision is informed by real deployment data plus user's qualitative judgment from playing against the agent.

---

## 7. Relative difficulty and effort

No hard timeline. For solo hobby work, calendar timelines age badly. Relative shape of effort:

- **Bucket 2 is the meatiest by a wide margin.** Implementation is bounded by the architecture spec, but the iteration loop (debug, revise, retrain) is where most wall-clock time goes. PPO is famously finicky; BC is more tractable but still has its own failure modes.
- **Bucket 1 is bounded but real.** The architecture sketch needs deliberate thought (own session). Parser implementation is unglamorous infrastructure that quietly determines whether everything downstream works.
- **Bucket 3 is small in raw effort but has subtle correctness traps.** The format-matching requirement between training and deployment is the kind of thing that produces silent failures.

Compute is not the constraint. BC training is sub-day on a single GPU; cloud cost is dollars-to-low-hundreds. Debugging time is the real cost.
