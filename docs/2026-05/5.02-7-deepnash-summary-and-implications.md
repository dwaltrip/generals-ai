**Date:** 2026.05.02
**Source:** Perolat et al., "Mastering the Game of Stratego with Model-Free Multiagent Reinforcement Learning," arXiv:2206.15378 (30 Jun 2022). Published in Science as 10.1126/science.add4679.

# Part 1: Careful summary of the paper

## 1.1 Big-picture orientation

DeepNash plays Stratego at human-expert level — top-3 on the Gravon platform, both yearly (2022) and all-time. It uses no search and no human demonstration data; pure self-play from scratch. Three components:

1. **R-NaD core training:** a model-free RL algorithm aimed at converging to Nash equilibrium in two-player zero-sum imperfect-info games.
2. **Fine-tuning:** replaces the policy's softmax projection with a thresholded + discretized one to eliminate low-probability blunders that accumulate over long games.
3. **Test-time post-processing:** hand-crafted rules (memory heuristic, value-bounds heuristic, anti-repetition, eagerness) that filter or modify the policy's outputs at inference time.

For our purposes (we use PPO, not R-NaD), the load-bearing part is the network architecture itself. R-NaD is summarized briefly below; the architecture, observation, heads, and the test-time/fine-tuning patterns get the full treatment.

## 1.2 The R-NaD algorithm (skim if just looking for architecture)

**The problem R-NaD solves.** In zero-sum imperfect-info games, naive self-play with policy gradient cycles around a Nash equilibrium rather than converging to one. This is a known instability. Existing fixes (best-response training, fictitious self-play, league play) are computationally expensive and don't scale.

**The R-NaD trick.** Add a regularization term to the reward that penalizes policy divergence from a "regularization policy":

```
r_modified(a, π) = r_original(a)
                 − η · log(π(a)     / π_reg(a))      for own player
                 + η · log(π_opp(a) / π_reg_opp(a))  for opponent
```

This modified game has a unique fixed point (proven via a Lyapunov function). The fixed point is *not* a Nash equilibrium of the original game. But if you iterate — letting the policy converge to the fixed point, then setting the regularization policy to *that* fixed point, then converging again — the sequence of fixed points provably converges to a Nash equilibrium of the original game.

**Why the form matters.** `r_modified = r_original − η · log(π / π_reg)` is structurally identical to KL-regularization back to a reference policy, which AlphaStar and others use for stability during BC→RL handoff. R-NaD is the principled iterated version of the same intuition — same shape of correction, with formal convergence guarantees in the zero-sum setting.

**NeuRD (Neural Replicator Dynamics)** is the specific policy update rule used inside R-NaD's dynamics step. It's a policy gradient method derived from replicator dynamics in evolutionary game theory. Same family as PPO (soft policy updates) but a different update rule.

**v-trace estimator** is the off-policy correction for value estimation, adapted from IMPALA for the two-player case. Used inside R-NaD's value-function update.

## 1.3 Architecture: the Pyramid Module

The network is built from a single reusable module — the Pyramid Module, a U-Net with residual blocks at each level. The torso uses one configuration; each head uses a smaller configuration. Same building block throughout.

**Pyramid Module structure** (input to output):

1. 1× 2D conv: C=256 channels, no stride, 3×3 kernel, ReLU
2. N× outer conv ResBlocks: C=256, no stride
3. 1× strided conv ResBlock: C=256, stride=2 (spatial dim halves)
4. M× inner conv ResBlocks: C=320, no stride (channels widened at the bottleneck)
5. M× inner deconv ResBlocks: C=320, no stride
6. 1× strided deconv ResBlock: C=256, stride=2 (spatial dim restored)
7. N× outer deconv ResBlocks: C=256, no stride

**Skip connections:** each conv ResBlock has a skip-out connection to its symmetric deconv ResBlock.

**N and M parameters by use:**

| Component | N (outer) | M (inner) |
|---|---|---|
| Torso | 2 | 2 |
| Policy heads | 1 | 0 |
| Value head | 0 | 0 |

So the torso has roughly 10 conv layers; policy heads are shallower; value head is just the contraction-expansion path.

**ResBlock structure:**

- *Conv ResBlock*: skip-out connection, then conv (C//2 channels, optional stride, 3×3, ReLU), then conv (C, no stride, 3×3, ReLU), then residual sum. When striding, the residual connection has its own 1×1 conv to handle spatial dim mismatch.
- *Deconv ResBlock*: deconv (C//2, optional stride, 3×3, ReLU), then skip-in summation (skip from symmetric conv ResBlock processed via 1×1 deconv), then deconv (C, no stride, 3×3, ReLU), then residual.

**Activations:** all ReLU.

**Normalization:** *not specified anywhere in the paper*. Genuine gap in the spec.

**Input/output:** observation enters as 10×10×82, torso outputs a 10×10×256 spatial embedding, heads read off this embedding.

## 1.4 Heads

Four heads in DeepNash. Three policy heads (one per Stratego game phase) plus one value head:

1. **Deployment head** (used for first 40 steps, the piece-deployment phase): reads torso embedding, outputs 10×10 distribution over board cells indicating where to place the next piece. Piece order is fixed, so the head doesn't need to know which piece type.
2. **Piece-selection head** (first sub-step of play phase): reads torso + tiled no-attack ratio. Outputs 10×10 distribution over which of own pieces to move.
3. **Piece-displacement head** (second sub-step of play phase): reads torso + no-attack ratio + 10×10×10 one-hot of the selected piece (10 movable piece types). Outputs 10×10 distribution over destination cells.
4. **Value head:** reads torso + no-attack ratio + selected-piece one-hot. Outputs scalar.

**Important pattern:** each head is itself a Pyramid Module reading off the torso. Heads are not just linear projections — they have their own conv structure for task-specific spatial reasoning.

**Shared value head across phases:** for memory savings, one value head handles all three phases. They keep input shape constant by zeroing out phase-irrelevant channels (no-attack ratio = 0 during deployment, etc.).

**Output projections:**
- Policy heads end with a 3×3 conv (1 channel, ReLU), then masked softmax over legal actions.
- Value head ends with a 3×3 conv (1 channel, ReLU), then flatten, then linear to scalar.

## 1.5 The observation tensor

10×10×82 total. Component-by-component:

| Component | Shape | Purpose |
|---|---|---|
| Lakes | 10×10 | Impassable cells |
| Player's private info | 10×10×12 | Binary: which of 12 piece types is at each own cell |
| Opponent's public info | 10×10×12 | Probability dist. over piece types per cell |
| Player's own public info | 10×10×12 | What opponent could deduce about own pieces |
| Last 40 moves | 10×10×40 | Per-move tensor with attack/move encodings |
| Game length ratio | scalar (broadcast) | Turns played / max turns before draw |
| No-attack ratio | scalar (broadcast) | Turns since last attack / max no-attack turns |
| Phase indicator | scalar (broadcast) | Deployment (1) or play (0) |
| Sub-step indicator | scalar (broadcast) | Selecting source (0) vs destination (1) |
| Previous-step selected piece | 10×10 | For the second sub-step of play phase |

Notes on individual components:

- **Public information tensor.** Probability that each cell holds each piece type, computed by counting unrevealed pieces of each type, conditioned on whether the piece has moved. This is a *policy-independent inference* — derivable from raw history, but encoding it as an input feature is much more sample-efficient than asking the network to learn it. Important design pattern.
- **Player's own public info.** Reasoning about the agent's own information leakage. The network sees what the opponent could deduce about its pieces. Real meta-cognitive layer.
- **Last-40-moves encoding.** Each move tensor has values: −1 at source of regular move; −(2 + t/12) at source of attack with revealed piece type t; +1 at destination; 0 elsewhere. The attacker's revealed piece type is folded into the source-cell value — a single channel encodes both location and revealed identity.

## 1.6 Action space

- **100 discrete actions.** One per board cell (10×10).
- **Three game phases**, each with different action interpretation:
  - Deployment: action = where to place the next piece (in fixed piece order).
  - Play, sub-step 1: action = which own piece to move.
  - Play, sub-step 2: action = where to move the selected piece.
- **Each play move requires two forward passes.** The network sees the previously-selected piece in observation for the second sub-step. Real complication.

## 1.7 Player-centric observations

- The 10×10 board tensor is rotated 180° for the second player so a single network plays both sides.
- Action interpretation is also player-relative.
- Standard pattern; only works because Stratego is symmetric across players.

## 1.8 Training infrastructure

- **Sebulba / Podracer architecture.** Separate Actors / Replay Buffer / Learner / Evaluators.
  - Actors: self-play, write full games to buffer.
  - Buffer: stores variable-length games (not fixed-length sequences).
  - Learner: samples batches, computes gradients, periodically syncs weights.
  - Evaluators: separate flock of low-priority machines, play vs. fixed opponents for monitoring.
- **JAX + pmap** for data-parallel training across many learners.
- **Scale:** 768 TPU nodes for learners, 256 TPU nodes for actors.
- **Episode-level batching.** Full games in buffer, padded and chopped into fixed-length chunks. Forward pass computes exact returns and network state per timestep (no bootstrapping); backward pass computes gradients.
- **Game length up to 3,600 steps**, batch size 768 trajectories.

## 1.9 Fine-tuning phase

After R-NaD converges, a separate fine-tuning phase changes the policy's *output projection*:

- **Original projection:** softmax over logits. Gives every action non-zero probability.
- **Replaced with:** thresholding + discretization.
  - Drop actions with probability < `t_res = 0.03`. Renormalize.
  - Sort remaining probabilities high to low. Round each up to nearest multiple of `1/n_disc` (32 in deployment, 16 in play). Discard once cumulative sum reaches 1.
- **Why:** Stratego games run up to 2,000 moves. Tiny non-zero probabilities on bad actions accumulate into occasional blunders over long games. The opponent can prolong by avoiding attacks until a low-probability mistake fires.
- The fine-tuning loop uses this projection during both rollouts and value/Q-function estimation.

## 1.10 Test-time post-processing

Four hand-crafted modifications applied to a trained policy at inference time. Each is specific in detail but represents a generalizable pattern.

**(a) Memory heuristic.** A policy-independent deductive layer encoding three behavioral inferences:

- A Spy would attack a known Marshal (Spy beats Marshal in Stratego).
- A Marshal would attack a known General or Colonel if no recapture is possible.
- A General would attack a known Colonel if no recapture is possible.

If a piece had a clear opportunity to make one of these attacks and didn't, the agent zeroes out that piece's probability of being the corresponding type in the public-info tensor. So it's narrowing the belief distribution about hidden piece types based on observed (in)action.

The paper notes this is a kludge to compensate for the network's limited 40-move memory window — proper recurrent or transformer memory would subsume it. Listed as future work.

**(b) Value-bounds heuristic.** Two-step lookahead using the value function:

- For each candidate action, compute an upper bound on its value, assuming worst-case opponent response and best-case own piece information.
- "Safe" actions are defined narrowly: piece is known movable, no possible recapture, attack is a guaranteed win for any consistent private info.
- If the value upper-bound for an action is lower than the current state's value, the action is a guaranteed mistake — drop it from the policy.
- Affected <1.5% of turns in human games. Rare intervention but high-leverage when it fires.

**(c) Anti-repetition.** The agent kept moving a piece back-and-forth threatening a known weaker piece. Allowed by Stratego rules up to a limit but annoying to humans. Hand-coded rule: after one threat-and-retreat cycle on the same 2×2 region, don't allow re-threatening the same piece unless the policy has no alternative.

**(d) Eagerness.** The no-attack ratio feature (turns since last attack / max no-attack turns) gets transformed at test time as `r' = 1 − (1−r)^α` with α=2. This makes the agent perceive less time remaining before draw, biasing toward more decisive play. Cheap behavior-tuning lever without retraining.

## 1.11 Drawing rules

Stratego doesn't naturally end in many cases — pieces can hide forever. They imposed:

- 2,000-move automatic draw, or
- 200-no-attack-move automatic draw.

## 1.12 What the paper doesn't specify

- **Normalization scheme.** No mention anywhere. Genuine gap.
- **Optimizer details for the SGD step.** Adam mentioned with R-NaD-specific hyperparameters; not directly portable to other settings.
- **Number of contraction levels for larger maps.** Their single 10×10→5×5 contraction might not scale cleanly to larger inputs — would need either deeper pyramid or different approach.
- **Weight initialization.** Not mentioned.
- **Data augmentation.** Stratego doesn't have rotational symmetry (red bottom, blue top) so 4× rotations don't apply, but vertical reflection would. Not mentioned either way.

# Part 2: Possible implications for our FFA project

## 2.1 TL;DR

- **The architecture spec is now usable.** We have concrete depth, channel widths, block counts, and ResBlock structure. This was the single biggest gap in our planning after the strakam paper said "same architecture as Perolat 2022."
- **Heads-as-Pyramid-Modules pattern transfers cleanly.** Heads have their own conv structure rather than being linear projections.
- **The normalization gap is genuine.** We'll need to either inspect strakam's `network.py` or pick a default.
- **Several patterns from outside the architecture proper are worth keeping in our toolkit** for later phases: policy-independent inference layers, value-function-based pre-action filtering, test-time observation tweaks, fine-tuning to harden the policy projection.
- **R-NaD is conceptually adjacent to KL-to-BC regularization** that we already plan to use in Phase 2. Worth noting the connection; we don't need to implement R-NaD itself.

## 2.2 The architecture (load-bearing for Phase 1)

We can now spec our network with concrete numbers. Provisional architecture for our BC MVP:

- **Input:** observation tensor of (C × H × W). Strakam uses 14 base obs channels + memory features; for FFA we'll add per-opponent channels (decision deferred to architecture session). Likely C in the 25-40 range.
- **Torso:** Pyramid Module with N=2 outer, M=2 inner blocks. Channel widths 256 outer / 320 inner. 256-channel output spatial embedding.
- **Policy head:** Pyramid Module with N=1 outer, M=0 inner. Final 3×3 conv to per-cell logits, masked softmax. Output H×W×9.
- **Value head:** Pyramid Module with N=0 outer, M=0 inner. Final 3×3 conv, flatten, linear to scalar (or vector — see 2.5 below).

Two questions to park for the architecture session:

- **Number of contraction levels.** DeepNash uses one (10×10 → 5×5 → 10×10). Stratego is fixed-size and small. FFA generals maps are larger and variable, with mountains creating non-trivial topology. The strakam paper called out mountain dead-ends as a CNN failure mode. A deeper pyramid (more contraction levels, or alternative multi-scale design) might help with global navigation. Worth investigating during the session.
- **Channel width.** 256/320 is a starting point, not a target. Smaller might be fine for our compute budget; we should be willing to tune downward.

## 2.3 Patterns worth keeping in the toolkit

These aren't immediate Phase 1 work but are real levers we should know about. Listed roughly in decreasing order of likely usefulness:

### (a) Policy-independent inference as input features

The DeepNash public-info tensor encodes a known-correct deduction (probability distribution over hidden piece types given history) as observation channels rather than asking the network to learn it. The memory heuristic extends this with behavioral inferences. The pattern: any time you can compute a policy-independent function from history to inferred state, doing it at the input is essentially free — the network learns to *use* the feature rather than to *compute* it.

For FFA, the obvious analog is **scoreboard inference**. The strakam paper notes a global scoreboard always showing each player's land + army totals (paper §3). The diff between ticks is informative: "opponent X gained 8 land — those tiles are most likely adjacent to X's last-known territory." Strakam's current observation doesn't include this kind of derived feature; we could.

Worth a list of such inferences before the architecture session:

- Per-tick land/army gains per opponent (suggests where they expanded)
- Time since each opponent was last seen (staleness indicator on stale beliefs)
- Inferred 25-turn tick alignment (which players are due for reinforcement soon)
- Whether each opponent has captured a castle recently (army production rate change)

Probably none needed for BC MVP. But cheap to add and exactly the kind of polish that could differentiate a working agent from a strong one.

### (b) Two-step training with hard policy projection

DeepNash's fine-tuning phase replaces softmax with thresholding + discretization to combat blunder accumulation in long games. Generals games are 1800+ turns. Same pathology likely applies. If our BC agent shows a "occasionally blunders catastrophically" failure mode in live play, this is a known fix.

### (c) Pre-action policy filtering via value function

DeepNash's value-bounds heuristic does a short lookahead and filters actions whose worst-case-value is lower than current state. Even a weaker version (without lookahead — just check the post-action value-function estimate against the current state) could be a cheap test-time defense against blunders.

### (d) Phase-conditioned heads

DeepNash has three policy heads for three Stratego phases. We don't have phases, but the pattern (specialized heads for different game states) connects to the threat-conditioned BC heads idea in notes-and-decisions §7. Empirical evidence from DeepMind that this design pattern works at scale — not just speculative MoE. Worth keeping warm if BC produces a single-mode policy that struggles with state transitions (e.g., leading vs. trailing in FFA).

### (e) Test-time observation tweaks

DeepNash modifies the no-attack ratio at inference to bias toward decisive play. The pattern: observation features that are constants from the network's perspective during training can be modified at test time to shift behavior without retraining. Generals analogs:

- Modify perceived enemy strength to bias aggression up or down
- Modify perceived turn count to bias toward early-game vs. late-game style
- Modify visibility of opponent leaderboard rank

Cheap behavior-tuning lever for live-ladder deployment if BC produces an agent that's competent but mis-calibrated on aggression.

### (f) Reference-policy regularization

R-NaD uses iterated reference-policy regularization for self-play stability. Notes-and-decisions §7 already mentions KL-regularization back to BC for the BC→PPO handoff. R-NaD is the principled cousin — same intuition, formal convergence guarantees in zero-sum games. We don't need to implement R-NaD, but the conceptual connection is real and worth knowing for Phase 2.

## 2.4 What does NOT transfer (to avoid accidental over-copying)

- **Three policy heads for game phases.** We have one continuous play phase. One policy head, one value head.
- **100-action discrete space + two-step action decomposition.** Strakam's H×W×9 single-step action space is *better* than DeepNash's two-pass piece-selection-then-destination setup. We keep strakam's choice.
- **Public-information tensor for piece types.** No analog in generals. Pieces are armies, not typed.
- **Memory heuristic specific rules.** Stratego-specific deductions about Spy/Marshal/General behaviors. The pattern (encode policy-independent deductions as features) transfers; the specific rules don't.
- **Anti-repetition rule.** Stratego-specific (chasing weaker pieces). Generals has its own pathological behaviors but they're different.
- **Player-centric 180° rotation.** DeepNash rotates for two players. FFA's 8-player canonicalization is a harder version of this question — there's no obvious unique rotation that puts the agent in canonical position. Open design question (already in notes-and-decisions §5).
- **R-NaD itself.** We use PPO. R-NaD's value is being principled about Nash equilibrium convergence, not beating PPO empirically.
- **Stratego's drawing rules.** Generals has natural endings (general capture, last-survivor in FFA). Self-play could still produce pathological turtle play, but that's a different problem from what DeepNash's draw rules address.

## 2.5 New open questions surfaced

Genuine new gaps from reading DeepNash carefully:

- **Normalization.** Need to either inspect strakam's `network.py` or pick a default. LayerNorm is the safe choice for RL settings (BatchNorm interacts poorly with online policy gradient), but worth confirming what strakam actually uses.
- **Number of contraction levels for FFA-sized maps.** Single-level pyramid may not be deep enough for ~20×20+ grids with mountain topology. Architecture session decision.
- **Scoreboard-derived feature engineering for FFA.** Strakam doesn't currently encode per-tick scoreboard diffs. We could. Cheap to implement, potentially high-value.
- **Whether to keep FFA value head as scalar or extend to placement distribution.** Already an open question in notes-and-decisions §5; DeepNash gives no signal here since 1v1 is binary, so its scalar value head isn't a useful constraint.
- **Test-time policy-projection-hardening for long games.** Worth knowing as a backstop if BC shows blunder-accumulation problems in live play.

In dependency order:
