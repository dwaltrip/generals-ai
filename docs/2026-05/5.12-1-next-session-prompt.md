## A few key highlights from the recent session / discussion

- **Repo structure:** uv workspace, single root venv/lockfile. Cross-package deps go through the `_collector/`-style bridge pattern (one file per upstream module, re-exports only, ruff-enforced). Shared-utilities extraction deferred until two real consumers exist.
- **Simulator framework:** NumPy + plain Python. JAX was considered (Strakam's reference implementation uses it) and rejected: their speedup comes from `vmap` across batched parallel envs for RL self-play, which doesn't apply to single-game sequential replay processing. The FFA priority-sort + inward-first chain is genuinely sequential control flow that resists vectorization in any language. Our parallelism is `multiprocessing.Pool` across CPU cores at the per-game level.

## Open questions on simulator internals

Synthesized across the docs; expect to refine and possibly reorder these during the session. **Lean NumPy-first** for speed.

1. **Internal vs. output state separation.** §3 fixes the output triple (ownership `int8` grid, armies `int16` grid, bitpacked cities mask). Does the sim run directly on that triple, or carry richer internal structure (per-player tile-index arrays, alive masks, cached per-player aggregates) and snapshot down per timestep?
2. **Move / AFK iteration pattern.** Direct positional indexing into the wire array vs. parallel NumPy column arrays vs. (least preferred) typed records. Affects how cleanly the priority sort + inward-first dependency check can be expressed.
3. **Per-player derived state caching.** Things `lbSort` and per-tick logic need (army total, tile count, alive, kills, death order): recompute from grids each tick, or maintain incrementally?
4. **Per-perspective view materialization.** §3 puts fog-of-war assembly in the dataloader, so the parser's per-perspective work is action streams + metadata only — confirm this and pin down action-stream shape.
5. **Production / land-tick implementation.** Tracked "owned tiles" list per player vs. mask-multiply the army grid each tick.
6. **Captured-general → city transition.** How the cities mask is mutated in lockstep with ownership flips on general-capture events.
