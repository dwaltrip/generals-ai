smoke runs, the rest fresh):

Final tally (new work only):

  skip:army-overflow                       60   (0.04%)
  skip:exception                           66   (0.05%)
  (plus 38,686 skip:already-written logged at pre-filter)

~94% of new candidates survived both filter stages + noise floor and
produced a sibling-file pair. ~6 minutes wall time. Total on-disk
intermediate so far: ~165k games, well under 10 GB (matches the
empirical estimate from the 50-game pilot; supersedes
`replay-parser-design.md` §3 / §A.1 envelope of 50–65 GB —
[[project-parser-output-size]]).

Schema-lock posture: deliberately leaving the schema **not** fully
locked yet. Re-runs are cheap (~6 min for the corpus), so if a downstream
consumer surfaces a field gap, we can iterate without it being a
load-bearing commitment.

## Non-obvious findings worth keeping

### DEFLATE compresses snapshot arrays ~30:1 (not the assumed ~3:1)

The design doc's 50–65 GB envelope assumed a generic ~3:1 gzip ratio.
`np.savez_compressed` (DEFLATE) achieves ~30:1 on the per-timestep
ownership / armies arrays because frame-to-frame redundancy in
`[T, HW]` shape compresses far harder than generic estimates predict.
A handful of tiles flip per step; long repeated runs across snapshot
rows crush down. The cities-mask drop the 5.16-3 doc cited as the
reason for the size delta is **not** the dominant factor — it was a
small per-game item. The 5–7× gap is almost all compression ratio.

### Single time-ordered walk over games covers all players

The design doc framed rolling-rate computation as a "per-player
chronological walk." Per-player isn't needed — the dependency is
*temporal* (prior → current), not per-player. One sweep over
`replays JOIN replay_players ORDER BY started ASC` maintains every
filtered player's 200-game deque in lockstep. ~15s for the full
corpus, no parallelism wanted (state is inherently sequential).

### Runtime git-state capture beats build-time SHA

The "I edit, rebuild, test, *then* commit" workflow makes any
build-time SHA capture (e.g. `build.rs` invoking `git rev-parse`)
wrong-by-construction — the captured SHA is the parent commit, not the
working code. Runtime capture in the driver, gated on a clean working
tree (`--allow-dirty` to bypass; bypass stamps output as
`<sha>-dirty`), moves that ambiguity from silent to visible. Pattern
lives in `replay_parser/git_state.py`; can be lifted to other corpus-
producing pipelines later.

The dirty check uses `git diff --quiet HEAD` — tracked-file changes
only. Untracked scratch outputs in `tmp/` or `data/` don't block.

### Pre-filter already-written candidates in the orchestrator, not the worker

Resume-skip is the whole reason restartability works, but letting
already-written ids flow through the pool as cheap skip results
inflates the denominator and games/s rate (~38k of 165k on the second
full run). Single stat-pair check up front, log to the skip CSV, and
exclude from the pool. The per-worker existence check stays as a
race-condition belt-and-suspenders.

### `RollingStat | None` for "no prior data" only fires once per player career

Because snapshots are strictly-prior (per design §5.2), `None` only
fires on a player's first-ever FFA game. After that, even `count=1`
with a 0%/100% rate is a real stat object. The edge case is smaller
than the type hint suggests. The `None → -1.0` / `-1` conversion
happens at the numpy-serialization boundary in `build_metadata`, so
the rest of the dataloader pipeline can branch on `prior_games_count`
or pre-compare against the sentinels without re-interpreting nulls.

## Open / TBD

- **The 66 `skip:exception` cases** in the full run are unaccounted
  for. ArmyOverflow gets its own bucket (60); the generic-exception
  catch-all caught the rest. Likely decode errors or sim invariants
  firing; the skip CSV captures `type(e).__name__` + truncated
  traceback. Worth a one-pass categorization before treating the
  corpus as final.
- **Wire/listing name disagreements.** Driver emits
  `skip:listing-wire-name-mismatch` if a curated name appears in
  `wire.usernames` but not in `replay_players.current_name` for that
  game (or vice versa). Tally in this run was 0. Worth eyeballing if
  it ever becomes non-zero.
