## 1. Framing and goals

The deliverable from this session is the input observation tensor for our FFA BC network. Two layers of decision:

1. **What information do we want to expose to the network?** — feature engineering question.
2. **How do we structurally encode each piece of information into channels?** — representation question.

Both decisions are downstream of an "extremely conservative v1" frame: stay close to the DeepNash/Strakam architectural blueprint, modify input/output where FFA forces us to, and add features beyond the published baselines only where the gain is large and the cost is small. The intent is that v1 is debuggable — if BC underperforms, we know it's because of something specific we changed for FFA, not because we restructured the whole network.

Some grounding principles we settled on early and used throughout:

- **Static tensor shape.** The input tensor's shape is fixed at architecture-definition time. Per-opponent slots exist always, even if some opponents are dead or uncontacted; we use binary alive/contact flags + zero-fill to communicate "this slot is empty."
- **Spatial vs. scalar encoding.** Spatially-localized information goes in `(C, H, W)` channels and is processed by convolutions. Scalars get *broadcast* across `H × W` to become constant-valued channels. This trades parameter efficiency for making global information locally available to convs at every position. (Detailed reasoning in §3.)
- **Split when in doubt.** When a piece of information has multiple semantically-distinct axes (location vs. magnitude, ownership vs. army count, presence vs. recency), giving each axis its own channel is the default. Channels are cheap; convs combining them are flexible. Combining axes pre-network costs nothing structurally but makes the network spend capacity to disentangle.
- **Time as channel index, not scalar.** When encoding temporal sequences, indexing the time dimension as the channel dimension (e.g., "channel k = state at t-k") is dramatically more sample-efficient than feeding a continuous timestep scalar and asking the network to learn time arithmetic. This shapes the choice between "snapshot stack" representations and "compressed history with timestep scalar" representations.

These principles get applied repeatedly below.

## 2. The strakam baseline (what we're starting from)

For reference, strakam's 1v1 observation has 14 channels (from `generals/core/observation.py`):

**Per-cell values:**
1. `armies` — int32, army count per cell, owner-agnostic

**Boolean masks:**
2. `generals` — visible generals
3. `cities` — visible cities
4. `mountains` — visible mountains
5. `neutral_cells` — visible neutral cells
6. `owned_cells` — agent's own cells
7. `opponent_cells` — visible opponent cells
8. `fog_cells` — non-structure cells in fog
9. `structures_in_fog` — mountains/cities in fog (must explore to know which)

**Broadcast scalars:**
10. `owned_land_count`
11. `owned_army_count`
12. `opponent_land_count`
13. `opponent_army_count`
14. `timestep`

Plus the four DeepNash-style memory features (paper-described, not necessarily what's in published repo code):

- Revealed castles/generals positions
- Cells the agent has explored
- Cells the agent knows the opponent has seen
- Last 7 moves on each side

Approximate trained-network channel count is 20-25 (paper doesn't specify exactly).

The FFA design works from this baseline outward — most channels translate, some need per-opponent generalization, and a few new categories get added based on FFA-specific needs and the threat-intuition interview.

## 3. Encoding principle: spatial vs. broadcast scalars (deeper)

We spent meaningful time on this and it shapes several decisions, so worth being precise about.

**Why broadcast scalars at all?** Convolutions are *local*. A 3×3 kernel at position `(i, j)` only sees inputs in a 3×3 window around `(i, j)`. This locality is the inductive bias that makes convs sample-efficient on grid data — a single learned function applied everywhere via parameter sharing. But locality is also a constraint: information at one cell can only affect another after enough conv layers have stacked their receptive fields together.

When a feature is genuinely *global* (e.g., total army count, current timestep, "is opponent N alive"), it doesn't have a natural board location. Two ways to handle:

1. **Broadcast as constant channel.** Tile the scalar across `H × W`, concat with spatial channels. Conv kernels see it at every position from layer 1 onward. *Inefficient in pure parameter count* (you're storing the same number `H × W` times), *but* makes the global signal locally available to a fundamentally local operation. This is what strakam does for `owned_army_count`, `opponent_land_count`, `timestep`, etc., and what DeepNash does for game-length-ratio, no-attack-ratio, and phase indicators.

2. **Separate scalar input head.** Feed scalars to a small MLP, get an embedding, concat into the trunk somewhere. More parameter-efficient but architecturally more complex (where do you concat — before the trunk, after, at every layer?). DeepNash and strakam both essentially use option 1.

For v1, **broadcast is the default** for any global-feature scalar. Simpler, follows the published baselines, channel cost is small at our scale.

**One subtle thing worth knowing.** The U-Net architecture mitigates the locality problem partially via the contraction step. At the bottleneck (e.g., `H/2 × W/2`), each cell's receptive field over the original input is much wider, so global information propagates through the bottleneck even without broadcast. But broadcasts still help, especially for kernels operating at full resolution near the policy head. So the principle holds even with U-Net.

**On "emphasis" intuitions.** A common (but slightly wrong) intuition is that broadcasting "emphasizes" a value by repeating it. The correct framing is that broadcasting *makes the value available everywhere a conv kernel might need it, at the cost of channel storage*. Not emphasis, availability.

## 4. Section-by-section channel decisions

This is the meat of the doc. Each subsection covers one logical category of player data, what we considered, what we decided, and why.

### 4.1 Ownership and presence (current state)

**The question:** how do we encode "who owns each cell right now" in an N-player setting?

**Options considered:**

- **Aggregated "any opponent" mask** (1 channel). Direct extension of strakam. Simplest. *Rejected because it loses opponent identity entirely.* A conv kernel can't tell the strongest player from the weakest from the closest from the farthest. Identity is needed for any threat-aware reasoning.
- **Per-opponent mask, slot N for opponent N** (7 channels). Each opponent gets a binary mask of their visible cells. Identity preserved via slot index.
- **Per-opponent + aggregated** (8 channels). Per-opponent for identity, plus aggregated as a "shortcut" channel. *Rejected as redundant* — the aggregated mask is the OR of the per-opponent masks, which a conv can compute trivially in the first layer. No information added.

**Decision:** 7 per-opponent ownership masks. Plus 1 self mask.

**Sub-question: what determines slot ordering?** Slot 0 needs to mean *something* consistent across all training examples and inference calls, otherwise the network learns nothing useful from "what's in slot 0." Options for the sort key:

- **Canonical-by-ID** (e.g., player ID, or order they joined the game). Simplest. No threat-aware sorting; slot 0 is just "the player whose ID is 0." Works as long as the encoding is *consistent* across training and inference. *Lean for v1.*
- **Sorted by army count.** Slot 0 = strongest opponent. Network can directly attend to "strongest other player." *Risk:* slot identity changes during the game as army counts shift, which means a given opponent moves between slots. Network sees this as a discontinuous swap. May confuse training.
- **Sorted by distance / proximity.** Slot 0 = nearest opponent. Same risk as army-sorted, but distance is a smoother signal.
- **Learned threat score.** Sort by output of an auxiliary head trained to predict "how likely is each opponent to interact with me in the next K turns." Speculative; *deferred*.

**Decision for v1: canonical-by-ID** (simplest, most stable, requires no architectural commitment to a sort key choice). We accept that this means slot identity carries no semantic threat info — the network has to learn to extract that from the per-opponent feature data within each slot.

**Open if v1 underperforms:** if BC fails to generalize across slot orderings (i.e., performs differently on the same situation with opponents in different slots), the threat-sort-key question becomes load-bearing.

### 4.2 Army magnitudes

**The question:** how do we encode "how many armies are at each cell"?

**Options considered:**

- **Single owner-agnostic channel** (strakam approach). One `armies` integer channel showing army count regardless of owner. Owner is determined by separately checking the relevant ownership mask.
- **Per-owner army channels.** `owned_armies`, `opp_0_armies`, ..., `opp_6_armies`. Eight separate channels, each non-zero only at cells owned by that player.

The per-owner version *seems* like the "split when in doubt" principle would prefer. But on closer inspection it's wrong. The "split when in doubt" principle applies when two semantically-distinct facts get combined into one channel. Here, both options encode the same two facts (ownership and army count); the question is whether to combine them at the input or let the network combine them via a learned operation.

The single-channel version has practical advantages:

1. **Owner-agnostic queries are useful.** "Is there a big army anywhere in this neighborhood?" is a useful primitive that doesn't need owner info first. Single channel makes this trivial; per-owner version requires a sum across 8 channels.
2. **Network can combine ownership × armies in the first conv layer.** A 3×3 conv reading both `armies` and `owned_cells` can compute "owned armies at this cell" in a single weighted combination. No capacity is being wasted to disentangle anything.
3. **Smaller channel count.** 1 channel vs. 8.
4. **Strakam precedent.** They use this approach in 1v1; nothing about FFA breaks the logic.

**Decision:** single owner-agnostic `armies` channel. Same as strakam.

This was a useful refinement of "split when in doubt" — the principle is really "split when combining axes hides information from convs." If the network can recover any combination it needs in one conv operation, splitting isn't needed.

### 4.3 Per-opponent broadcast scalars (current state)

**The question:** what global facts about each opponent do we want broadcast across the board?

The threat-intuition interview surfaced that strong play involves tracking each contacted opponent's current state and recent trajectory. The minimum-viable per-opponent scalar set:

- `opp_N_alive` — binary. Necessary because dead-opponent slots are zero-filled, and the network needs to distinguish "this opponent is dead" from "this opponent is alive but I have no current vision of them."
- `opp_N_contacted` — binary. Have I ever made contact (had vision of any of their cells)? Surfaced directly from the interview as a meaningful state — "haven't seen them yet" is qualitatively different from "I've seen them and they're now in fog."
- `opp_N_army_count` — current total army.
- `opp_N_land_count` — current total land.

7 opponents × 4 scalars = 28 channels.

**Decision:** include all four for v1.

### 4.4 Per-opponent broadcast scalars (scoreboard inferences)

This is the area we most argued about for v1 inclusion. Scoreboard-derived features encode signals you specifically described using during play — "their army went 120→122→124 with no fighting, so they have 1 city beyond the general," "their land is shrinking consistently, so they're being attacked." These are *policy-independent inferences* derived from the scoreboard's per-tick state, in the same spirit as DeepNash's public-information tensor.

**Candidate scoreboard-derived features:**

- `opp_N_army_growth_rate` — average army delta per turn over last K turns.
- `opp_N_land_growth_rate` — average land delta per turn over last K turns.
- (Considered, deferred) Inferred city count from growth-rate signature — derivable from army_growth_rate during peaceful periods, but extracting it cleanly requires distinguishing "peaceful" from "fighting." Skipped for v1.
- (Considered, deferred) Time since each opponent was last seen by *any* player on the leaderboard — inferable from leaderboard activity patterns. Hard to compute, skipped.

**Variants on the K parameter:**

- **K=1** (instantaneous delta from previous turn). 14 channels (7 per opponent × 2 features). Cheapest, but noisiest. A single 25-turn city-tick produces a one-turn spike that K=1 captures literally.
- **K=3-5** (smoothed over recent turns). 14 channels still — *K affects parser, not channel count*. Smooths out the city-tick spike, slower to react to genuine changes.
- **K=10+** (longer smoothing). 14 channels still. Even smoother, even slower.

This was the source of a confusion worth flagging: at one point in the discussion, K was misread as a *channel multiplier* (i.e., "K=2 means 2 channels per opponent per feature"). It is not. K is purely a parser-side window length; the network always sees a single rate scalar per (opponent, feature) regardless of K. Total channel count for these features is `7 × 2 = 14`, period.

**Considered alternative: raw historical scoreboard rather than computed rates.** Feed K turns of raw army/land counts as separate channels and let the network learn the rate computation. *Rejected* because (a) network capacity to learn arithmetic on continuous scalars is poor (see §3 / §4.7), (b) channel count blows up at `7 × 2 × K` for the raw version, (c) the rate computation is cheap and unambiguous at parse time.

**Decision: include in v1, with K=3-5.** 14 channels total. The arguments against were all variants of "this is a departure from the strakam baseline, which makes debugging harder." The arguments for: directly encodes signals we know strong humans use, parser cost is trivial, network cost is small. We landed on inclusion but flagged it as the most aggressive of our v1 additions — it's the one most worth reconsidering if v1 has unexpected behaviors.

**Open question for the next session:** is the K=3-5 lean defensible, or do we want to revisit? K affects training-data preprocessing and live-deployment inference both; making the wrong choice means re-parsing.

### 4.5 Self broadcast scalars

Simpler. Strakam has three; we add two FFA-specific ones:

- `owned_army_count` — kept from strakam.
- `owned_land_count` — kept from strakam.
- `timestep` — kept from strakam.
- `num_opponents_alive` — FFA addition. Useful for game-state awareness ("we're in a 4-player situation," etc.).
- `num_opponents_contacted` — FFA addition. Direct read of "how much of the field is unknown to me." Surfaced from the interview ("are there any unexplored pockets nearby where another player might be?").

5 channels total.

**Decision:** include all five.

### 4.6 Long-term memory: per-cell historical state

This is where we did the deepest revision relative to strakam.

**The starting point: strakam's "explored cells" feature.** Strakam describes this as "cells the agent has already explored — so it doesn't waste time re-searching the same ground." But generals' mechanics make this framing slightly off. Generals doesn't have "exploration as a resource you spend" — there's no stamina, no fuel, no return-to-base. What strakam's "explored cells" really encodes (when ported to generals) is **historical vision**: cells you have at any point had vision of, even if currently fogged.

This matters because the difference is what makes the channel useful. Currently-owned cells are already covered by `owned_cells`. Currently-visible-but-not-owned cells (the Moore-neighborhood vision frontier) are derivable from `owned_cells` + adjacency. The genuinely new information is **cells you've seen historically but no longer have vision of** — the cells where you have *some* idea what was there even though you don't know what's there now.

**The reframe:** "explored cells" → "historically seen" — binary, 1 if I've ever had vision of this cell, stays 1 forever once flipped.

But once we're encoding historical vision, the strakam paper itself flags the gap: "the agent does not know *when* it uncovered certain parts of the map, or what were the exact army counts of these cells at that time." We can fill that gap directly:

**The full long-term memory channel set:**

- `historically_seen` — binary mask of cells I've ever had vision of.
- `last_seen_armies` — last observed army count per cell (integer, 0 if never seen).
- `last_seen_owner` — last observed owner per cell. Encoding sub-question below.
- `turns_since_seen` — staleness scalar per cell.

**Encoding sub-question for `last_seen_owner`:**

- **One-hot over [self, opp_0...opp_6, neutral]** = 9 channels. Conv-friendly: each cell has a sparse activation in one of 9 channels indicating owner. Works the same way as a standard categorical embedding.
- **Single integer index channel.** 1 channel where cell value is the owner's index (0=self, 1=opp_0, ..., 7=opp_6, 8=neutral). Compact, but conv kernels reading it as a continuous value have to learn that 3 and 4 are arbitrary categories rather than ordered values. Possible but loses the inductive bias of categorical-as-distinct.

**Decision: one-hot.** 9 channels. Channel budget is comfortable enough for v1; conv-friendliness wins.

**Considered and rejected: per-opponent revealed-structures** (`revealed_structures_opp_N` × 7). The original idea was to inherit DeepNash's revealed-structures memory feature with per-opponent attribution. But `last_seen_owner` already carries the ownership info. Replaced with a single `revealed_structures` channel marking "this cell has a known structure (mountain or city or general) on it" — ownership inferred from `last_seen_owner` if needed.

**Total long-term memory channels:** 1 + 1 + 9 + 1 + 1 (revealed_structures) = 13.

Note these are all K=1 in a temporal sense — "the most recent observation per cell." The trade-off vs. multi-K is discussed in §4.7.

### 4.7 Short-term temporal: recent action/change history

**This was the most-iterated section.** Walking through the iterations because the final design depends on understanding why intermediate options got rejected.

**Iteration 1: Import DeepNash's last-N-moves encoding.**

DeepNash uses 40 channels, one per recent move, encoding source + destination + (regular vs. attack) + piece type via a clever signed-value trick:

```
Mov_m_i[r, c] =
    −1            if a piece made a regular move from square (r, c)
    −(2 + t/12)   if a piece of type t attacked from square (r, c)
    +1            if a piece moved to or attacked square (r, c)
    0             elsewhere
```

The "2 + t/12" packs *three* facts into a single scalar: was-a-source (negative), was-attack (magnitude > 1.5ish), and piece-type (fractional offset). Player identity is implicit via turn-order parity — Stratego is strictly turn-alternating, so even-indexed moves are one player, odd-indexed moves are the other. **40 channels for 40 moves.** Compared to a "clean" encoding (separate channels for source, dest, attack-flag, piece-type one-hot) that would need 15 channels per move = 600 channels, this is a ~15× compression.

**Initial lean: port this directly.** 7 channels for last-7 moves, similar dense encoding adapted for generals. Departure from "clean separate channels" but justified by the channel-count efficiency.

**Iteration 2: Source-deducibility problem.**

User flagged: in generals, **we often can't deduce a move's source.** If we have vision of cell `(x, y)` but only some of its cardinal neighbors (because we own `(x+1, y+1)`), and an army arrives at `(x, y)`, the army could have come from any of the unowned-neighbor cells — we don't know which.

DeepNash sidesteps this entirely because Stratego has full piece-position visibility (you just don't know piece *types*). Generals doesn't.

If we ported DeepNash's encoding directly, we'd have to either:
- Make up sources we don't observe (bad — pollutes the training distribution with phantom signals).
- Leave the source cell at 0 when we couldn't deduce it (creates an inconsistent training distribution where some moves have a source signal and some don't, network has to learn to handle both).

Neither is clean. **Reframe needed.**

**Iteration 3: Reframe as "observed changes" rather than "moves."**

What we *honestly* observe is changes in cell state between consecutive timesteps:
- Army count went from X to Y at this cell (signed delta).
- Ownership flipped at this cell.

This sidesteps source-deducibility entirely — we're not encoding sources, we're encoding the deltas we directly observe.

Initial proposal: two parallel channel sets:

- `army_count_change[t-k][r, c]` — signed integer, observed delta.
- `army_ownership_change[t-k][r, c]` — categorical, encoding ownership transitions.

But ownership change is mostly *derivable* from army change + current ownership masks (army went from 5 of mine to 8 of theirs ⇒ ownership flipped, attacker came from somewhere adjacent). And the current-ownership and `last_seen_owner` channels already carry ownership state. So the ownership-change channel is mostly redundant.

**Simplification:** keep army-count-change snapshots; drop ownership-change as a separate channel set.

**Iteration 4: Revisit "K compressed changes with timestep scalar" as a middle ground.**

User raised: instead of N full snapshots, what about K "most-recent observed change per cell" channels, each carrying the change magnitude plus a separate timestep scalar marking when it happened? This is a compression of N-snapshots — fewer channels, but with longer temporal reach (a cell whose only recent change was 30 turns ago can still carry that info).

This led to a deep discussion of **how networks handle time-as-channel-index vs. time-as-scalar.** Summarizing the conclusion:

- **Time as channel index** (e.g., snapshot stack `(N, H, W)` where N=time): conv kernels treat each timestep channel independently with its own learned weights. Cross-time comparisons are essentially free. The network doesn't have to learn time arithmetic.
- **Time as continuous scalar broadcast**: networks have to learn non-linear functions of the scalar to interpret "5 turns ago" differently from "1 turn ago." This is doable but costs sample efficiency. Networks are bad at scalar arithmetic relative to channel-indexed computation.

The K-compressed-change representation puts time in scalars (the "when did this change happen" feature). The N-snapshot representation puts time in channel index. **The N-snapshot version is significantly easier for the network to learn from**, at the cost of channels.

Crucially, the K-compressed representation has one *unique* advantage: longer temporal reach. A cell whose last observed change was 50 turns ago can still carry that info at K=1, where N-snapshots at N=7 would simply not have observed it.

**Resolution: use both, for different timescales.** They're complementary, not competing.

- **N-snapshots for short-term tactical reasoning.** Channels-as-time, conv-friendly, expensive but cheap at small N. Use for the "last 7 turns" window.
- **K=1 last-seen / last-change channels for long-term memory.** Cell-level "what's the most recent observation here" with staleness scalar. Reaches arbitrarily far back. Use for the "everything before that" memory.

The K=1 long-term part is what §4.6 covers (`last_seen_armies`, `last_seen_owner`, `turns_since_seen`). The short-term part is what this section settles.

**The middle ground (K=2-N, partially compressed) gets rejected** because it combines the worst of both: doesn't have channel-as-time indexing (so the network has to do scalar arithmetic), and doesn't have the K=1 "reach back to first observation" property either. Strict K=1 or full N-snapshots are both useful; intermediate K values are awkward.

**Final design:** N=7 channels of observed-change snapshots. Each channel encodes army-count delta at each cell at that timestep (signed integer, 0 where no observed change). Channel index = timestep offset.

**Sub-question: how to encode actor identity within these snapshots?**

Three variants considered:

- **Variant A: single channel per timestep, all-actor encoding via signed values.** Self source = -1, self dest = +1, opp_N source = -(N+2), opp_N dest = +(N+2). 7 channels total. Most compressed, follows DeepNash precedent. Requires conv kernels to interpret signed-value magnitude as actor identity.
- **Variant B: split self vs. aggregated-opponent.** Two channel sets, "self changes" (7 channels) and "any-opponent changes" (7 channels). 14 channels total. Loses opponent attribution entirely (opp_0 vs. opp_3 indistinguishable in this representation), but cleanly separates self from other. Conv kernels don't have to decode actor-from-magnitude.
- **Variant C: per-opponent channel sets.** "Self changes" (7) + "opp_0 changes" (7) + ... + "opp_6 changes" (7). 56 channels. Full attribution. Most expensive.

Discussion went through several phases:
- Initial lean toward Variant B (self + aggregated) for clarity.
- Reconsidered toward Variant A after appreciating the DeepNash signed-value trick. Reasoning: DeepNash has shown empirically that networks can decode dense signed-value encodings; channel budget matters; minimum-departure-from-baseline argument.
- Then reconsidered again given the source-deducibility issue: the signed-value trick was specifically about packing source + dest + piece-type. With "observed changes" we don't have piece type to pack, and source is often unobserved. So the dense-encoding rationale partially evaporates.

**Lean for v1: Variant B (self + aggregated-opponent).** 14 channels. Reasoning:
- Self vs. world is a meaningful split that the network shouldn't have to learn from signed values.
- Within "all opponents," aggregation is honest about the fact that for many observed changes, we don't know which opponent caused it (an army arrives at a fogged cell; we see the destination but not who sent it).
- Channel cost is acceptable.

**Open question for next session:** is Variant B right, or should we try Variant A for tighter channel budget? Worth weighing one more time.

**Considered and rejected encodings for the change snapshots themselves:**

- **Boolean "did anything change"** per cell. *Rejected* — loses magnitude info entirely. A 1-army change and a 50-army change look the same.
- **Multiple parallel change channels** (army-change + ownership-change + visibility-change). *Rejected* per Iteration 3 reasoning — ownership-change and visibility-change are mostly derivable.
- **Encoding ownership-change-direction in the snapshot** (e.g., signed value where sign = "I lost it" vs. "I gained it"). *Rejected* as adding complexity without clear gain.

### 4.8 Things considered and parked

These are ideas that came up in discussion but aren't in v1, mostly captured for the next-session reading list:

- **DeepNash's per-opponent piece-type belief tensor.** No analog in generals — we don't have hidden piece types. The *pattern* (encode policy-independent inferences as features) is what we apply with the scoreboard-derived rates.
- **DeepNash's memory heuristic** (test-time policy modification based on opponent inaction). Stratego-specific; doesn't transfer.
- **Per-opponent recent-move channels** (Variant C above). 56 channels for full attribution. Too expensive for v1; could escalate if move-attribution turns out to matter and Variant B underperforms.
- **Recurrent / transformer memory.** Both papers flag this as future work. We follow strakam's lead and use hand-crafted memory channels as the substitute. Real Phase 2 lever.
- **Inferred city count for opponents.** Derivable from army-growth-rate during peaceful periods, but extracting cleanly requires distinguishing "peaceful" from "fighting." Skipped for v1; could revisit.
- **Whether the agent has been seen by each opponent.** Symmetric to `opp_N_contacted` but the other direction. Deeper inference (requires modeling what opponents see); skipped.
- **Threat-conditioned policy heads.** Speculative MoE-style design — different policy heads for different threat configurations. Both DeepNash (phase-conditioned heads) and strakam (single-head) are precedents. Notes-and-decisions §7 already flags this; not in v1.

**With Variant A short-term (7 channels instead of 14):** -7

So total v1 input channel count is in the **65-90 range** depending on the open-question resolutions.

For comparison: strakam's 1v1 trained network is estimated 20-25 channels. DeepNash trained network is 82 channels. Our v1 is roughly the same scale as DeepNash, somewhat larger than strakam's 1v1 setup. The DeepNash 256-wide first-conv lift can absorb this without straining; if we end up much past 100 we should reconsider, but at 65-90 we're comfortable.

## 6. Open questions for the next session

In rough priority order:

1. **Whether to include scoreboard-derived growth rates in v1.** The most aggressive of our v1 additions, the one furthest from the strakam baseline. Argument for: directly encodes interview-validated signals, cheap. Argument against: harder debugging if v1 underperforms. Lean: include with K=3-5.

2. **Variant A vs. Variant B for short-term change snapshots.** 7 vs. 14 channels for last-7-turns observed changes. Lean: B (self + aggregated-opp), but A is defensible.

3. **Per-opponent slot ordering.** Canonical-by-ID is the v1 lean. If v1 BC fails to generalize across slot orderings, the threat-sort-key question becomes load-bearing. Worth sanity-checking the assumption now rather than discovering after a training run.

4. **`last_seen_owner` encoding: one-hot vs. integer index.** Lean: one-hot (9 channels), conv-friendly. But integer index is 1 channel and could free up budget if we decide to be tighter.

5. **K parameter for growth rates.** Lean: K=3-5. But this affects parser, so committing wrongly means re-parsing the corpus.

6. **`turns_since_seen` cap or normalization.** Open detail — do we cap this at some value (e.g., 100 turns) and saturate? Normalize by total game length? Leave raw and let the network handle the wide scale? Probably needs thought; default could be "raw integer, no cap."

7. **Whether "any opponent has seen this cell" channel is worth including.** strakam has it; we discussed but didn't fully resolve. Possibly include as an aggregated single channel (matching strakam's approach) rather than per-opponent. Real signal but probably not load-bearing for v1.

8. **Channel ordering / indexing convention.** Practical detail for implementation: what order do channels go in the input tensor? Group by category? By spatial-vs-broadcast? This is mostly cosmetic but worth deciding before the parser writes anything.
