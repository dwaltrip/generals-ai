## Accuracy issues

### A1. §2.2 sample dump has the wrong header values [load-bearing]

The dump claims:
> One real sample: `3KXOs-pPx.npz` for replay `Z2l6i_RZh`, 8 players, T=346, HW=957 (a 29×33 board, roughly), 75 KB sim file.

Empirically (`np.load` on `replay-parser/data/intermediate/3K/3KXOs-pPx.npz`):
- `replay_id = "3KXOs-pPx"` (not `Z2l6i_RZh` — that's a different file under `z2/`).
- `T = 810` (not 346); `ownership.shape = (810, 675)`.
- `HW = 675 = 27 × 25` (not 957; not 29×33).
- `mountains.shape = (131,)` (the doc's "e.g. 210" is fine as illustrative, but the as-cited file has 131).
- File size 74,908 B (75 KB) — that part matches.
- `actions_source.shape = (8, 809)` — confirms P=8 and T-1 = 809 (consistent with T=810).

So `3KXOs-pPx` is the correct file path and the schema rows below the header are correct, but the header attributes belong to a *different* sample (or were transcribed from a draft / older parse). Doc B should regenerate the dump from the actual file before quoting numbers.

While doing the re-dump, also note for spike planning: this sample has **56.7% pass frames** for `actions_source` (3670 / 6472 entries are -1). That's at the top end of the §7 parser-design napkin estimate (~30–45%) and is consistent with one open-turtle game inflating the rate. Worth caveating "pass-frame rate is high; plan loss/eval accordingly" in doc B.

### A2. §2.3 `sim_core_version` dtype claim is consistent with the code, but disagrees with 5.16-3's table

The scratchpad lists `sim_core_version <U24 scalar`. The implementation (`replay_parser/metadata.py:46`) does write `<U24`. But the design doc 5.16-3 §4.1 specifies `<U16`. This is a **stale design doc, not a scratchpad error** — flagging it because doc B's "schema not yet locked" framing is the right moment to note that the meta-sidecar dtypes already drifted from 5.16-3 §4.1 (the `<U16` → `<U24` change for SHA-plus-dirty), and that `perspective_usernames <U32 [K]` (present in the code and in the scratchpad) is not in the 5.16-3 §4.1 table at all (it was added in commit 87da180 per the git log).

So: the scratchpad reflects current truth; 5.16-3 reflects design-time. Doc B can either lock the schema as-implemented or call out the design/implementation drift.

### A3. §2.1 "~64² = 4k shards" — theoretical, not actual

The corpus on disk has **1,444** 2-char-prefix shard directories (`find … -maxdepth 1 -type d | wc -l`), not ~4k. Generals.io IDs use a base-~64 alphabet, so 4k is the theoretical upper bound; actual is ~36% of that because prefix distribution isn't uniform. Cosmetic but doc B may want "~1.4k shards (theoretical max ~64² = 4k)" instead of "~64² = 4k".

### A4. §4.3.4 baseline channel count "~99" doesn't match observation-tensor-design.md at N=10

`observation-tensor-design.md` §2.1 explicitly computes the post-delta total as **86 + 2N**, which is **106 at N=10**, not 99. The scratchpad's "~99" lines up if N≈6.5, or if the scratchpad is silently assuming the 7 `opp_N_has_seen` are *already* dropped (in which case "99 base, 92 after dropping has-seen" double-counts the drop).

For doc B: pick a concrete N (the inherited recommendation is 10–12) and recompute. Probable cleaner picture:
- Spec baseline @ N=10: 106 channels
- Spike v1 (drop 7 `opp_N_has_seen`): 99
- Spike v1 + N=5 (per §8): 89
- Spike v1 + N=0 (per §8): 79

§9's "~92 channels" then needs to either pick N (≈3) explicitly or be revised.

### A5. §7.1 citation is to the wrong doc

§7.1 says:
> Per `compute-considerations.md` §5: torso forward ≈ 5–6 GFLOPS at 256/320 widths, 25² spatial.

`compute-considerations.md` §5 gives total torso as "~10–15 GFLOPS per forward pass." The 5.7–6.3 GFLOPS figure is from **`network-architecture-design.md` §5.1** (the per-component table). The two numbers don't contradict — they're slightly different scoping (compute-considerations §5's number is rough, network-arch §5's is the more detailed envelope), but the cite is wrong. Either fix the doc reference or use the higher number.

The downstream wall-clock math in §7.1 ("Per epoch over 1k games × ~500 frames/game … ~7 min on M1") is order-of-magnitude reasoning anyway, so the citation fix is what matters.

### A6. §4.2.1 vision rule phrasing is slightly garbled (minor)

The scratchpad says:
> Vision range = Moore neighborhood (8-neighbor + self) of any cell owned by the perspective player at this snapshot, PLUS any cell adjacent to a cell I currently own that's mine or a city of mine

Per `generals-io-game-mechanics.md` line 81: "a player has Moore-neighborhood visibility (the 8 surrounding cells) around every tile they own." The "PLUS any cell adjacent to a cell I currently own that's mine or a city of mine" clause looks like an incomplete-thought addition — the first half already covers your own/city tiles' Moore neighborhoods. Not actionable for spike planning but worth tightening if doc B repeats the rule.

## Completeness gaps

### C1. The §2.4 inventory should include the corpus-driver script entry point

`replay-parser/scripts/run_corpus_driver.py` is the user-facing CLI for the parser pipeline; for doc B, the training spike will likely have a mirror (`training/scripts/run_train.py` or similar). The scratchpad mentions `driver.py` (the library) but not the script that drives it. Worth a one-line mention so doc B's "where does the training code live" picks up the existing convention.

### C2. Curated-list manifest gives concrete K-per-game number, not "unknown"

§4.2.7 says "Curated list size unknown without inspecting `curated-player-lists.txt` files; likely a few hundred. **Need to check, easy.**" Doing the check now:

The manifest at `/Users/danielwaltrip/all-files/projects/code/generals-ai/curated-player-lists.txt` points at four files totaling **334 lines (union ≲ 334 unique names)**. The 5.17-1 driver output said 165,608 SQL candidates, ~127k written + ~38k pre-existing = ~165k games with ≥1 curated perspective. From the 3KXOs-pPx meta sidecar, K=1 for that game; the meta-sidecar shape `[K]` averages somewhere in the 1–2 range for the corpus. Doc B can probably commit to game-level random split for v1 with confidence — you do not have ≥150 *distinct* strong players surviving the parse-time noise floor across each individual game.

### C3. The validator could double as a model-quality eyeball harness

`replay_parser/validator.py` already does end-of-game ranking-match against listings. For a spike, this is "is the sim right?" not "is the model right?" — but the same harness shape (load corpus → walk games → score) could be adapted into the "move-review on saved positions" workflow in §3.2. Worth pointing the spike planner at it as a starting template rather than building from scratch.

### C4. No mention of the FFA noise-floor as a v1 training-time filter

The parser already drops perspectives below `prior_games ≥ 50 ∧ 1st-rate > 12.5% ∧ top-3 > 37.5%` (driver.py:48–58). Spike training will inherit this floor whether it wants to or not. Doc B should note: the v1 corpus is *already* skill-floored at the random-baseline level, so "train on whatever the parser emitted" (per §8) gets you a noise-floor-filtered corpus for free. The §8 wording "no additional filter" is fine, but flagging the *existing* floor is worth one line.

### C5. AdamW hyperparams — betas and warmup are unspecified

§4.4.2 lists `weight_decay=1e-4, lr=3e-4` but not `betas=(0.9, 0.999)` (default) and not lr warmup. For a *spike* this is fine — pick the defaults and move on — but doc B should commit explicitly so the trainer doesn't sit in "default" mode that later experiments need to chase back.

### C6. Pass-frame rate empirical number changes the §4.2.3 framing

The 3KXOs-pPx sample shows **56.7% pass frames** (slightly above the §7-design napkin of 30–45%, almost certainly because this single game is one of the high-pass cases). Two implications:

- The "pass head" design call (§4.2.3) gets more weight: if half the frames are pass, the choice between "pass as 10th class" vs. "separate sigmoid + class-mask on H×W×9" materially affects gradient signal balance. Worth noting.
- Move-distribution sanity (§3.2 / §12) should compare to a corpus-level pass rate baseline, not assume "should be ~30%."

This is information the design docs don't have yet — empirically verifying pass-rate distribution across a few hundred games is cheap (`np.load` loop + count) and would be a useful 30-minute checkpoint for doc B.

### C7. Slot canonicalization: doc B should commit to a perspective-channel layout

§11 raises slot canonicalization as a recommendation but the scratchpad doesn't pin down what the input tensor looks like under canonicalization. Two options live in `5.05-3-session-notes.md` §3.3:

- Self is always slot 0; opponents are slots 1..7 in canonical-by-ID order with slot-permutation augmentation.
- Self has dedicated channels (`self_owned`, `self_army_count`); per-opponent channels are `opp_1..opp_7` in canonical-by-ID order.

These end up the same shape, but the implementation differs depending on whether the parser slot-0-canonicalizes at parse time (it doesn't — the per-game intermediate keeps natural slot ordering) or the dataloader does the rotation per perspective. The dataloader has to do it. Doc B should make the "dataloader rotates" explicit.

### C8. Train/val split needs a deterministic seed scheme

§4.2.7 mentions "95/5 game-level random" for v1 spike. For the spike to be reproducible / iterable, doc B should specify the seed / hashing scheme (e.g. "split by `hash(replay_id) % 100 < 5`" or "fixed seed RNG over `sorted(replay_ids)`"). Otherwise the val set drifts every run, and "did the change help" becomes noisy.

### C9. Skip-CSV review is mentioned in §6 but the action is vague

The "66 unexplained `skip:exception` cases" are flagged as risk, but doc B should call it: bucket them now (cheap — `csv` parse on the skip log + `type(e).__name__` count) or commit to "ignore for v1 spike, document as known unaccounted." Open-ended risks tend to migrate from "known" to "blocker" without being acted on; either action closes it.
