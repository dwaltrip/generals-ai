# Strakam generals-bots: Repo Survey & FFA Refactor Notes

**Date:** 2026-05-02
**Context:** Planning an FFA (8-player) agent for generals.io. Strategy: behavioral cloning on expert FFA replays from AWS, then PPO self-play. The reference repo is JAX-based but its env is hardcoded 1v1.

## 2. Environment Internals

### State

`GameState` (NamedTuple) at `generals/core/game.py:34`:

```python
armies:            (H, W) int32
ownership:         (2, H, W) bool       # ← 2-player hardcode lives here
ownership_neutral: (H, W) bool
generals:          (H, W) bool
cities:            (H, W) bool
mountains:         (H, W) bool
passable:          (H, W) bool
general_positions: (2, 2) int32
time:              int32
winner:            int32   # -1 ongoing, else 0 or 1
pool_idx:          int32
```

### Step

`step(state, actions)` at `game.py:315`. `actions` shape `(2, 5)`:
`[pass, row, col, direction, split]` per player.

`env.step(state, actions, pool)` at `env.py:197` wraps it with auto-reset and
returns a `TimeStep` (observation stacked across both players, rewards `(2,)`,
terminated/truncated flags, GameInfo, last_state for bootstrap).

### Env config

`GeneralsEnv.__init__` (`env.py:73-107`) supports:
- Fixed grid: `grid_dims=(H, W)`
- Variable grid: `min_grid_size`/`max_grid_size`/`pad_to` with mountain-padding
  for batching
- Pool size, mountain density range, num cities range, general distance bounds,
  castle value range

The default fallback is 4×4, but the env itself is **not** size-locked — only the
PPO network in `_experimental/ppo/network.py` is.

### Hardcoded to 1v1 — confirmed pervasive

The `(2, ...)` assumption is structural, not parametric:
- `global_update` (`game.py:264, 273`): `ownership[0] + ownership[1]` literal
- `_apply_move` (`game.py:228-241`): four enumerated `lax.cond` branches over
  `(attacker, defender) ∈ {(0,1), (1,0)}` — won't generalize via simple loop
- `get_info` (`game.py:356`): `jnp.stack([sum(armies*ownership[0]), sum(armies*ownership[1])])`
- `_transfer_loser_cells_to_winner` (`game.py:340`): `loser_idx = 1 - winner_idx`
- `get_observation` (`game.py:369`): single `opponent_cells` channel,
  `opponent_idx = 1 - player_idx`
- `_determine_move_order` (`game.py:281`): pairwise chase/reinforce/army
  tiebreaker
- `grid.py:60-82`: places exactly two generals with one distance constraint

**Good news:** `get_visibility` (`game.py:139`) operates on a single ownership
mask — generalizes by calling N times.

## 3. Observation & Action Spaces

**Observation: 14 channels** (`generals/core/observation.py:13`):

```
### Spatial values

1.  armies (int32) - integer of army size in that cell (irrespective of owner)

### Boolean masks

2. generals - visible generals
3. cities - visible cities
5. mountains - visible mountain cells
6. neutral_cells - visible neutral cells
7. owned_cells - player's own cells
8. opponent_cells - visible opponent cells
8. fog_cells - non-structure cells in fog 
9. structures_in_fog - structures - mountain or city - in fog (must explore to know which it is)

### Scalars (broadcast)

10. owned_land_count - total cell count  (in-game leaderboard)
11. owned_army_count - total army sum  (in-game leaderboard)
12. opponent_land_count - total cell count  (in-game leaderboard)
13. opponent_army_count - total army sum (in-game leaderboard)
14. timestep
```

Tensor `(14, H, W)` per player. Vectorized form is **channels-first**:
`(N, P, 14, H, W)` — `observation.py:75-93` stacks with `axis=2`.

**Action mask:** `compute_valid_move_mask()` at `action.py:29` returns
`(H, W, 4)` bool — owned + army>1 + dest passable.

## 5. Replay Handling

**No AWS / `.gioreplay` parsing.** Search for "replay", "S3", "bucket", "boto"
yields only:
- `remote/generalsio_client.py` — stores `_replay_id` after a live game.
- `gui/replay_gui.py` — local playback of in-engine recordings.

User has a separate side-project for pulling FFA replays from the AWS bucket; not
covered here.

**However: `remote/generalsio_state.py` is more relevant than it looks.** It
implements the full live-game diff decoding for the generals.io websocket
protocol — `apply_diff()` (lines 26-37) reconstructs `map` and `cities` arrays
from server diffs, and `get_observation()` rebuilds an `Observation` from the
server state. If `.gioreplay` files use the same diff format as the live
protocol (very likely — generals.io replays are typically the recorded server
stream), this file is a head-start on the replay parser AND the eventual live
deployment path. Note line 10 explicitly hardcodes 1v1:
`self.opponent_index = 1 - self.player_index  # works only for 1v1` — this also
needs FFA generalization.

## 6. Training Infrastructure (Actual State)

`examples/_experimental/ppo/train2.py` is a **single 313-line file**, plus
`train.py` (raw-game-API variant, not read in detail) and `network.py`.

**Has:**
- ✅ Vectorized rollout via `jax.lax.scan` (`train2.py:87-106`)
- ✅ GAE (`train2.py:109-140`)
- ✅ PPO clipped objective + value loss + entropy (`train2.py:143-155`)
- ✅ Minibatched epochs (`train2.py:175-216`)
- ✅ Final `eqx.tree_serialise_leaves` save

**Lacks:**
- ❌ Mid-training checkpointing / resume
- ❌ Self-play — opponent is hardcoded `random_action` (`train2.py:24-36, 58`).
  No opponent pool, no league.
- ⚠️ In-loop win-rate is printed every 10 iters (`train2.py:292-302`), but it's
  vs the random opponent only — no held-out eval against fixed opponents
- ❌ Logging beyond `print` (no wandb / tensorboard)
- ❌ Multi-device (single `jax.devices()[0]`)
- ❌ Hyperparam config (constants in `main()`)

**Network** (`network.py`): the conv backbone is fully convolutional and works
at any grid size — `grid_size=4` is just a default kwarg (`network.py:30`)
controlling the value-head linear layer. The actually-load-bearing limits are
(a) **9 input channels** hardcoded (`network.py:41`) and (b) `obs_to_array`
(`network.py:131-141`) drops 5 of the 14 obs channels (no land/army/timestep
scalars), and (c) **9-action policy head** (4 dirs + 4 half-moves + 1 pass).
Useful as a JAX/Equinox idiom reference; not reusable for FFA as-written.

**Env reward vs. shaped reward — important distinction.** `env.step` itself
returns hardcoded ±1 win/lose only (`env.py:224-225`). `composite_reward_fn`
(`rewards.py:117`: generals captured + army-ratio delta + land-ratio delta +
city captures) is **opt-in**, called only by the trainer (`train2.py:71-73`).
The engine doesn't dictate shaping. For FFA you can pick any shaping policy
without touching the engine.

**Note for FFA:** `composite_reward_fn` is not zero-sum hardcoded but does
assume one opponent — it'll need a per-opponent aggregation policy (max / mean /
sum) or a different signal entirely.

## 7. FFA Refactor Plan

Core insight: **the architecture is the right shape; the 1v1 assumptions are
scattered through the body.**

**Tradeoff: rewrite vs. parameterize-in-place.**
- *Rewrite* (recommended): copy `game.py` → `game_ffa.py` parameterized on
  `num_players`. Cleaner code, faster JIT, no dead `lax.cond` branches at N=2,
  but you fork the codebase and lose the existing tests as direct regression
  scaffolding (you can still port them).
- *Parameterize in place*: mutate the existing functions to handle N players
  uniformly (replace enumerated `lax.cond`s with `fori_loop` or vmap-over-
  players). Keeps tests + a single source of truth, but produces code that's
  harder to read and slower to JIT than either the original or a clean
  N-player version.

Recommendation is rewrite, but the call depends on whether keeping the 1v1
reference live in the same file matters more than code clarity. Either way the
N=2 equivalence test (§7 step 2 below) gives you regression safety.

### Keep as-is (architectural)
- State-pool reset pattern
- JIT + vmap pipeline
- Variable-grid + padding mode (FFA needs bigger grids; already supported)
- Per-player fog-of-war primitive (`get_visibility`)
- Action format `[pass, row, col, dir, split]`
- TimeStep / GameInfo NamedTuple shapes (just bump leading dim 2 → N)

### Mechanical generalizations (boring, bounded)
- `GameState.ownership: (N, H, W)`, `general_positions: (N, 2)`
- `global_update`: sum over all `ownership[i]` instead of `[0] + [1]`
- `get_info`: stack over N players
- `_transfer_loser_cells_to_winner`: needs **chain-of-custody**, not just
  attribution. 1v1 transfers in one shot since the game ends. FFA: A captures
  B → A inherits B's tiles. Then C captures A → C inherits everything A owned,
  *including B's tiles*. Need to either (a) eagerly merge tiles into the
  capturer's ownership mask at each capture (simplest, loses provenance), or
  (b) maintain a separate "captured-by" lineage tensor if you want to surface
  provenance to the agent. Eager merge is probably fine.

### Genuine redesigns (1v1 logic doesn't generalize)
1. **`_apply_move` ownership branches** — replace enumerated `lax.cond`s with a
   single "find prior owner, clear bit, set attacker bit" update. Cleaner than
   1v1.
2. **Move ordering** — the 1v1 chase/reinforce/army tiebreaker
   (`_determine_move_order`, `game.py:281`) is doing real gameplay work: chase
   priority lets defenders escape. For N players this becomes generalized chase
   resolution across all (attacker, target) pairs, then a tiebreaker. Naively
   "pick by slot" probably diverges from real generals.io behavior. **Verify
   against a real FFA replay before deciding.**
3. **Opponent observation channel** — most consequential design decision
   (affects network input shape, BC dataset format, replay-parser output):
   - Single "any enemy" mask (loses identity, simplest)
   - N-1 per-opponent masks in canonical order (by army? proximity? slot?)
   - Per-opponent scalar stat channels
4. **Win condition** — replace `winner: int32 ∈ {-1, 0, 1}` with either
   `alive: (N,) bool` (game ends with one survivor) or `placement: (N,) int32`
   (ranked finishes; more useful for training signal).
5. **Grid generation** — pairwise distance constraints for N generals. Likely
   sequential greedy ("min distance from all previously placed") since strict
   all-pairs is hard to JIT cleanly. Existing placement uses Manhattan distance
   (`grid.py:70`, cheap), not BFS — the BFS work is only for castle placement
   and post-hoc connectivity checks. The castle-near-each-general step
   (`bfs_reachable_within_k` per general) costs ~N× more BFS work in FFA and
   may need restructuring to JIT cleanly.
6. **Reward shaping** — pick a policy: `max` over opponents (paranoid), `mean`
   (balanced), or scrap shaping for placement-based terminal reward. Research
   question, not engineering.

### Suggested execution
1. Copy `game.py` → `game_ffa.py`, `env.py` → `env_ffa.py`, parameterize on
   `num_players`.
2. Start with `num_players=2` and verify trajectories match the original env on
   the same seeds. **This is the regression test** — without it you're debugging
   in the dark. `tests/test_game_jax.py` has 1v1 step-by-step assertions that
   can likely be repurposed directly rather than reinvented.
3. Bump to N=8 and fix what breaks.
4. Keep the original 1v1 env around — fast sanity-check environment, useful for
   ablations.

**Effort shape:** the mechanical generalizations are a focused stretch of
straightforward work. The genuine redesigns (especially move ordering, opponent
observation encoding, and grid generation) are where uncertainty lives — each
has a real design decision behind it that may need iteration. Plan for the
mechanical pass to be predictable and the redesign work to take **noticeably
longer than feels right at first**, particularly if you aim for live-deployment
parity with the real generals.io server. Getting to a self-consistent FFA env
that passes N=2 equivalence is the milestone to anchor on; tuning comes after.

## 8. Gaps Beyond the Env

Listed roughly in dependency order:

1. **FFA env refactor** (above) — blocks everything else.
2. **Replay parser + S3 loader** — handled by user's side project.
3. **BC pipeline** — fully missing. Need: dataset loader from parsed replays,
   state reconstruction from replay timeline, supervised loss (cross-entropy on
   action logits), BC training loop. ~300 lines.
4. **FFA-aware network** — expand input channels (currently 9 of 14 obs
   channels are dropped by `obs_to_array`), redesign action head for FFA
   action space, add the value-head linear layer for the larger grid size.
   The conv backbone itself is grid-agnostic.
5. **Self-play training infra** — opponent pool, checkpointing, eval, logging,
   probably league/PBT eventually. PPO scaffold gives the inner-loop math but
   not the outer-loop structure.
6. **FFA reward design** — see §7 redesign #6.

The env refactor and BC pipeline are the two load-bearing pieces of net-new
work. Everything else is either inherited (PPO inner loop, JAX idioms) or
straightforward once those exist.
