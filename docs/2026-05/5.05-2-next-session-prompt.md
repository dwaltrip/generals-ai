# Background

I'm designing a neural network agent for playing the strategy game **generals.io**. Specifically, the classic FFA mode, which is more complex and challenging for an AI bot, compared to the 1v1 duel mode.

In brief, we are aiming for pure behavioral cloning on a large corpus of expert replays, with a conv-based U-Net architecture that has a strong basis in the prior art.

Our main focus recently has been the observation tensor:

* Two sessions ago, we systematically designed the majority of the input channels, except long-term memory, and produced a detailed "partial spec" documenting our work.
* Our most recent session completed a "first draft" of the entire input tensor — adding long-term memory channels (the biggest remaining gap), plus contact-history and capture-event channels, and resolving deferred details from Part 1. We produced another detailed doc.

The rough high-level strategy so far has been to borrow as much as possible from the two major pieces of prior art:

1. The Straka and Schmid "generals.io 1v1" agent. Referred to in our docs as "strakam" or "generals 1v1" paper / bot / agent / etc.
2. The Perolat et al. Stratego agent, which was the key architectural inspiration for strakam. Referred to as "Perolat", "DeepNash", or "Stratego" paper / bot / agent / etc.

Worth noting: the actual U-Net architecture is being borrowed close to as-is from Perolat. Our recent design work has been on the input to that network — the observation tensor — not the network itself.

## Possible next steps

* Do a careful, detailed technical review of the current v1 draft design of the observation tensor.
    * `5.05-1` is the freshest/rawest of the obs tensor docs and hasn't undergone a detailed review yet. Its main contributions — long-term memory, contact history, capture events — are essentially first-pass and probably benefit most from scrutiny.
* Do a deep-dive on one of the biggest remaining questions for the observation tensor: How many channels can we afford before the network architecture we are borrowing from Strakam and Perolat starts to break down?
    * Our current stance is to be fairly conservative with design changes to the actual network architecture. We have been baselining using a near-exact copy of the layout from Perolat, which is what we believe Strakam did.
        * This is a CNN-based design: the "pyramid module", a U-Net with residual blocks at each level.
    * The logic is that these papers achieved strong results, and our objective is a very natural — if non-trivial :) — extension.
    * Are there any "relatively safe" / "architecturally conservative" tweaks we can make that would reduce the risk from having more channels than either Strakam or Perolat?
* Iterate on the explicit open questions already named in the docs (see appendix). More targeted than the other paths — possibly less high-level discussion and more chipping away at concrete unresolved items one at a time.
* Take a larger step back and do a pass on "checking the foundations":
    * Review the key literature again in very careful detail.
    * Look for any major misconceptions / misunderstandings / incorrect takeaways / incoherencies that have snuck into the work so far.
        * I think this might require a thorough review and cross-checking of each project doc, possibly in chronological order and / or multiple passes.
        * A few of the earlier docs might be best viewed as "snapshots" of our thinking at that point. They could be left as is or marked **"historical"**. E.g. The more recent docs largely supersede an older doc.
    * Make sure we've identified:
        * The most critical assumptions and articulated them clearly.
        * Likewise with the most important decisions that we've made.
