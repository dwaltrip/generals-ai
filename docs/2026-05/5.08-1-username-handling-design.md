# Username Handling — Design Synthesis (session note)

**Date:** 2026-05-08
**Status:** Design ~80% locked; implementation deferred to next session.
**Branch context:** `agent-v1-r-and-d`. Last commit `8564e3c` (sql_helpers + CSV refactor; unrelated).
**Origin:** Spawned from a username-handling repo audit. Discussion narrowed scope, gathered empirical data, and converged on a clean policy.

---

## 60-Second Briefing

The replay-collector pipeline ingests usernames from generals.io APIs, stores them in SQLite, and round-trips them through txt files (some of which are hand-edited / GSheets-mediated). A small population of "weird" usernames in our DB (12 with embedded `\r\n`, 1 in non-NFC normal form) creates risks of silent data substitution, log corruption, and lookup misses if any of them ever entered our action-pipelines.

We've designed a policy that **stores everything faithfully but refuses to act on invalid names as primary targets**, with a noisy warning. Empirically the affected names are isolated long-tail anomalies (~0.009% of 137K players, none in elite tier), so this policy buys us correctness with minimal data loss.

What's locked: the policy spine, the validity predicate, and the boundary list.
What's open: module location, function signature(s), warning-dedup mechanism, a handful of edge calls (see §9).

---

## 1. The Policy

### Spine framing

> **Validity is a property of usernames, checked when we act on a name as a primary entity. The DB stores everything faithfully; consumers refuse to act on invalid names with a noisy warning.**

### The five-point summary

1. **Storage is faithful** — `_player_id` and listing upsert never reject. Replay completeness is preserved (a Tier 1 player in 4th place of a replay we collect still gets a `replay_players` row).
2. **Refuses to act on invalid names as primary targets** — validity check fires at iteration / lookup / API-call / write-to-txt boundaries.
3. **Invalid names are still valid opaque keys** for aggregation — joins, GROUP BY, COUNT all work via `player_id` or byte-exact `name` match. Stats stay correct.
4. **Logs deduped warnings via `repr()`** when validity fails. Once per unique name per process.
5. **Display invalid names via `repr()`** in analysis-script output. No special filtering, no skip bookkeeping; `'bc\\n\\n\\n\\r\\n'` is readable enough.

### Why this works

- **Failure-modes are isolated** to a long-tail cohort that no current workflow targets.
- **The DB stays canonical to generals.io's byte-exact user table** (verified — see §5.2).
- **Aggregation over `player_id` sidesteps the name-string problem entirely.**
- **Action boundaries are few and discoverable** (see §3) — the discipline is enforceable.
- **Any new occurrence is loud** — warnings fire if the situation evolves.

## 2. The Validity Predicate

A name is **invalid** if any of:

- **Contains a Tier 1 character.** Tier 1 = anything `str.splitlines()` splits on or `str.strip()` discards from edges:
  - Splits: `\n` (LF), `\r` (CR), `\v` (VT), `\f` (FF), `\x1c` (FS), `\x1d` (GS), `\x1e` (RS), `\x85` (NEL), ` ` (LS), ` ` (PS)
  - Strip: leading/trailing whitespace of any kind (Unicode whitespace via `str.strip()` no-arg)
  - Note: `\t` is discarded by `strip()` if leading/trailing but not split by `splitlines()`. Treat tabs in any position as Tier 1 to be safe.
- **Is not in NFC normal form.** Test via `unicodedata.normalize("NFC", name) != name`.

The two categories share the policy but have **different rationales** (worth being deliberate about):

| Category | Rationale | Lifetime |
|---|---|---|
| Tier 1 chars | Mechanical: breaks our line-oriented txt format and log line discipline. *We* impose this rule. | Forever |
| Not NFC | Pragmatic: unrecoverable through human/clipboard workflows (any clipboard mediation NFC-normalizes). The player is real to generals.io; we just can't usefully target them. | Until we ever build a normalize-on-read helper |

Implementation note: a single `is_valid_username(name: str) -> bool` predicate covers both. If we later want to distinguish, return an enum/reason. For now, boolean is fine.

## 4. Empirical Findings (the data we won't want to re-derive)

### 4.1 Affected populations in the DB

Sample over all 137,021 player rows:

| Class | Count | Notes |
|---|---|---|
| Tier 1 chars | **12** | All match exact fingerprint `<2 ASCII>\n\n\n\r\n`: `bc`, `qg`, `ad`, `79`, `ek`, `20`, `31`, `ok`, `at`, `mx`, `hi`, `sa` (each followed by `\n\n\n\r\n`) |
| Non-NFC | **1** | Zalgo / combining-mark name |
| Bidi controls (U+202A-E, U+2066-9) | 0 | |
| ASCII control outside Tier 1 | 0 | |
| Names ≥30 chars | 0 | (generals.io has a length cap — somewhere ≤30) |
| ZWJ / BOM | 1 | But it's `'👁‍🗨'` — legitimate emoji ZWJ sequence (eye-in-speech-bubble). Not a problem. |

The 12 Tier 1 names share an identical suffix and 2-ASCII-prefix pattern, strongly indicative of a one-time historical generals.io anomaly (probably a closed input-validation gap). 451 total ranking-appearances across all 12; 145 valid-FFA games. None in top-1000 by game count. None present in any current player-list txt file.

### 4.2 Generals.io endpoint semantics (verified empirically)

| Endpoint | Behavior |
|---|---|
| `/api/validateUsername?u=NAME` | **Byte-exact existence check.** Returns `"true"` or `"false"`. Does NOT do format validation. (User confirmed this independently.) |
| `/api/replaysForUsername?u=NAME` | Internal fuzzy/normalized match. Returns same replays for raw, NFC, and NFD forms of a non-NFC name. |
| `/api/starsAndRanks?u=NAME` | Used by our existing `user_exists()` (`generals_api.py:29`). Behavior not separately probed but assumed byte-exact like validateUsername. |

**Critical inference:** generals.io's user table stores names byte-exactly; their endpoints disagree on whether to normalize. Our DB matches their byte-exact form.

### 4.3 Tooling normalization behaviors (informal verification + general knowledge)

| Tool / Path | Byte-faithful? |
|---|---|
| Python `json.dumps` / `json.loads` | ✓ |
| Python `Path.read_text` / `write_text` with `encoding="utf-8"` | ✓ |
| Python `csv.writer` / `csv.reader` (with `newline=""`) | ✓ for content; `csv.reader` normalizes `\r\n` inside quoted fields to `\n` |
| SQLite TEXT storage | ✓ |
| httpx URL params (`params={...}`) | ✓ (percent-encodes bytes faithfully) |
| Python source-code literals, argv (macOS/Linux) | ✓ |
| Most editors (vim, VS Code, Sublime) on save | ✓ |
| Google Sheets (CSV import + cell display + copy-out) | **NFC-normalizes** |
| macOS clipboard | Sometimes normalizes (app-pair-dependent) |
| Web browser rendered text → clipboard | **Typically NFC** |
| Filesystem (HFS+, legacy) | NFD-normalized filenames; APFS doesn't |
| Agent-mediated text round-trips | **Observed unreliable** in this session: literal Zalgo bytes copy-pasted into a `-c` arg got NFC-normalized somewhere in the chain |

### 4.4 Active-data risk verification (current state, today)

- **`leadeboard-s42-ffa-elite-gsheets.txt`** (45 names): 100% ASCII; all match DB byte-exact; all `validateUsername=true`; all return ≥1 replay from `replaysForUsername` endpoint.
- **`leadeboard-s42-ffawin-elite-gsheets.txt`** (50 names): 100% ASCII; ditto.

Takeaway: The GSheets-round-trip workflow used to produce those player lists has not corrupted any active player data. The risk is real in principle but doesn't bite at the elite tier (top players self-select toward ASCII handles).

## 5. Surface Area — the Survey

### 5.1 Sources of usernames (with trust classification)

"Trusted" = byte-canonical to generals.io's user table. "Less-trusted" = could have been mutated by tools/humans in transit.

| Source | Trust | Code path |
|---|---|---|
| Season leaderboard pages (HTML `__PRELOADED_STATE__` JSON) | trusted | `replay_collector/leaderboard.py:14-50` → `scripts/fetch_season_leaderboards.py` writes JSON |
| Replay listings API (`/api/replaysForUsername`, `ranking[].name` + `currentName`) | trusted | `generals_api.py:34-49` → `runner._process_entry` → `db.upsert_listing` |
| Hand-edited `players.txt` | less-trusted | `_shared.load_players` |
| GSheets-round-tripped `*-gsheets.txt` | less-trusted (NFC normalizer in path) | same loader |
| CSV outputs reopened in spreadsheet apps | less-trusted on round-trip | `compute_persistence_score.py`, `players_by_game_count.py --csv` |
| CLI args | less-trusted | (no current single-name CLI args) |

### 5.2 Sinks (where bytes end up)

- **`players.name`** + **`replay_players.current_name`** (DB, byte-exact)
- **`data/leaderboards/season-N.json`** (lossless)
- **`data/leaderboards/*.txt`**, **`data/top-players/*.txt`** (line-oriented; this is the format under threat from Tier 1)
- **CSV files** (UTF-8 text; lossless for most cases)
- **Logs** (`tmp/<runner>-*.log` via `logging_setup`, plus stderr/stdout)
- **Outbound to generals.io** (`?u=NAME` URL params; httpx percent-encodes)

### 5.3 User flows

**Flow A — Collect replays for a curated player list**
1. Fetch season leaderboard JSON
2. Inspect / filter (often via GSheets) → produces `*-gsheets.txt` (less-trusted)
3. Run `collect-recent` / `fetch-gior` / etc. with `--players <txt>`
4. For each name: `validateUsername` exists-check → walk replays → `_player_id` insert + listing upsert
5. Names go *back to generals.io* (URL params) AND *into the DB*

**Flow B — Backfill replay full-data (Pass-2)**
1. DB has listing rows; some have `wire_data IS NULL`
2. Run fetch-gior with `--players <txt>` filter
3. Names from txt drive `WHERE p.name IN (...)` against DB

**Flow C — Sweep / verify**
1. `sweep-metadata` walks DB, summarizes per-player coverage
2. `check_users_exist.py` verifies names from txt against `validateUsername`
3. Names from txt get matched to DB (the "absent" set check; currently buggy)

**Flow D — Analysis → curated list (the loop-closer)**
1. Run an analysis script (e.g., `compute_persistence_score.py`) over DB
2. CSV out → opened in GSheets → filtered → names back to txt
3. That txt becomes a future Flow A input

## 6. Design Subtleties (gotchas worth flagging at implementation time)

1. **The check fires at action-time, not storage-time.** The single most important "don't get this wrong" — `_player_id` MUST NOT call the validity check. Replays with invalid-named players in any position must still be fully stored.

2. **Warning logging itself must use `repr()` / `%r`.** Otherwise the warning *is* the corruption it's trying to flag. Build this into the helper, don't trust call sites.

3. **Warnings must be deduped per process.** If 200 collection iterations hit the same invalid name, a flat 200 warnings is noise, not signal. Module-level `set()` of names already warned about, log once per unique name per process. (We may want to log a final summary like "skipped N invalid names: {...}" at run end.)

4. **The txt-write boundary is irrecoverable.** If `build_player_list.py` writes an invalid name to txt, the file is corrupted *before* any reader sees it — `splitlines()` can't tell whether the original had 50 names or 49+1-fragmented. Filter at write time. Same predicate, two boundaries.

5. **Two distinct kinds of "invalid"** with different lifetimes: Tier 1 is forever; non-NFC is "until we ever build normalize-on-read." Document this in code or a comment so a future reviewer doesn't flatten them.

6. **`replay_players.current_name`** is a per-game display-name snapshot, separate from `players.name`. Same policy applies: store faithfully, refuse to act on invalid as a primary target. If/when any analysis script uses `current_name` as a key, validity-check it.

7. **The audit's `_log_user_intro` / `_log_user_summary` corruption** (originally Critical #3) is *resolved by validity skip* — we never log a skipped name through `%s` because we never *log it at all* (we skip the user before any logging). Validity gate eliminates the hazard at its source.

8. **Display via `repr()`** is the chosen policy for analysis-script output (no skip + bookkeeping). `'bc\\n\\n\\n\\r\\n'` is funky but readable; counts and totals stay correct.

## 7. Considered and Rejected (with rationale)

| Option | Why rejected |
|---|---|
| **NFC normalize-on-write to DB** | Would break alignment with generals.io's byte-exact user table. `validateUsername(NFC(zalgo))` returns `false`; `validateUsername(raw_zalgo)` returns `true`. Empirically verified. |
| **Add `name_nfc` sidecar column** | Not needed once we lump non-NFC under "invalid" and skip it from primary targets. |
| **Sanitize-on-output discipline for all logs** | Validity gate prevents invalid names from reaching primary log interpolation; broader sanitization unnecessary. (Caveat: if a future log line interpolates from non-name sources, that's a separate problem.) |
| **JSON / JSONL interchange format for player lists** | Filtering Tier 1 at txt-write time + elite-tier names being ASCII anyway → no current need. Revisit if the txt format causes "continual trouble." |
| **Trojan-source / bidi defenses** | Empirically zero hits in 137K names. Don't build defenses without evidence. |
| **Cleanup the 12 latent Tier 1 rows from DB** | Harmless dead matter under the new policy — no code path acts on them. Unnecessary churn. |
| **Use `validateUsername` as a registration-validity oracle at ingest** | It's an existence check, not a format check — confirmed empirically and by user. Wouldn't catch what we'd want it to catch (returns `true` for the 12 newline-suffix names). |
| **Adopt generals.io's registration rules as our reject set** | Their rules are unknown to us and undocumented. Tier 1 is a clearer, principled cut anchored in our own format constraints. |

## 9. Open Design Questions (for next session)

These are implementation-detail choices not yet locked. Roughly ordered by importance:

1. **Module location for the validity helper.** Candidates:
   - New `replay_collector/usernames.py` (new module, focused responsibility)
   - Extension of `_shared.py` (already a "boundary helpers" home)
   - Extension of `db_utils.py` (less natural — that's about schema introspection)
   - **Lean:** new `usernames.py`. Discoverable, focused, room to grow.

2. **Function signatures.** Candidates:
   - `is_valid_username(name: str) -> bool` + caller-side skip + warn loop
   - `validate_username(name: str) -> str | None` (returns name if valid, None + warns if invalid)
   - `iter_valid(names: Iterable[str]) -> Iterator[str]` (filter helper for the common case)
   - **Lean:** export both `is_valid_username` (predicate, no side effects) AND `iter_valid_with_warnings(names)` (filter + dedupe + warn). The first is composable; the second covers the dominant call shape.

3. **Warning dedup mechanism.** Candidates:
   - Module-level `set[str]` (process-wide; simple; resets on import)
   - Per-callsite (a `WarningTracker` class instance)
   - **Lean:** module-level set for now. Process-wide is fine; complexity not justified yet.

4. **Failure mode.** Candidates:
   - Skip silently after one warning per name (current plan)
   - Optional `--strict` flag at CLI level that turns skip → raise (for scripts that absolutely cannot proceed with partial results)
   - **Lean:** skip + warn by default, no flag yet. Add `--strict` if a real call site wants it.

5. **Confirm: leave the 12 latent rows alone?** Implied yes by policy. Just confirm explicitly so it's not relitigated.

6. **`replay_players.current_name`** — does any current code path use it as a primary key for action? (Glance suggests no.) If not, document the policy applies and move on.

7. **Sanitize-on-output discipline for unrelated log lines** — does the validity gate fully cover, or do we still need defensive `%r` in places like `replay_id` interpolations or per-replay summaries? (Probably yes — `replay_id` is hex and safe — but worth a 5-minute audit.)

8. **Do we add tests?** No tests exist for the existing path, but a small unit test for `is_valid_username` covering each Tier 1 char + an NFC/NFD pair + the 12 known names + a normal name would be cheap and high-value.

9. **Cleanup tmp/ probe scripts** (`probe_validate_name.py`, `probe_gsheets_names.py`) — keep for future leaderboard imports, or delete? User-deferred.
